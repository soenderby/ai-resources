# Adjacent Fields for Agent Memory

## What This Document Is

A synthesis of what three adjacent fields — **information retrieval**, **knowledge representation**, and **knowledge management** — contribute to current thinking about agent memory.

The main value of this pass is not that these fields solve agent memory directly. It is that they replace one overloaded word, **memory**, with three clearer design functions:

- **retrieval** — how the right prior information is found
- **representation** — what form durable knowledge takes
- **maintenance** — how accumulated knowledge stays useful over time

That reframing is the main result of the experiment so far.

## Executive Summary

The strongest lesson from this pass is that **agent memory is probably not one problem**.

The adjacent fields split it into three layers:

1. **Information retrieval** contributes relevance, evaluation, and query refinement.
2. **Knowledge representation** contributes scope, structure, and limits on how much should be formalized.
3. **Knowledge management** contributes lifecycle, ownership, review, and pruning.

This sharpens the earlier `memory-for-agents.md` synthesis in four ways:

- retrieval is under-specified in a lot of current agent-memory talk
- representation should stay minimal, scoped, and question-driven
- maintenance is a missing layer in many memory discussions
- the best practical model currently looks like a **stack**, not a single memory system

For agent systems, that means failures often come not just from “forgetting,” but from retrieving poorly, structuring knowledge badly, or neglecting maintenance.

## What Each Field Contributes

## Information retrieval

The IR sources were useful mainly because they impose discipline.

### What transfers well

#### Relevance should be judged against a real need
Retrieval quality should be judged against an **information need**, not just keyword overlap. For agent memory, that means a retrieval system can look plausible while still surfacing the wrong material.

#### Retrieval needs evaluation, not just anecdotes
A retrieval layer for this repo should eventually be tested with:

- benchmark questions
- expected relevant entries or sections
- some notion of retrieval quality

Otherwise occasional success is too easy to mistake for systematic usefulness.

#### Querying should be iterative
The relevance-feedback and query-expansion material suggests retrieval should often be treated as a **refinement loop**, not a one-shot lookup. In repo terms, that supports:

- adjacent-concept links
- query reformulation
- retrieval that expands beyond literal matching

### What does not transfer cleanly

Classic IR assumptions do not map perfectly to a small curated corpus:

- binary relevance can be too crude
- large-scale metrics may not fit well
- aggressive query expansion may add noise quickly

So the transferable lesson is not “adopt textbook IR wholesale.” It is: **treat retrieval as a design and evaluation problem.**

## Knowledge representation

The KR sources were most useful as a guide to adding structure without overcommitting.

### What transfers well

#### Start with scope, not schema
The strongest practical lesson is to define the **domain and scope** first, then decide what needs to be represented. For this repo, that means structure should be driven by real questions rather than by a desire for completeness.

#### Minimal commitment is highly relevant
Gruber’s principle of **minimal ontological commitment** transfers well: represent only as much as needed, leaving room for future change. That applies directly to the concept index and any future explicit memory structures.

#### Clarity and extensibility matter more than elegance
A representation that is elegant but awkward to use is not helpful. The useful target is structure that is:

- understandable
- incrementally extensible
- stable enough to support retrieval

### What does not transfer cleanly

A lot of ontology machinery is too heavy for likely needs here:

- formal ontology languages are overkill
- rich semantic modeling can create more maintenance burden than value
- it is easy to drift into representation for its own sake

So the useful import is not ontology engineering as a whole. It is **ontology engineering as a design discipline and brake on over-structuring**.

## Knowledge management

This was the most underrepresented perspective in the repo before the experiment, and one of the most practically useful additions.

### What transfers well

#### Knowledge changes state over time
The lifecycle material suggests different forms of knowledge need different management strategies at different stages. For this repo, plausible stages include:

- raw source notes
- exploratory concept notes
- syntheses
- stable reference material

Those should not all be managed the same way.

#### Ownership matters
The maintenance sources make a simple but important point: if nobody owns upkeep, decay is the default. A useful memory layer needs:

- stewardship
- review moments
- a correction path
- pruning
- low-friction updates

#### Knowledge should live near the work it supports
The docs-as-code material strongly supports local-first agent memory. When knowledge shares workflow and substrate with the work itself, it is more likely to stay aligned, reviewable, and trusted.

#### Retirement is part of quality
Removing stale material is as important as adding new material. Any future concept or memory layer will need pruning if it is to stay high-signal.

### What does not transfer cleanly

Some KM writing assumes large organizations, formal governance, and non-technical workflows. That does not map directly to a small repo. But the higher-level lessons transfer well:

- lifecycle awareness
- ownership
- maintenance cadence
- design for maintainability

## Practical Implications for This Repo

This experiment points toward a more concrete design for the repo as a knowledge garden.

## 1. Retrieval should be treated as a first-class problem
Future query tooling should support more than literal search.

Promising directions:
- benchmark questions for retrieval evaluation
- expected relevant entries for some questions
- query expansion via adjacent concepts
- retrieval of headings/sections, not just whole files

## 2. Representation should stay lightweight and iterative
The concept layer should remain:
- scoped
- question-driven
- revisable
- minimally committed

That argues against turning the repo into a formal knowledge graph too early.

## 3. Maintenance should be designed, not assumed
Any future concept or memory layer should probably have explicit hygiene rules such as:
- every entry needs a definition
- every entry links to multiple sources
- adjacent concepts are explicit
- weak or unused entries get pruned
- stale syntheses get revisited or marked as dated

## 4. Repo-native memory still looks like the right default
The docs-as-code and maintenance material strengthen the case for keeping durable memory artifacts:
- local
- versioned
- inspectable
- close to the work they support

This still looks more promising than relying primarily on opaque platform memory.

## Working Conclusion

The strongest result of this experiment so far is conceptual clarity.

The adjacent fields suggest that **agent memory should not be treated as a single design problem**. A better frame is:

- **retrieval** — getting the right prior information
- **representation** — deciding what durable knowledge should look like
- **maintenance** — keeping that knowledge useful over time

For this repo, that suggests the next practical steps should not just ask how to “add memory,” but how to:

- improve retrieval
- add just enough structure
