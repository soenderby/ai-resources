# Ensuring Code Quality When Working with AI Coding Agents

## What this document is

A thorough synthesis of what this repository has collected about producing and maintaining high-quality software when AI coding agents do a significant share of the code generation. It draws on the practitioner accounts, software-design literature, empirical research, and conceptual synthesis already present in the collection, and aims to be both a reference for *why* each practice matters and a guide for *how* to apply it.

This is not a tutorial for beginners. It assumes the reader already uses coding agents and has felt the tension between speed and quality that motivates the question.

---

## The foundational shift

The central claim, supported by nearly every source in this collection, is that coding agents do not reduce the need for software design discipline. They increase it.

When code generation becomes cheap, the constraint on software quality moves upward — away from the mechanics of writing code and toward the decisions that determine whether that code will be understandable, changeable, and trustworthy over time. These decisions include what abstractions to create, where module boundaries should fall, which invariants to protect, how to decompose a system around likely sources of change, and when to delete rather than extend. None of these are decisions that agents make well autonomously, because they all require sustained conceptual integrity across a codebase, a quality that depends on coherent intent held over time rather than on local fluency within a single generation pass.

Fred Brooks's distinction between essential and accidental complexity remains the sharpest frame. Agents are remarkably effective at reducing accidental complexity — the incidental friction of representation, boilerplate, mechanical transformation, and repetitive edits. What they do not reduce is essential complexity: the irreducible difficulty of modeling a domain, managing tradeoffs between competing concerns, preserving coherence as a system evolves, and designing structures that humans can reason about under change. The risk, stated plainly, is that agents make it easy to produce large quantities of code that is locally correct, syntactically clean, and architecturally corrosive.

This risk is not hypothetical. The METR study — a randomized controlled trial with experienced open-source maintainers — found that AI use made developers 19% slower on average, despite those developers expecting a 24% speedup and continuing to believe afterward that AI had helped. The study does not prove that agents are counterproductive in general, but it demonstrates that self-reports, benchmarks, and anecdotal evidence can all materially overstate real-world productivity gains, especially in high-context, high-quality codebases where the cost of getting a decision wrong compounds over time.

The question, then, is not whether agents are useful — they clearly are — but how to use them without converting speed of generation into speed of decay.

## The human role: from producer to architect of the agent's environment

Every practitioner source in this collection converges on the same reframing of the developer's job. You are no longer the bottleneck at code production. You are the bottleneck at intent, judgment, and quality control. The model is the intelligence for local generation tasks; you are the architect of the environment in which that intelligence operates. Your responsibilities are now:

1. **Holding coherent intent** across a complex change — deciding what should exist, what should not, and why.
2. **Filling the context window correctly** — the agent starts every session amnesiac, and every quality failure that stems from missing information is your failure of preparation, not the model's failure of capability.
3. **Judging output at higher throughput** — reviewing more code per unit of time, which demands sharper heuristics and better automated support.
4. **Catching architectural failures before they compound** — the kind of subtle boundary violation or coupling increase that looks fine in a single diff and becomes a serious liability over twenty diffs.
5. **Deciding when to stop** — when the system is good enough, when a different approach is needed, and when the agent is making things worse.

The Pydantic article names the emotional dimension of this shift: *supervision fatigue*. The satisfying parts of programming — solving a puzzle, watching a design click into place, making a failing test pass — are increasingly delegated. What remains is the cognitively demanding, emotionally unrewarding work of monitoring, reviewing, and course-correcting. The HBR study of roughly 200 workers documents the organizational version: task expansion (taking on work outside your formal role because AI makes it feel possible), blurred boundaries (one more prompt leaking into evenings), and multitasking overload. These are not arguments against using agents. They are arguments for designing workflows that concentrate human attention where it has the highest leverage and refuse to scatter it across low-value monitoring tasks.

NK Schjødt's operator metaphor is useful here: "AI is an excavator, you are the operator." An excavator amplifies a skilled operator and is dangerous for an unskilled one. Expertise does not become less valuable when production gets cheaper. It becomes the only scarce input.

## Planning before generation: quality assurance moves upstream

The single most consequential workflow change when using agents is to move quality assurance from code review to planning.

In a traditional workflow, the developer writes the code and a reviewer checks it. The review catches bugs, style issues, and occasionally architectural concerns, but the developer's own design thinking has already shaped the structure. With agents, the code itself is often syntactically fine — the failures are upstream: a wrong decomposition, a misunderstood constraint, an architectural decision that was never explicitly made and therefore never explicitly reviewed.

Harper Reed's three-loop workflow makes this concrete:

1. **Brainstorm → Specification.** Use a conversational model to extract a complete specification through structured questioning. One question at a time. The goal is not to tell the model what to build, but to force yourself to articulate what you want, what the constraints are, what is fixed, and what is fuzzy. The artifact is a spec, not code.
2. **Specification → Plan.** Feed the spec to a reasoning model and ask for a sequenced set of implementation steps — each small, each building on the previous, each with tests or acceptance criteria. This is the artifact you review. If the plan is wrong, the code will be wrong, and no amount of post-hoc review will make it right cheaply.
3. **Plan → Execution.** Feed the steps one by one to a coding agent.

NK Schjødt frames the *why* clearly: in the old world, quality assurance happened at code review. In the agentic world, the code is often fine; the failures are architectural decisions that should have been caught before a line was written. The plan is now the primary artifact of human judgment.

Kieran Klaassen's three-fidelity model adds useful nuance. Not every task requires the same planning overhead:

- **Quick obvious fixes** — small, well-understood changes with clear scope. Minimal planning. Let the agent do the work, verify the diff.
- **Bounded but non-obvious work** — this is where planning pays off most. The problem is understood but the implementation path is not obvious. A structured plan prevents the agent from choosing a path that creates hidden coupling or violates unstated constraints.
- **Genuinely uncertain work** — the problem itself is not well enough understood to plan against. Here, the right move is a disposable prototype: let the agent explore the problem space, learn from the result, discard the code, and then write the real plan from a position of understanding rather than speculation.

The critical discipline is recognizing which fidelity level applies and not treating every task as if it were the first kind.

## The feedback loop: small steps, executable verification, continuous refactoring

Agents make feedback loops faster. They do not make them obsolete. The temptation to prompt for a large feature, let the agent generate a large blob, and review at the end is the single most common source of quality failure in agent-assisted development. It is also the most seductive, because the agent produces plausible-looking output quickly and the developer mistakes speed of production for evidence of correctness.

The disciplined alternative is Kent Beck's TDD cycle, adapted for agent-assisted work:

1. State intent clearly in domain terms.
2. Define or refine a failing test or check.
3. Ask the agent for one small change.
4. Review the diff — names, interfaces, and module boundaries, not just logic.
5. Refactor for clarity.
6. Run the tests.
7. Repeat.

Each iteration should be small enough that the diff is genuinely reviewable and the change is genuinely reversible. Simon Willison's formulation: several small PRs beat one big one. An agent that has modified thirty files has produced something almost impossible to review meaningfully.

Automated feedback is the force multiplier within this loop. The more verifiable constraints you can express mechanically — tests, linters, type checkers, schema validators, conformance suites — the longer-horizon tasks the agent can reliably complete, because it can check its own work rather than relying on your review for every iteration. This is why coding is currently the most successful domain for agents: code is verifiable. Tests pass or fail. Linters complain or don't. The agent can iterate against these signals autonomously.

Practical forms of automated feedback, in approximate order of leverage:

- **Tests first** (Willison): write the failing tests, then let the agent implement until they pass. The tests *are* the specification — executable, unambiguous, and self-checking.
- **Linters and validators** (NK): enforce conventions mechanically so the agent cannot claim "done" when conventions are violated. Telling the agent to follow a convention is weaker than making the convention enforceable.
- **Pre-mortems** (NK): ask a fresh session to assume the plan has catastrophically failed and diagnose why. A different model instance with no sunk-cost bias often catches specification gaps the original session missed.
- **The agent reviews the review** (NK): after an automated review tool flags issues, ask the coding agent to evaluate the review skeptically. The third-opinion effect catches both false positives in the review tool and genuine issues the developer might have dismissed.

The underlying principle, observed by Andrej Karpathy at the model-training level and by practitioners at the workflow level: systems that optimize against automatically verifiable rewards improve dramatically compared to those that rely solely on human feedback. The same holds for your agent workflow. Every constraint you can express as an automated check is a constraint the agent enforces on itself, freeing your attention for the decisions that cannot be automated.

## What to look for: design principles as review heuristics

Reviewing agent-produced code requires sharper heuristics than reviewing human-produced code, because the failure mode is different. Human code tends to fail through inconsistency, oversight, or fatigue — the developer knew the right thing but didn't do it everywhere. Agent code tends to fail through plausible coherence: it looks clean, follows patterns, and compiles, while making structural choices that a knowledgeable developer would reject on closer inspection. The following principles, drawn from the software-design literature already in this collection, serve as review heuristics specifically calibrated for this failure mode.

### Decomposition around decisions, not workflow

David Parnas's 1972 criterion remains one of the most useful tests for agent-produced modularity. The key move is to decompose systems around design decisions likely to change — storage choice, protocol, vendor integration, policy variation, coordination logic — rather than around the processing steps in a workflow.

Agent-generated modularity characteristically follows the workflow pattern: a controller layer, a service layer, a repository layer, a helper layer, a util layer. This decomposition can be tidy without actually hiding anything. Every layer knows about every other layer's concerns; a change to the storage mechanism ripples through the entire stack. Good decomposition, by contrast, encapsulates the volatile decision behind an interface so that changing it does not require changes elsewhere. The test is straightforward: pick a design decision that might change (the database, the authentication provider, the pricing algorithm) and ask whether it is contained behind a single interface or smeared across the system.

### Module depth

John Ousterhout's concept of module depth sharpens the same concern from a different angle. A deep module absorbs a large amount of complexity behind a simple interface. A shallow module exposes nearly as much complexity as it contains, forcing its callers to understand its internals.

Agent-produced code has a pronounced tendency toward shallow layering: many small modules with thin abstractions that merely delegate to the next layer, adding indirection without reducing the information the caller must hold. The review question is: does this module hide complexity, or does it merely move it? If removing the module and inlining its implementation would not make the caller harder to understand, the module is not earning its existence.

### Simple versus easy

Rich Hickey's distinction between simple and easy is one of the most important conceptual tools for evaluating agent-generated systems. Easy means near at hand, familiar, quick to produce. Simple means not tangled — not interleaved, not coupled in ways that make independent reasoning impossible.

Agents produce easy code at scale: familiar patterns, symmetric structures, conventional naming, framework-shaped organization. Easy code *feels* productive to generate and clean to read. But it is not necessarily simple. A system with twelve microservices, each following the same template, each with its own ORM layer, each duplicating the same validation logic in slightly different ways, is easy to generate and easy to read in isolation but deeply tangled as a system. The review question is: is this actually simpler — fewer moving parts, fewer hidden dependencies, less state to track — or is it merely easier to produce?

### Domain fit

Eric Evans's domain-driven design and Daniel Terhorst-North's CUPID framework converge on one of the strongest practical quality filters for agent-produced software: whether the code is shaped by the problem domain or by the technical framework.

Framework-shaped code is what agents produce by default. Names mirror transport objects. Boundaries mirror framework directories. Logic mirrors database tables. Abstractions mirror technical layers. This is not because the agent has reasoned about the domain and concluded that these are the right boundaries; it is because the agent's training distribution is dominated by framework tutorials, Stack Overflow answers, and boilerplate-heavy codebases.

Domain-shaped code uses the language of the business. Boundaries reflect real distinctions in the problem. Invariants are visible. Architecture follows the domain more than the tools. This is harder for agents to produce because domain fit requires contextual judgment: you have to know what the domain distinctions *are* and insist that the code reflect them. It is, correspondingly, one of the most valuable things a human reviewer can enforce.

The CUPID checklist provides a compact review framework:

- **Composable** — does this play well with others? Small surface area, minimal dependencies, intention-revealing APIs.
- **Unix philosophy** — does it do one thing well?
- **Predictable** — does it behave as expected, deterministically, with observable behavior?
- **Idiomatic** — does it feel natural to people fluent in the language, ecosystem, and local conventions?
- **Domain-based** — does the code model the problem domain in names, structure, and boundaries?

### Habitability

Richard P. Gabriel's concept of habitability adds a dimension that purely structural heuristics miss: software is not just an artifact to generate and execute; it is a place people must live in, navigate, repair, and extend over time.

Habitable code has recognizable patterns, names that make sense without decoding, ease of navigation, room for piecemeal growth, and a texture that gives the reader confidence when making a change. Uninhabitable code may compile, may pass tests, may even exhibit reasonable structure, but when a developer enters it to make a change they feel lost, anxious, and unable to predict the consequences of their edit.

This is a particularly important heuristic for agent-produced code because agents optimize for local correctness, not for the experience of a future reader entering the codebase for the first time. The review question is subjective but essential: would another developer feel comfortable entering this code?

### Changeability as the integrating metric

Across Beck, Parnas, Ousterhout, Feathers, Gabriel, and Brooks, the recurring thread is that good software is software that can be changed safely. Not software that is elegant, or minimal, or theoretically optimal, but software where the next change is easier than the last — where the developer can make a modification with confidence that they understand the consequences.

This is the integrating metric for all the heuristics above. Deep modules make change safer because changes to hidden decisions are encapsulated. Domain fit makes change safer because the structure tracks the axis along which the real world changes. Habitability makes change safer because the developer entering the code can form accurate expectations. Tests make change safer because they detect regressions mechanically.

After every agent-assisted change, the question worth asking is: **is the next change now easier or harder?** If harder, the code may be locally correct while globally degrading.

## Context engineering: the quality of the agent's environment

A large share of what practitioners attribute to model capability or model failure is actually a function of context quality. An agent operating with the right information, constraints, and feedback produces dramatically better output than the same model operating with a vague prompt and an empty context window. Context engineering — the discipline of designing what information reaches the model, when, and in what form — is arguably the highest-leverage skill in agent-assisted development.

Andrej Karpathy's framing is the most compact: the context window is RAM; the LLM is the CPU. Your job is to fill that RAM correctly before asking the CPU to do anything.

### The input-to-output ratio

NK Schjødt offers a useful quantitative heuristic: if your input is less than 20% of total conversation tokens, you are under-specifying. A 4,000-line code output from a three-line prompt means the model is guessing — filling in constraints, assumptions, and design decisions from its training distribution rather than from your intent. High-quality agent output requires high-quality input: constraints, prior decisions, domain vocabulary, non-goals, examples of what you want and what you do not want.

### Context rot

Simon Willison coined the term *context rot*: as the context window fills with back-and-forth — instructions, questions, corrections, partial outputs, abandoned approaches — quality degrades. The model loses specificity, becomes vague, and starts contradicting earlier decisions. The practical response is to recognize when context rot is occurring and start a fresh session rather than patching an increasingly degraded one. Long sessions are not inherently better than short ones; a fresh session with well-prepared context often outperforms a long session with accumulated noise.

### Progressive disclosure and the attention budget

Anthropic's context engineering guide introduces two useful architectural concepts. The *attention budget* treats context as a finite, depletable resource: every token loaded into the context window competes for the model's attention, and irrelevant tokens dilute the signal from relevant ones. *Progressive disclosure* follows from this: load detail only when needed, not all at once. A project-wide constitution belongs in every session; a detailed specification of the save-system's synchronization edge cases belongs only in sessions that touch the save system.

### The hot/warm/cold loading model

The most architecturally complete model for context loading comes from Aris Vasilopoulos's Codified Context paper, which developed a three-tier system across 283 AI-assisted sessions on a 108,000-line codebase:

- **Hot context** — always loaded at session start. The constitution, the CLAUDE.md, the top-level AGENTS.md. Universal conventions, architectural principles, workflow instructions. The constraint is space: context files longer than roughly 200 lines show reduced model adherence. This layer should be budgeted carefully and kept under pressure.
- **Warm context** — loaded per task. Specialized agent specifications invoked through trigger tables, path-scoped rules, skills loaded on demand. This is where domain-specific knowledge lives: the failure modes of a particular subsystem, the rationale behind a specific boundary, the patterns that have been tried and rejected for a specific kind of problem.
- **Cold context** — retrieved on demand. Detailed specifications, past solution writeups, architectural rationale documents. Available through MCP servers, grep, or explicit tool calls. Too large to fit in context by default, but available when the agent knows to look.

The practical discipline is to push everything possible from hot to warm, and from warm to cold, so that the always-loaded context stays compact, focused, and high-density. The temptation to dump everything into the top-level file produces exactly the context bloat that reduces adherence.

## Judgment memory: making quality compound across sessions

Context engineering addresses the *mechanism* of getting information into the model's working memory. Judgment memory — a term coined in this repository's synthesis work — addresses the *content* problem: what should that information contain if the goal is not just to configure the agent but to transmit the accumulated judgment of the team?

An agent that loads "use 2-space indentation" is following a configuration rule. An agent that loads "use composition over inheritance because our domain model evolves faster than our type hierarchy, and deep inheritance hierarchies have repeatedly caused cascading changes during domain refactors" has inherited judgment. The distinction matters because configuration handles the predictable cases and judgment handles the novel ones: when the agent encounters a situation not directly covered by any explicit rule, configuration gives it nothing to reason from, while judgment gives it principles that generalize.

Judgment memory is the subset of persistent agent context that encodes rationale, heuristics, preferences, lessons, and failure-mode knowledge. It sits at the intersection of three concerns: memory for agents (it is the highest-value kind), context engineering (it is the highest-value content to engineer), and software quality (it is what makes quality compound over time rather than resetting with each session).

### Four production mechanisms

The research identified four distinct ways judgment memory is produced, each with different effort-quality tradeoffs:

**Manually authored by humans.** The developer writes CLAUDE.md, AGENTS.md, Architecture Decision Records, specification documents, or review checklists. This produces the highest-quality judgment — it captures rationale, taste, and domain knowledge that no automated system can extract — but it requires the highest effort and depends on the developer noticing and articulating implicit knowledge, a skill most people have not cultivated. The Agent READMEs paper's empirical finding is sobering: these files grow through incremental additions but rarely shrink. Pruning is neglected. Instructions are heavily skewed toward functional operations (coding conventions, build commands, project structure); non-functional requirements like security, performance, and maintainability are almost absent. The hard judgment — the kind that prevents architectural drift — is the kind least often written down.

**Automatically captured by agents.** The agent notices patterns, corrections, and preferences during work and records them for future sessions. Claude Code's auto memory is the primary example. The strength is low effort: capture happens in the flow of work, and corrections the developer made but might not think to document get preserved. The weakness is quality uncertainty: the agent decides what is worth remembering, and there is no evidence yet on whether auto-captured judgment is as useful as manually authored judgment. Accumulation without pruning is a known problem.

**Automatically detected from the codebase.** Tools like Codebase Context MCP analyze code and git history to infer conventions, patterns, and trends without anyone writing rules. This can detect conventions the team follows but has never articulated, track whether patterns are rising or declining in adoption, and identify "battle scars" — code that looks wrong but exists for a reason. The weakness is that it captures *what* but not *why*: it can tell the agent that the team uses a particular error-handling pattern but not what failure mode motivated that pattern.

**Mined from human practice at scale.** The Awesome Reviewers project mines real code review comments from over 1,000 open-source projects, abstracts them into patterns, and encodes them as reusable AI-ready prompts. This produces broad coverage — 470+ review prompts across 15 languages — but abstracts away the project-specific context that makes judgment most valuable. A review pattern that is good advice for Next.js may be harmful for a real-time embedded system.

These four mechanisms are not alternatives; they are layers. A well-maintained project might use manually authored judgment for its most critical architectural principles, auto-capture for session-level corrections, codebase detection for convention enforcement, and community-mined prompts for generic review coverage.

### Architecture Decision Records as decision memory

Michael Nygard's ADR format deserves specific attention because it addresses a failure mode that becomes more dangerous as agents accelerate the rate of change: the loss of design rationale. When changes happen quickly — and agents make them happen very quickly — teams forget why an interface looks awkward, why a dependency was rejected, what tradeoff a boundary protects, and which non-functional requirement motivated a design decision. The decision itself is visible in the code; the reason behind it is not.

ADRs are short documents (typically a title, status, context, decision, and consequences) stored alongside the code, ideally in version control. They are small enough to be realistic to write and maintain, and they are structured enough to be consumed by agents as well as humans. The Archgate project takes this further by attaching executable rules to ADRs that can be checked in CI, turning passive decision memory into enforced decision memory.

The agent-era extension, described by Chris Swan and by the Adolfi.dev workflow, is to configure the agent to create ADRs automatically when it makes changes that affect the overall architecture. The rationale is that the agent, having just made the decision, is in the best position to document it — the context is still fresh. This is a form of judgment capture that happens as a side effect of work rather than requiring a separate documentation effort.

### Failure modes of judgment memory

Six confirmed failure modes, all documented in the research:

1. **Specification staleness.** The most dangerous. When code changes but the spec does not, agents generate code based on outdated information. Agents trust documentation absolutely; they have no mechanism for suspecting that a spec might be wrong. The Codified Context paper documented at least two occasions where outdated context caused agents to generate code that conflicted with recent refactors.

2. **Context bloat.** Files grow through incremental addition and rarely shrink. Over time, context files become cluttered with stable conventions, ephemeral learnings, and contradictory instructions. The model cannot prioritize in a bloated context; adherence drops.

3. **Brevity bias.** The opposite of bloat. When automated systems optimize context iteratively, they tend to collapse toward short, generic prompts. The specific failure-mode knowledge and domain-specific rationale — the actual judgment — gets stripped out because its contribution is harder to measure than prompt length.

4. **Contradictory instructions.** When judgment accumulates across multiple files (CLAUDE.md, rules directories, auto memory, nested AGENTS.md files), contradictions are inevitable and currently undetected. The model picks arbitrarily.

5. **Convention without rationale.** Rules without reasons are configuration, not judgment. They are easy to generate and easy to follow, but they do not help the agent make better decisions in novel situations.

6. **Judgment that prevents learning.** Judgment encoded as dogma ("always use Redux") rather than as guidance ("prefer Redux for state management in this project because X, but reconsider for isolated components where simpler state suffices") can prevent the agent from discovering better approaches for new problems.

### The maintenance gap

Maintenance is the weakest area in the entire judgment memory space. Everyone acknowledges that stale judgment is dangerous; nobody has a complete answer for keeping it fresh.

What exists: manual review cycles (the Codified Context paper reports biweekly review, roughly 30-45 minutes), context drift detection (a script that flags when code changes without spec updates), memory decay (stale entries get flagged rather than blindly trusted), convention trend tracking (rising and declining adoption percentages), CI enforcement (ADR rules checked at merge time), and size caps (the 200-line guideline for CLAUDE.md).

What does not exist: systematic pruning methods, contradiction detection across multiple context files, quality measurement for judgment memory (does this context actually improve agent output?), or decay curves (how quickly do different kinds of judgment go stale?).

The Codified Context paper reports the most honest maintenance data: 1–2 hours per week for a single-developer project with a 108,000-line codebase. Whether this scales linearly or superlinearly to teams is unknown. The key insight from Vasilopoulos's experience report: "AI automation consistently eliminated tedium but not judgment." The maintenance of judgment memory is itself a judgment task.

## The task shape: small, isolated, reviewable

The shape of the work unit is one of the most consequential design decisions in agent-assisted development, and one of the hardest to get right because the agent exerts constant pressure toward larger, less bounded units.

The principle is simple: tasks should be small enough that the diff is genuinely reviewable, isolated enough that concurrent work does not collide, and bounded enough that failure is detectable before it compounds. Several sources articulate specific aspects:

- **Several small PRs beat one big one** (Willison). A thirty-file change is not reviewable in any meaningful sense. It is a transfer of responsibility from the developer to the reviewer, and the reviewer could have prompted an agent themselves.
- **Never let two agents dig the same trench** (NK). Concurrent agents touching the same files produce merge conflicts and cascading failures. Give each agent its own isolated workspace, branch, or worktree.
- **Re-plan when requirements change** (NK). Do not let an agent continue building when you have had an insight that invalidates the plan. Stop, update the plan, restart. Continuing from a broken foundation produces subtly wrong code that is harder to diagnose than a clean restart.
- **Right-size the step** (Harper Reed). His planning prompt explicitly asks the model to iterate on step granularity until each step is small enough to be safe and big enough to be meaningful.

The anti-pattern is seductive and common: prompting for a large feature, watching the agent produce thousands of lines of apparently correct code, and committing or opening a PR without genuinely understanding the structural choices embedded in the output. Willison names this explicitly: *inflicting unreviewed code on collaborators*.

## Skills and hoarding: making expertise persistent

One of the most frustrating aspects of agent-assisted work is the reset problem: you correct the agent, it adjusts, and in the next session it makes the same mistake because the correction was ephemeral. Human memory degrades and drifts; file-based encoded knowledge does not.

Two complementary strategies address this:

**Skills as persistent context injection** (NK). Every time you find yourself giving the same instruction twice, encode it as a skill file. Skills are auto-selectable, permanent, and accumulate compound interest over time. The agent that loads your architecture review skill at the start of every session has access to your years of judgment without you needing to remember to state it. The practical artifacts are CLAUDE.md, AGENTS.md, skills directories, `.claude/rules/` files, and `.cursorrules`. They are all instances of the same pattern: encode your judgment in a form the agent consumes automatically.

**Hoarding working examples** (Willison). Maintain a personal library of code that works — things you have figured out how to do, prototypes you have validated, integration patterns you have proven. When you need to build something new, find the two nearest existing examples in your hoard and tell the agent to combine them. This is dramatically more reliable than asking the agent to figure out a library from documentation alone, because documentation is often incomplete, outdated, or subtly misleading, while working code is ground truth.

Both strategies share the same insight: the most valuable form of context for an agent is not documentation or explanation; it is encoded experience — the accumulated record of what has actually worked, what has failed, and why.

## Legacy code: where agents are most tempting and most dangerous

Michael Feathers's work on legacy code is especially relevant because agents are most tempting to use recklessly in exactly the codebases where recklessness is most expensive: large, poorly understood systems where behavior is unclear and tests are sparse.

The danger is straightforward. An agent can refactor, restructure, and "clean up" legacy code with impressive speed. But if no one understands the existing behavior precisely — including the agent — then every change is a gamble. Feathers's core contribution is to make changeability the goal rather than perfection: get the code under test, find seams (points where you can alter behavior without modifying the code directly), and make safe, incremental progress without requiring a rewrite.

Characterization testing is the key technique: when you do not know what a piece of code is *supposed* to do, write tests that describe what it *actually* does. Run the code, observe the output, and lock it in with a test. Now you have a safety net against unintended changes, even if you do not understand the intent behind the current behavior.

This approach maps naturally onto agent-assisted work in legacy systems:

1. Before letting the agent modify anything, have it write characterization tests for the area under change.
2. Confirm that the tests pass against the current code.
3. Now let the agent make changes, with the tests as a safety net.
4. Review the diff for intent, not just correctness.

The worst failure mode is letting the agent refactor legacy code without characterization tests. The agent produces clean, well-structured code that behaves differently from the original in subtle ways that no one detects until production.

## Multi-agent work: when parallelism helps and when it hurts

As agent-assisted development scales from a single agent to multiple concurrent agents, the quality problem compounds. The core issue, identified clearly by Cognition, is that parallel agents diverge because actions carry implicit decisions that other agents cannot see. Agent A chooses to use a factory pattern for object creation; Agent B, working on a related module, creates objects directly. Both are locally correct. Together they produce an inconsistent system.

The emerging consensus, drawn from both Anthropic's multi-agent research system and Cognition's counterargument, is that the distinction matters more than the choice: **multi-agent for breadth-first exploration** (research, parallel search, independent investigation) and **single-agent for depth-first coherence-heavy execution** (coding, sequential design work where every decision constrains the next).

HYBRD Engineering's experience sharpens this further: supervision is too deterministic a job to leave to an LLM. Task dispatch, dependency checking, nudging stuck workers, and policy enforcement are better implemented in code for lower token cost, better observability, and less behavioral drift. Their recommendation: use code for the control plane, agents for the work plane. A deterministic orchestrator dispatches tasks, enforces constraints, and tracks progress; the agents do the creative work within the boundaries the orchestrator defines.

For multi-agent quality specifically, the requirements include:

- **Isolation.** Each agent gets its own workspace, branch, or worktree. No shared mutable state.
- **Shared context.** All agents in a project load the same judgment memory (constitution, conventions, architectural decisions). Divergence often stems from different agents operating with different context.
- **Explicit coordination.** Work tracking through structured artifacts (Beads, task lists, issue graphs) rather than through chat or implicit convention.
- **Automated conflict detection.** Merge-level checks, test suites, and conformance tools that detect when parallel work has produced inconsistencies.

## Security as a quality concern

Agent security is a quality concern, not a separate topic. Simon Willison's lethal trifecta identifies the core risk: an agent that combines private data access, external communication capability, and exposure to untrusted content creates a prompt injection surface that can be exploited for data exfiltration. Each capability alone is manageable; all three together are currently unsolved.

The practical implications for quality-conscious agent use:

- **Be intentional about tool grants.** Each tool you give an agent expands the attack surface. Grant the minimum capabilities needed for the task.
- **Treat untrusted content as adversarial.** Agents reading emails, web pages, or user-submitted content while having write access to production systems is the threat model.
- **Resist YOLO mode normalization.** Running agents without confirmations is addictive. Each skipped safety check that does not cause harm makes the next one easier to skip. Willison draws the Challenger analogy: normalization of deviance.
- **Design for constrained consequences.** Assume some manipulative inputs will get through, and architect the system so that dangerous consequences are constrained anyway — through limited-action pathways, confirmation steps, and source-sink analysis.

OpenAI's prompt injection defense guide reframes the problem usefully: prompt injection is a social engineering problem, not a string-filtering problem. The defense is architectural, not perimeter-based.

## A synthesis of the workflow

Pulling together the practices described above into a coherent operating model:

### Before the session
- Ensure your judgment memory is current: CLAUDE.md, AGENTS.md, ADRs, and relevant rules files reflect the actual state of the project.
- Identify the fidelity level of the work: quick fix, planned implementation, or exploratory prototype.
- For planned work, write a specification and have the agent produce a sequenced implementation plan. Review the plan.

### During the session
- Load appropriate context: hot (always), warm (per-task), cold (on demand).
- Work in small steps. One change at a time. Run tests after each.
- Review every diff for domain fit, module depth, simplicity (not just ease), and dependency direction.
- In legacy code, write characterization tests before making changes.
- Monitor for context rot. Start a fresh session rather than patching a degraded one.
- When the plan proves wrong, stop and re-plan rather than continuing from a broken foundation.

### After the session
- Refactor for clarity and habitability.
- Record significant design decisions as ADRs.
- Capture any judgment that would prevent future agents from making mistakes you corrected during this session.
- Ask: is the next change now easier or harder?
- Commit in small, logical steps.

### Periodically
- Review judgment memory for staleness, bloat, and contradiction.
- Prune context files: remove instructions that are no longer relevant, consolidate duplicates, sharpen rationale.
- Verify that automated checks (tests, linters, conformance suites) still cover the areas of the codebase that agents touch most.
- Review whether agent-produced architecture is drifting from domain-shaped toward framework-shaped.

## The review checklist

A compact checklist for reviewing agent-produced code, synthesized from the design principles described above:

- **Domain fit.** Do names and boundaries reflect the problem domain, or the framework and the prompt?
- **Module depth.** Do modules hide complexity behind simple interfaces, or merely delegate to the next layer?
- **Dependency direction.** Do important policies depend on implementation details?
- **Simplicity.** Is this actually simpler — fewer entanglements, less state to track — or merely easier to generate?
- **Test support.** Is there enough automated checking to make future change safe?
- **Changeability.** Would the next change now be easier or harder?
- **Decision memory.** If the change is structural, is the rationale recorded?
- **Habitability.** Would another developer feel comfortable entering this code?
- **Consistency.** Does this change contradict conventions or decisions already captured in judgment memory?
- **Proportionality.** Is the scope of change proportional to the task, or has the agent over-generated?

## What remains uncertain

Honesty about the limits of current evidence:

- **The productivity question is not settled.** The METR study shows that real-world effects can diverge sharply from expectations. Self-reports and benchmarks are not sufficient evidence of productivity improvement in high-context codebases.
- **Judgment memory maintenance is an open problem.** No source describes a systematic, principled approach to pruning, contradiction detection, or quality measurement for agent context. The current best practice — manual periodic review — does not obviously scale.
- **Team-scale evidence is thin.** Almost all evidence in this collection is from single developers or very small teams. Whether judgment memory, context engineering, and feedback-loop discipline scale to larger teams is unknown.
- **The stale-judgment risk is real and underappreciated.** Agents trust documentation absolutely. When documentation is wrong, the agent produces confidently wrong code. The rate at which judgment memory goes stale relative to the rate at which it is maintained is not well characterized.
- **The emotional and organizational costs are real.** Supervision fatigue, work intensification, and the loss of the satisfying aspects of craft are not merely complaints; they affect retention, quality of judgment, and the sustainability of agent-heavy workflows.
