# Judgment memory research plan

## Intent

Design a research process for investigating the emerging concept currently being called **judgment memory** in this repo.

The goal is **not** to force that label too early or to jump straight to synthesis. The goal is to:

1. find the different terms people are using for similar ideas,
2. collect concrete primary sources and inspectable artifacts,
3. separate adjacent-but-different concepts,
4. produce a terminology and source map that can support a later synthesis.

This plan is written so an autonomous agent with the same capabilities as the current assistant can execute it.

---

## Working definition

Use this only as a **temporary operational definition** during research:

> Judgment memory is durable captured judgment that changes how future agents plan, implement, review, or decide.

This includes things like:
- code review heuristics,
- architectural preferences,
- coding conventions with reasons,
- lessons extracted from failures,
- reusable reviewer prompts,
- solved-problem writeups that shape future work.

It does **not** primarily mean:
- generic long-context memory,
- chat history,
- vector search over arbitrary docs,
- task tracking alone,
- generic note-taking that never affects future behavior.

This definition is provisional and may be revised after terminology gathering.

## Failure modes to watch for

Explicitly look for evidence of these:
- stale encoded preferences,
- duplicated or conflicting guidance,
- rules without rationale,
- context bloat from too much persistent guidance,
- overfitting to past mistakes,
- local taste calcifying into dogma,

- missing ownership for updates.

This is important because a concept built only from success cases will be too vague.

## Anti-hype rules

1. Prefer builders and inspectable artifacts over commentary.

4. Distinguish clearly between:
