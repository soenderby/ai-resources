# Patterns in LLM Agent Usage

> Generated by Claude on 2026-03-09.  
> Synthesized from sources in [resources/](../resources/) — practitioners, researchers, and academics converging on similar ideas from different directions.

---

## 1. Context Engineering Is the Whole Game

Every source touches this. Karpathy's framing is the clearest: **the context window is RAM, the LLM is the CPU**. Every session starts with amnesia. Your job as the operator is to fill that RAM correctly before asking the CPU to do anything.

What this means in practice:

- **Input/output ratio as a quality signal.** NK's Cookbook: if your input is less than 20% of total conversation tokens, you're under-specifying. A 4,000-line code output from a three-line prompt means the model is guessing where you should be telling.
- **Front-load everything.** Meeting notes, ticket threads, constraints, prior attempts, what's fixed vs. fuzzy — dump it all in before you ask for anything. The model doesn't get confused by complexity. It gets confused by ambiguity.
- **Context rot is real.** Willison coined the term: as the context window fills with back-and-forth, quality degrades. The model starts losing specifics, getting vague, drifting. Recognize when this is happening and start a fresh session rather than continuing to patch.
- **Skills as persistent context injection.** The solution to starting every session blank is to encode your accumulated knowledge in reusable skill files that get loaded automatically. NK: "every time you find yourself giving the same instruction twice, that's a signal — encode it as a skill." Unlike humans, skills don't forget.

## 2. Separate Thinking from Doing

Harper Reed's three-loop workflow, NK's "plan before coding", Willison's red/green TDD, and Yegge's AGENTS.md onboarding are all variations of the same pattern: **force the model into a planning mode before it touches anything**.

The concrete workflow Harper Reed uses:
1. **Brainstorm → Spec**: Use a conversational model, one question at a time, to extract a complete spec. End with: *"compile our findings into a developer-ready specification."*
2. **Spec → Plan**: Feed the spec to a reasoning model. Ask it to produce a sequenced set of implementation prompts, each small, each building on the previous, with tests. This is the artifact you review — not the code.
3. **Plan → Execute**: Feed the prompts one by one to a coding agent.

NK's framing of *why* this matters: in the old world, quality assurance happened at code review. In the agentic world, the code itself is often fine — the failures are architectural decisions that should have been caught before a line was written. **Quality assurance has moved upstream.** Review the plan, not the code.

Willison's version of this is red/green TDD: write the tests first, confirm they fail, then let the agent implement until they pass. The tests *are* the plan — they're executable, unambiguous, and the agent can verify its own work against them.

## 3. Zero Framework Cognition (ZFC)

Yegge's most transferable principle. The pattern: **build a thin, deterministic shell that handles IO and policy, and delegate all reasoning to the model.**

What belongs in the shell (ZFC-compliant):
- File IO, parsing, serialization
- Schema validation, safety checks, budget caps, rate limits
- State tracking, progress monitoring
- Mechanical transforms (parameter substitution, formatting)
- Typed error handling using SDK error classes

What must go to the model (ZFC violations if done in code):
- Ranking, scoring, selection among alternatives
- Keyword matching or regex-based classification
- Inferring complexity, scope, or what should happen next
- Dependency ordering, retry policy decisions
- Any heuristic that substitutes for genuine reasoning

The failure mode ZFC prevents: agents instinctively write local heuristics instead of calling back to the model, because calling the model feels expensive and uncertain. But a keyword list for detecting completion states ("finished", "done", "complete") will miss edge cases your regex never anticipated. The model handles this trivially. **When you reach a decision point, send it to the model.**

The theoretical ancestor is Martin Fowler's "Smart Endpoints and Dumb Pipes" from microservices — the intelligence lives at the edges, not in the plumbing. Yegge maps ZFC directly onto this. Unix's separation of mechanism from policy is the same idea, older still.

## 4. The Loop Pattern and When to Abandon It

Huntley's RALPH pattern and the broader practitioner consensus: **start with a monolithic loop before reaching for multi-agent complexity.**

The loop:
1. Gather raw context (IO only)
2. Call AI for decisions
3. Validate structure (schema, safety)
4. Execute mechanically
5. Repeat

Huntley's argument against premature multi-agent systems: *"Consider microservices and all the complexities that come with them. Now consider what microservices would look like if the microservices themselves are non-deterministic — a red hot mess."* Non-deterministic agents communicating over unclear protocols compound errors in ways that are nearly impossible to debug. Scale vertically first (make the loop better) before scaling horizontally (multiple agents).

Yegge's Gas Town represents the point where horizontal scaling becomes justified: once a single loop is stable, reliable, and well-understood, you can run many of them in parallel. But Gas Town explicitly required months of a stable single-agent loop before federation made sense.

**When to move from loop to multi-agent:** when tasks are genuinely independent (different files, different concerns), when they're parallelisable without stepping on each other's work, and when you have sufficient automated feedback to detect when agents conflict.

## 5. Automated Feedback Extends Task Horizon

Huntley's "back pressure" post, Karpathy on RLVR, Willison on conformance suites — all pointing at the same thing: **the more automated, verifiable feedback you give an agent, the longer-horizon tasks it can reliably complete.**

This is why coding is currently the most successful domain for agents. Code is verifiable. Tests either pass or fail. Linters either complain or don't. The agent can check its own work.

The pattern: wherever you can provide an agent with an environment where it can verify correctness itself, do it. This is more valuable than better prompts.

Practical forms:
- **Tests first** (Willison): the agent runs tests, sees failures, iterates. Executable truth.
- **Linters and validators** (NK): don't just tell the agent your conventions — enforce them mechanically so it can't claim "done" when they're failing.
- **Pre-mortems** (NK's Cookbook): ask a fresh session to assume the plan has catastrophically failed and diagnose why. Catches specification gaps.
- **The agent reviews the review** (NK): after an automated review tool flags issues, ask the coding agent to review the review skeptically. Third-opinion effect.

Karpathy's RLVR observation at the model-training level maps onto this exactly: models that are trained against *automatically verifiable rewards* improve dramatically compared to those that rely on human feedback. The same principle applies to your agent workflows: verifiable > human-judged.

## 6. Keep Tasks Small, Isolated, and Reviewable

This pattern appears everywhere, usually learned from painful experience:

- **Several small PRs beats one big one** (Willison). An agent that has modified 30 files has produced something almost impossible to review meaningfully. Small changes, committed separately, are reviewable.
- **Never let two agents dig the same trench** (NK). Concurrent agents touching the same files produce merge conflicts and cascading failures. Give each agent its own isolated workspace or branch.
- **Break the plan into steps small enough to be safe, big enough to move forward** (Harper Reed). His planning prompt explicitly asks the model to iterate on step size until right-sized.
- **Re-plan when requirements change** (NK). Don't let an agent continue building when you've had an insight. Stop it, update the plan, restart. Continuing from a broken foundation produces subtly wrong code that's hard to diagnose.

The anti-pattern Willison names explicitly: **inflicting unreviewed code on collaborators**. If you open a PR with thousands of lines of agent-generated code you haven't reviewed, you're delegating your actual job to your reviewers. They could have prompted an agent themselves.

## 7. Skills and Hoarding as Compound Investment

Willison's "Hoard things you know how to do" and NK's skills library are the same insight from different angles.

Willison's version: maintain a personal library of working code examples — things you've figured out how to do. Prototype freely, keep the results. When you need to build something new, find the two nearest existing examples in your hoard and tell the agent to combine them. This is dramatically more reliable than asking the agent to figure out a library cold.

NK's version: every time you give the same instruction twice, encode it as a skill. Skills are auto-selectable, permanent, and accumulate compound interest over time. The agent that loads your architecture review skill from a file has access to your years of judgment at the start of every session.

The asymmetry: **humans forget, skills don't.** Correcting an agent and then having it drift back to old behavior in a later session is one of the most common frustrations. A skill file that gets loaded every time doesn't drift.

The practical form: AGENTS.md (Yegge), CLAUDE.md (this repo), skills directories (pi, Claude Code). These are all the same pattern — encode your judgment in a form the agent can consume automatically.

## 8. The Human Role Has Shifted, Not Disappeared

The Pydantic article, Willison's "code is cheap, good code isn't", and NK's operator metaphor all describe the same shift: **the human has moved from producer to quality gate.**

You are no longer the bottleneck at code production. You are the bottleneck at:
- Holding coherent intent across a complex change
- Deciding *what* to build (feature discipline becomes more important when you can build anything)
- Judging quality at higher throughput
- Catching subtle architectural failures before they accumulate

The failure mode is treating this as a passive role. NK: "the context window starts blank every time — your entire job as the operator is to fill that room with the right information." The model amplifies skilled operators and is dangerous for unskilled ones. Expertise doesn't become less valuable; it becomes the only scarce thing.

The cognitive hazard: because the agent is so capable and fast, it's tempting to let it run and check it later. This is where drift, hallucination, and accumulating technical debt come from. The operator role requires active engagement at the planning stage, not just review at the end.

## 9. Exploration Over Single-Path Execution

Less commonly named but surfaces clearly in Residue and Yegge's Gas Town. The pattern: **when the solution space is uncertain, run multiple independent explorations rather than committing to one path early.**

The Residue project uses this explicitly for mathematical proofs — the problem space is too large and too uncertain to commit to a single approach, so agents explore different strategies in parallel, and the results inform which direction to pursue.

Gas Town's "Wanted Board" is a scaled version: ideas go up, multiple agents (and contributors) attempt approaches, the community stamps what works.

At the individual level, this means: instead of asking one agent to solve a hard problem, ask two or three independently, compare approaches, and synthesize. The cost of running parallel agents has dropped dramatically. The cost of going down the wrong path has not.

## 10. The Security Surface Grows With Capability

Willison's "lethal trifecta": an agent becomes dangerous when it has all three of **private data access + external communication capability + exposure to untrusted content**. Each alone is manageable. All three together create a prompt injection attack surface that's currently unsolved.

The practical implications:
- Be intentional about what tools you give an agent. Each tool expands the attack surface.
- Agents reading untrusted content (emails, web pages, user input) while having write access to production systems is the threat model.
- "YOLO mode" (running agents without confirmations) is addictive and normalizes deviance — Willison's Challenger analogy. Each skipped safety check that doesn't cause harm makes the next one easier to skip.

This is one area where the practitioner community is notably ahead of the tooling. The security properties of agentic systems are much less understood than the capability properties.

## The Common Thread

Almost everything above is a variation of two principles:

1. **The model is the intelligence. You are the architect of its environment.** Your job is to give it the right context, the right tools, verifiable feedback, and small enough tasks that failures are detectable and correctable. Don't encode intelligence in the framework code — that's what the model is for.

2. **The constraints that shaped software engineering still apply, but they've shifted.** Code is cheap; good code isn't. Small tasks beat large ones. Isolation prevents interference. Tests are truth. Expertise matters more, not less. None of this is new — but the point where these principles apply has moved upstream, from writing to planning.
