# Document Review Process for `ai-resources`

## What This Document Is

A record of the review process used on recent AI-generated documents in this repository, and a set of guidelines for reviewing future documents.

This is meant to help future work stay consistent. The repo now has enough generated synthesis and design material that document quality cannot be left to first drafts.

---

## Why This Matters

The recent documents were often good on first draft, but not good enough to keep unchanged.

The recurring pattern was:
- first drafts were usually **strong on coverage**
- but often **too long**
- sometimes **too abstract**
- sometimes **repetitive**
- and occasionally missing the one concrete example or design decision needed to make the whole thing truly useful

The review process improved them materially. It did not just polish wording. It clarified:
- what the document was actually claiming
- what level of confidence it had
- what structure was useful
- what should be cut
- what needed to become more concrete

That review process should now be treated as part of the normal workflow.

## The Review Process Used So Far

The process that emerged was iterative but fairly consistent.

## 1. Write the first useful draft

The first draft should aim for:
- clear scope
- meaningful synthesis
- explicit structure
- enough detail to expose the real design questions

It does **not** need to be minimal on the first pass. In practice, first drafts were often allowed to be somewhat long because that made hidden assumptions visible.

## 2. Get critical feedback

The most useful reviews did not ask, “is this good?”
They asked more pointed questions such as:

- Is this too abstract?
- Is this longer than it needs to be?
- Where is it repeating itself?
- What is missing for this to be usable rather than merely plausible?
- Does it include one concrete worked example?
- Is the structure helping, or just sounding organized?

Good review feedback usually attacked one of four things:
- **missing concreteness**
- **repetition**
- **over-claiming**
- **weak operational usefulness**

## 3. Revise for specificity before elegance

The strongest revisions were the ones that made the document more concrete, not just smoother.

Typical improvements included:
- adding a worked example
- adding explicit criteria or thresholds
- turning philosophy into design rules
- turning recommendations into phased next steps
- distinguishing what is known from what is uncertain

This mattered more than stylistic cleanup.

## 4. Tighten aggressively once the core idea is sound

Once the central idea was clear, the next pass usually needed to remove:
- repeated framing
- duplicated conclusions
- parallel sections saying the same thing in different words
- overlong introductions
- grand summaries that merely restated the document

A recurring lesson: many documents became stronger after cutting **20–40%**.

## 5. Preserve the key distinction or contribution

Every strong document ended up having one main contribution that had to survive revision.

Examples from recent work:
- **knowledge gardening** → canonical vs derived layers
- **memory for agents** → task memory / context memory / knowledge memory
- **adjacent fields** → retrieval / representation / maintenance
- **concept index** → concepts as navigational handles, not ontology

The review process should protect that core contribution while cutting everything around it that does not strengthen it.

## 6. Add explicit uncertainty where needed

One of the better habits that emerged was to include sections like:
- what seems strongest
- what is still uncertain
- where evidence is weak
- what still seems missing

This is especially important in a fast-moving field. It makes the documents more trustworthy and more useful for future synthesis.

## 7. Commit only after the document feels structurally right

The process used here was often:
- draft
- critique
- revise
- sometimes critique again
- then commit

That is worth preserving. It is better to commit the reviewed version than to commit a weak draft and hope to revisit it later.

## Common Failure Modes in First Drafts

These appeared repeatedly.

## 1. Coverage without enough discrimination

The draft contains many true things, but they are not well ranked. The reader cannot tell what matters most.

### Fix
- add an executive summary
- state the main claim earlier
- rank the strongest conclusions explicitly

## 2. Abstract architecture with no worked example

The design sounds plausible, but there is no concrete unit-level example to test whether it would actually be useful.

### Fix
- add one fully worked example
- derive design rules from the example

## 3. Repetition disguised as structure

Different sections restate the same idea in slightly different language.

### Fix
- merge overlapping sections
- cut repeated framing
- keep one clear formulation of the core idea

## 4. Strong thesis, weak operational consequence

The document explains the world well but does not help decide what to do next.

### Fix
- add explicit next steps
- add criteria or thresholds
- add implications for this repo specifically

## 5. Conceptual clarity without maintenance thinking

The document proposes structure but does not ask what it will cost to keep that structure useful.

### Fix
- ask what needs ongoing maintenance
- ask what should be pruned
- distinguish exploratory from stable artifacts

## Guidelines for Future Documents

These should be treated as the default quality bar.

## 1. Every document should have a clear scope statement

At the top, say what the document is and what it is **not**.

Useful distinctions:
- synthesis vs design note
- scoped view vs field-wide survey
- exploratory artifact vs stable reference

This prevents over-claiming.

## 2. Every document should have one main contribution

A reader should be able to answer:
- what is the one thing this document clarifies?

If the answer is “several things,” the document is probably too diffuse.

## 3. Prefer concrete distinctions over broad summaries

Good documents usually hinge on a sharp distinction, for example:
- canonical vs derived
- task memory vs context memory vs knowledge memory
- retrieval vs representation vs maintenance

These are better than generic summaries because they become reusable mental tools.

## 4. Add one worked example when structure is being proposed

If a document proposes a new layer, format, or framework, include one concrete example.

This is especially important for:
- concept layers
- schemas
- maintenance protocols
- query tools

The example should force answers to practical questions like:
- how granular is this?
- how much annotation is enough?
- where are the boundaries?
- how much maintenance would this require?

## 5. Include uncertainty explicitly

Where relevant, include:
- strongest evidence
- weak evidence
- what is still unknown
- what additional sources or experiments would strengthen the claim

This is part of the repository’s anti-hype standard.

## 6. Make repo-specific implications explicit

If a document is about a broad topic, it should still answer:
- what does this imply for `ai-resources`?

Otherwise it risks becoming merely a general essay.

## 7. Review for repetition as a separate pass

Do not try to solve structure, content, and concision all at once.

A good pattern is:
1. get the ideas right
2. make the structure clear
3. do a dedicated concision pass

That third pass matters a lot.

## 8. Generated documents should usually optimize for usefulness, not completeness

A document does not need to say everything. It needs to:
- capture the key idea
- support future thinking
- be easy to revisit
- stay close to the repo’s actual source base

## 9. If a document suggests a process, it should include review triggers or stop conditions

This emerged strongly from the experiment work.

Examples:
- when to stop collecting sources
- when to revise a synthesis
- when to prune a concept
- when a structure should be promoted or retired

Without this, the process remains too vague.

## 10. Treat maintenance as part of design

If a document proposes a new structure, ask immediately:
- who or what maintains this?
- how does it decay?
- when should it be pruned?
- how will we know if it stopped being useful?

This should become a standard review question.

## A Practical Review Checklist

Before finalizing a future document, check:

### Scope
- Does it clearly say what it is and is not?
- Is the level of ambition honest relative to the source base?

### Core contribution
- Is there one main idea or distinction?
- Is that idea visible early?

### Concreteness
- Is there at least one worked example if needed?
- Are there explicit criteria, thresholds, or design rules where appropriate?

### Evidence and uncertainty
- Does it distinguish strong claims from weak ones?
- Does it say what is still missing?

### Repo usefulness
- Does it help future retrieval, synthesis, curation, or tool design?
- Does it say what changes in this repo as a result?

### Concision
- Are there sections repeating the same point?
- Can 20–30% be cut without losing meaning?

### Maintenance
- If it proposes structure, does it account for upkeep and pruning?

If several of these answers are “no,” the document is not ready.

## Recommended Default Workflow for Future Documents

A good default process now looks like this:

1. **Draft** the document with enough detail to expose the real structure
2. **Critique** it for abstraction, repetition, and missing examples
3. **Revise for specificity**
4. **Tighten for concision**
5. **Check uncertainty and repo-specific implications**
6. **Commit only when the structure feels right**

This is slightly slower than writing once and moving on, but the result is materially better.
