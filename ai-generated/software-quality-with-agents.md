# Software quality with agents

## Thesis

Agents do not remove the need for software design discipline. They increase the need for it.

When code generation gets cheaper, the bottleneck moves upward:
- deciding what should exist
- choosing the right boundaries
- preserving conceptual integrity
- keeping complexity under control
- maintaining confidence while changing the system
- recording why important decisions were made

The core question is not **"How do agents write good code?"**
It is **"How do humans and agents work together so the system stays understandable, changeable, and trustworthy?"**

## What the source cluster says

### 1. Quality is mostly about changeability
Across Beck, Parnas, Ousterhout, Brooks, Feathers, and Gabriel, the common thread is that good software is software that can be changed safely.

That implies more than surface cleanliness. It means:
- understandable structure
- tests or other checks that preserve confidence
- interfaces that hide complexity instead of exporting it
- names and modules that reflect the domain
- a codebase future people can enter without dread

In an agent-heavy workflow, this matters even more because large volumes of plausible code can be produced faster than humans can build understanding.

### 2. Decomposition matters more than generation
Parnas is still one of the best guides here. The key move is to decompose around hidden decisions and likely sources of change, not around the steps of a workflow.

Bad agent-generated modularity often looks like:
- controller layer
- service layer
- repository layer
- helper layer
- util layer

That can be tidy without actually hiding anything.

Better modularity hides volatile decisions such as:
- storage choice
- protocol choice
- vendor integration
- policy variation
- coordination logic

Ousterhout sharpens the same point from another angle: good modules are **deep**. They absorb complexity and present simple interfaces. Generated code often produces the opposite: many shallow layers that still force callers to know too much.

### 3. Small feedback loops beat large generation bursts
Beck's TDD and XP work remains highly relevant. The right pattern is not:
- prompt for a large feature
- let the agent generate a big blob
- review at the end

The better pattern is:
1. state intent clearly
2. define or refine a failing check
3. ask for one small change
4. review the diff
5. refactor
6. run the tests
7. repeat

Agents make this loop faster. They do not make it obsolete.

### 4. Simple is not the same as easy
Rich Hickey's distinction becomes more valuable in the age of coding agents.

Easy code is:
- familiar
- convenient
- close at hand
- quick to generate

Simple code is:
- not tangled
- not interleaved
- easier to reason about over time

Agents can generate easy code at scale. That does not mean they are producing simple systems. A lot of agent output looks clean because it is symmetrical and pattern-shaped, while actually increasing indirection and coupling.

### 5. Domain fit is one of the best defenses against plausible garbage
Evans and CUPID fit together well here.

Framework-shaped code is often easy for an agent to produce:
- names mirror transport objects
- boundaries mirror framework folders
- logic mirrors database tables
- abstractions mirror technical layers

Domain-shaped code is easier for humans to verify and maintain:
- names match the business language
- boundaries reflect real distinctions in the problem
- important invariants are visible
- architecture follows the domain more than the tools

This is one of the strongest practical quality filters for agent-produced software.

### 6. Preserve design memory deliberately
Nygard's ADR idea becomes more useful as the pace of change rises.

When many changes happen quickly, teams forget:
- why an interface looks awkward
- why a dependency was rejected
- what tradeoff a boundary protects
- which non-functional requirement motivated a decision

ADRs are valuable because they keep rationale close to the code in a small format that is realistic to maintain.

### 7. Optimize for habitability
Gabriel adds a dimension that purely structural advice often misses: software is a place people live in.

Habitability means:
- recognizable patterns
- names that make sense
- ease of navigation
- confidence when making a change
- room for piecemeal growth

This is a useful counterweight to judging generated code only by whether it runs or whether it looks conventionally clean.

### 8. AI does not abolish essential complexity
Brooks is still the right antidote to overclaiming. Better tools can reduce accidental complexity, but they do not remove the conceptual work of modeling, tradeoffs, coordination, and coherent system design.

Agents can help a lot with representation and local implementation.
They do not automatically solve the hard part.

## A practical operating model

### Human responsibilities
Humans should usually retain authority over:
- domain concepts and naming
- module boundaries
- architectural direction
- invariants
- risk and failure handling
- what should be simplified or deleted
- what counts as done

### Agent responsibilities
Agents are most useful for:
- drafting small implementations
- generating and expanding tests
- performing mechanical refactors
- making repetitive edits
- summarizing code paths
- exploring alternatives
- finding inconsistencies
- helping recover understanding in legacy code

This is not because agents are only good at low-level work. It is because the cost of being wrong is highest at boundaries, interfaces, and conceptual structure.

## Recommended workflow for high-quality software with agents

### Before coding
- describe the change in domain terms
- write down constraints and non-goals
- note any architectural decision if the change is structural
- define acceptance via tests, examples, or checks

### During coding
- ask for one small change at a time
- keep tests and linters running
- review names and interfaces, not just logic
- prefer deletion over abstraction when possible
- check whether modules hide complexity or leak it

### After coding
- refactor for clarity
- record significant design decisions
- verify that the next change would now be easier, not harder
- commit in small logical steps

## A review checklist for agent-produced code

- **Domain fit:** Do names and boundaries reflect the problem domain?
- **Depth:** Do modules hide complexity, or just move it elsewhere?
- **Dependency direction:** Do important policies depend on details?
- **Simplicity:** Is this actually simpler, or merely easier to generate?
- **Test support:** Is there enough checking to make future change safe?
- **Changeability:** Would the next change now be easier?
- **Decision memory:** If the change is structural, is the rationale recorded?
- **Habitability:** Would another developer feel comfortable entering this code?

## Productive tensions in the source set

### SOLID vs CUPID
SOLID is strongest as a dependency-management lens. CUPID is stronger as an outside-in test for whether code is pleasant to work with. They are better as complements than as rivals.

### Beck vs Ousterhout
Beck emphasizes design through test-driven feedback. Ousterhout worries more about avoiding complexity and shallow layering. Together they suggest a useful balance: iterate in small steps, but keep judging the emerging module structure from a complexity point of view.

### Brooks vs optimistic automation narratives
Brooks argues that software's core difficulty is largely conceptual. Agent tooling may cut more accidental complexity than older tooling did, but it still does not eliminate the need for modeling and coherent design.

## Bottom line

The best use of agents in software development is not to maximize code generation.
It is to accelerate safe evolution.

That means using agents inside a disciplined process that keeps systems:
- modular
- test-backed
- domain-shaped
- simple where possible
- explicit about decisions
- habitable for future humans

A good summary sentence is:

> Agents should accelerate software evolution, not accelerate complexity.
