## What is Context Pruning

Context pruning means selectively removing, truncating, or replacing parts of the conversation history before it reaches the LLM — without modifying the actual session on disk. The goal is to reduce token usage and cost by eliminating redundant or stale information from what gets sent to the model.

This is distinct from **compaction**, which summarizes the full history into a compressed form and is typically triggered when the context window is nearly full. Pruning is lighter-weight, rule-based, and can run on every request.

---

## How Pi Handles Tool Calls and Context

Pi stores sessions as tree-structured JSONL files. Every message (user, assistant, tool result, custom) is a tree node. Before each LLM call, Pi walks from the current leaf to the root, collecting all messages on the path, and sends them as the conversation context.

### Extension Points for Context Manipulation

1. **`context` event** — Fires before each LLM call. Receives a deep copy of all messages; you can filter, prune, or transform them. Session on disk is never modified. This is the primary hook for pruning.

2. **`tool_result` event** — Fires after each tool execution. Can modify the result content before it's stored (e.g., truncate aggressively).

3. **`session_before_compact` event** — Intercept compaction to provide custom summarization.

4. **Built-in compaction** — Triggers when context approaches the limit. Summarizes older messages, keeps recent ~20k tokens. A blunt instrument.

### How Pi Sends Requests to Providers

Pi's `pi-ai` package handles prompt caching automatically:

- **Anthropic**: Adds `cache_control: { type: "ephemeral" }` to tool definitions, the system prompt, and the last content block of the last user message. This means the full conversation prefix up to the latest message is eligible for caching. Supports configurable TTL via `PI_CACHE_RETENTION=long` (1 hour).

- **OpenAI**: Caching is fully automatic — no special markers needed. OpenAI's API automatically caches prompt prefixes ≥1024 tokens based on routing hash.

This is important because it means **Pi is already getting significant cache benefits on every request**. Any context modification must be weighed against the risk of breaking this cache.

## Existing Solutions

### For Pi

**[pi-context](https://github.com/ttttmr/pi-context)** (`pi install npm:pi-context`)
Agent-driven, git-like context management. Gives the LLM three tools: `context_tag` (mark milestones), `context_log` (view context skeleton + token usage), `context_checkout` (jump to a tag with a summary, using Pi's tree navigation). Philosophy: let the *agent* decide when to compress. Uses Pi's session tree for lossless time-travel. Not really pruning — more like structured compaction the agent controls.

**[pi-extension-observational-memory](https://www.npmjs.com/package/pi-extension-observational-memory)** (`pi install npm:pi-extension-observational-memory`)
Custom compaction replacement with observational memory patterns (Observations / Open Threads / Next Action Bias). Observer/reflector two-threshold system with auto-triggering. Not pruning — an improved compaction strategy.

**OpenClaw's context-pruning** — SDK-level integration with cache-TTL based pruning. Not published as a package.

### For OpenCode (Inspiration)

**[opencode-dynamic-context-pruning (DCP)](https://github.com/Opencode-DCP/opencode-dynamic-context-pruning)** — The most sophisticated approach found. Three categories of strategies:

| Strategy | How it works | LLM Cost |
|----------|-------------|----------|
| **Deduplication** | Same tool + same arguments → keep only the most recent output | Zero |
| **Supersede Writes** | If a file was written then later read, the write content is redundant | Zero |
| **Purge Errors** | Remove inputs from errored tool calls after N turns (keep the error message) | Zero |
| **Compress** | LLM-driven tool the model calls to summarize a *range* of messages | LLM call |

Key design principles: session history is never modified; protected tools are never pruned; dedup and error purging run automatically. Reports ~85% cache hit rate vs ~90% without (acceptable trade-off in their assessment).

## How LLM API Caching Works

Understanding this is critical to evaluating whether pruning is net-positive.

### Core Mechanism: Prefix Matching

Both Anthropic and OpenAI use **exact prefix matching**. The conversation is processed as a sequence: tools → system prompt → messages. The API checks whether the beginning of this sequence matches a recently cached version. If it does, the cached portion is reused.

**Crucially**: any change to any message in the middle of the sequence invalidates the cache for everything after that point. If you modify message #5 in a 20-message conversation, messages 1-4 may still cache, but 5-20 must be reprocessed.

### Anthropic Caching

- **Explicit opt-in**: Requires `cache_control` markers on content blocks (Pi adds these automatically)
- **Up to 4 cache breakpoints** per request (Pi uses: tools, system prompt, and last message)
- **TTL**: 5 minutes default, refreshed on each use; 1-hour option available (Pi's `PI_CACHE_RETENTION=long`)
- **Minimum**: 1,024 tokens for Sonnet/Opus; 2,048 for Haiku 3.5; 4,096 for Haiku 4.5
- **Pricing**:
  - Cache write: base price × 1.25 (25% premium)
  - Cache read: base price × 0.10 (90% discount)
  - Example at Sonnet 4 ($3/MTok): write = $3.75/MTok, read = $0.30/MTok
- **Breakeven**: 2nd request (the 25% write premium is recovered immediately on first cache hit)

### OpenAI Caching

- **Fully automatic**: No code changes required; works on all prompts ≥1024 tokens
- **Routing-based**: Requests are routed by prefix hash to the same machine that has the cache
- **TTL**: 5-10 minutes (in-memory); up to 24 hours with extended retention (GPT-5.x, GPT-4.1)
- **No write premium**: Unlike Anthropic, there's no extra cost to create the cache
- **Pricing**: Discount varies by model generation:
  - GPT-4o: 50% discount on cached tokens
  - GPT-4.1: 75% discount
  - GPT-5.x: 90% discount
  - GPT-5-nano: 90% discount

### What This Means for a Coding Agent Conversation

In a typical Pi session, each request to the LLM looks like:

```
[Tools] → [System Prompt] → [User msg 1] → [Assistant msg 1] → [Tool result 1] → ... → [User msg N]
```

Each turn appends new messages at the end. The prefix (everything before the new messages) is the same as last time. This means **the vast majority of tokens are cache reads** — the 90%-discount kind for Anthropic, 50-90% for OpenAI.

Pi's footer shows this in real-time: `total token/cache usage`. In a long session, you'll typically see cache reads dominating input token counts.

### Cache Processing Order (Anthropic)

```
1. Tools       ← cached together
2. System      ← cached (includes AGENTS.md, skills, etc.)
3. Messages    ← cached incrementally up to the last cache breakpoint
```

Changing tools invalidates system and message caches. Changing system invalidates message cache. This hierarchy means the most stable content goes first.

## Is Context Pruning Actually Beneficial?

This is the central question. The answer is nuanced: **it depends on what you prune, when you prune, and which provider you use.**

### The Cache Invalidation Problem

The fundamental tension: **pruning saves tokens but breaks caches**.

Consider a 50-message conversation with a duplicated `read` call at message #10 and #40. The deduplication strategy would remove message #10's tool result. But this changes the content of the prefix from message #10 onward — invalidating the cache for messages 10 through 50.

**Without pruning:**
- Messages 1-49 are cache reads (90% discount on Anthropic)
- Only message 50 (the new one) is full price
- Cost: very low for the cached portion

**With pruning (removing message #10):**
- Messages 1-9 are cache reads
- Messages 10-50 are cache writes (25% premium on Anthropic!) or full-price input
- Next request: messages 1-50 minus the pruned one are cached at the new shape

So the first request after pruning is **more expensive**, not less. You only break even if the cache stabilizes on the new shape and the total token savings from removing content outweigh the cache-miss penalty over subsequent requests.

### When Pruning IS Clearly Beneficial

1. **Subscription/flat-rate billing** (Claude Pro/Max, Copilot): No per-token cost. Any reduction in context size improves model attention and reduces latency. Cache invalidation is irrelevant to cost. **Pruning is purely beneficial here.**

2. **Request-based billing** (GitHub Copilot): No per-token cost either. Same as above.

3. **Very long sessions approaching context limits**: Pruning extends the useful life of a session before compaction triggers. Even if it costs more tokens short-term, it delays the lossy compaction event.

4. **Model quality/attention**: Large contexts degrade model performance (Anthropic calls this "context rot"). Removing truly irrelevant content can improve the quality of responses regardless of cost.

5. **Latency**: Fewer tokens = faster time-to-first-token, especially for the non-cached portion.

### When Pruning May HURT

1. **API billing with good cache hit rates**: If most of your tokens are cache reads at 90% discount, the absolute cost of those "redundant" tokens is already very low. Breaking the cache to save them can cost more than keeping them.

2. **Frequent pruning that changes on every request**: If the pruning decisions change turn-by-turn (e.g., progressive truncation that shortens old results each turn), you never stabilize the cache. Every request is a cache write.

3. **Small savings, high churn**: Deduplicating a 500-token read result saves $0.0015 per request (at Sonnet rates, cache-read pricing). But the cache-miss penalty for the remaining 50,000 tokens could be $0.175 (difference between cache-write and cache-read rates). That's a net loss of over 100x the saving.

### The DCP Approach: Batch Cache-Breaking Changes

OpenCode's DCP handles this intelligently: the zero-cost strategies (dedup, supersede writes, purge errors) are **not recalculated every request**. They are recalculated only when the compress tool runs. This means:

- Normal turn-by-turn requests: no pruning, full cache benefits
- When the model decides to compress: pruning runs alongside compression, accepting a single cache-miss event rather than continuous invalidation

This is a critical design insight. The cache is already being broken by the compression event, so bundling the pruning with it costs nothing additional.

### Cost Model: Worked Example

**Scenario**: 100-turn session on Anthropic Sonnet 4 ($3/MTok input, $0.30/MTok cached read, $3.75/MTok cache write). Average context size: 80,000 tokens. One duplicated 2,000-token read at turn 20 (read again at turn 60).

**Without pruning (status quo):**
- Each turn: ~78,000 cached tokens + ~2,000 new tokens
- Cache reads: 78,000 × $0.30/MTok = $0.0234
- New input: 2,000 × $3.75/MTok (cache write) = $0.0075
- Cost per turn: ~$0.031
- Over 40 turns (60-100): ~$1.24

**With pruning at turn 61 (remove duplicate read):**
- Turn 61: ~20,000 cached (prefix before change) + ~58,000 cache write (after change) + ~2,000 new
- Cache miss cost: 58,000 × $3.75/MTok = $0.2175 (vs $0.0174 if cached)
- Extra cost: ~$0.20
- Savings per subsequent turn: 2,000 × $0.30/MTok = $0.0006
- Turns to break even: ~333 turns

**Conclusion**: For a 2,000-token dedup on a long-lived session, it takes **hundreds of turns** to break even on API-billed Anthropic usage. The math gets worse for smaller savings and better for larger ones.

**But**: On a subscription plan, the break-even is instant (no per-token cost).

### Summary Table

| Factor | Pruning Helps | Pruning Hurts |
|--------|:---:|:---:|
| Subscription billing (Claude Pro/Max) | ✅ Always | — |
| API billing with good cache hit rates | — | ⚠️ Usually |
| Approaching context window limit | ✅ Delays compaction | — |
| Model attention/quality (long context) | ✅ Reduces context rot | — |
| Latency reduction | ✅ Less prefill | — |
| Very large redundant content (>10k tokens) | ✅ Even on API | — |
| Small redundancies (<2k tokens) on API | — | ⚠️ Cache penalty > savings |
| Batch pruning with compression events | ✅ Cache already broken | — |

## Decision Framework

### If Building a Context Pruning Extension for Pi

The evidence suggests a **tiered approach**:

**Tier 1: Always beneficial (any billing model)**
- Add a `/context` command showing token breakdown by category (like DCP's `/dcp context`)
- Track and display what *would* be prunable, without actually pruning — awareness is free
- Prune only during compaction events (batch with the cache break)
- Prune very large redundancies only (>10k token threshold)

**Tier 2: Beneficial for subscription users, risky for API users**
- Deduplication of tool calls (same tool + same arguments)
- Supersede writes (written then read → prune write)
- Purge old error inputs
- These should have a config toggle and clear cost-tracking so users can see the trade-off

**Tier 3: Agent-initiated (most flexible, most complex)**
- Register a compress tool the model can call to summarize a range
- Let the model choose when context has become stale enough to compress
- Bundle all automatic pruning with this event

### Key Implementation Constraint

**Use the `context` event, not `tool_result`.** The `context` event modifies what's sent to the LLM without changing what's stored in the session. This is essential — the session must remain the source of truth for branching, export, and resume.

### Open Questions

1. **Does Pi already batch cache breakpoints optimally?** Pi adds `cache_control` to tools, system, and the last message. If a `context` event handler modifies an old message, does the Anthropic cache find the longest matching prefix automatically (yes, per the 20-block lookback), or does the breakpoint placement mean the entire message chain after the change is a miss?

2. **What does Pi's actual cache hit rate look like?** The footer shows cache usage, but we don't have aggregate data. Understanding baseline cache rates would let us calculate the real cost of pruning.

3. **How much context is actually redundant in typical sessions?** Without measurement, we're theorizing. A `/context` diagnostic command would be the right first step.
