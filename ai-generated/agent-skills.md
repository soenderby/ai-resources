## What Are Agent Skills

A skill is a directory containing a `SKILL.md` file and optional supporting assets (scripts, reference docs, templates). When an agent starts, it scans skill directories, reads the frontmatter of each `SKILL.md`, and keeps those lightweight descriptions in its context. When a task matches a skill, the agent loads the full `SKILL.md` and follows its instructions — potentially executing scripts, reading reference files, or using templates bundled alongside it.

The key architectural insight, from a first-principles analysis of how Claude Code implements this: **there is no algorithmic routing**. The agent does not use regex, classifiers, or embeddings to decide which skill to invoke. The full list of skill names and descriptions is injected into the system prompt as text, and the model uses its own language reasoning to decide when a skill is relevant. The description field *is* the routing mechanism.

This is called **progressive disclosure**:
1. At startup: only name + description are loaded (lightweight, always present)
2. When the model decides a skill is relevant: full `SKILL.md` loads into context
3. During execution: helper scripts and reference files are read as needed

Skills are not executable code in the traditional sense. They are specialized prompt templates that inject domain-specific instructions into the conversation context and modify what tools the agent has permission to use. When invoked, a skill *prepares* the agent to solve a problem — it doesn't solve it directly.

### Skills vs. Other Mechanisms

| | Skills | CLAUDE.md / AGENTS.md | MCP Tools |
|---|---|---|---|
| **Scope** | Specific capability | Whole repo / project | Specific function |
| **Loaded when** | Agent decides it's relevant | Always | Called explicitly |
| **Contains** | Instructions + scripts + assets | Conventions + context | Code that executes |
| **Portable** | Yes, across machines and agents | Project-scoped | Requires server |
| **Best for** | Reusable workflows, tool wrappers | Project norms, repo context | External APIs, side effects |

CLAUDE.md is for *this project* — what's true here always. A skill is for *this capability* — what's true when doing this type of task. The distinction matters: deployment conventions belong in CLAUDE.md; a deployment workflow with scripts belongs in a skill.

## When to Create a Skill

The clearest signal, from NK's AI Cookbook: **every time you find yourself giving the same instruction twice, that's a signal to encode it as a skill**. But more specifically:

**Create a skill when:**

- You have a **reusable capability with setup steps** — tool installation, API keys, environment configuration. These belong in a skill so any agent on any machine can get set up correctly.
- You have **supporting scripts or assets** that an agent needs to invoke. A skill is a natural home for scripts the agent will run; they stay alongside the instructions that explain how to use them.
- You want **cross-machine and cross-agent consistency**. Skills in a shared repository (like this one) mean the same capability behaves identically whether you're on your laptop, a VPS, or running a different agent tool.
- The task involves a **non-obvious workflow** with multiple steps that an agent would otherwise have to figure out each time — scraping a JS-heavy site, processing a PDF, interacting with a specific API.
- You have **domain knowledge** that should be permanently available — your architecture conventions for a specific tech stack, your review standards, your deployment checklist.

**Don't create a skill when:**

- The instructions are simple enough to fit naturally in CLAUDE.md or a prompt.
- The capability is project-specific and wouldn't be useful outside of one repo.
- You're trying to automate *judgment* rather than *process*. A skill can encode how to do something; it can't encode whether to do it.
- You'd be creating a skill with no supporting scripts or assets — plain instructions with no scripts probably belong in CLAUDE.md instead.

The rule of thumb: if it has scripts, it's a skill. If it's just instructions, it's probably CLAUDE.md.

## How to Create a Skill

### Structure

```
skill-name/
├── SKILL.md              # Required — frontmatter + instructions
├── scripts/              # Helper scripts the agent will execute
│   └── process.sh
├── references/           # Detailed docs loaded on-demand
│   └── api-reference.md
└── assets/               # Templates, config files, examples
    └── template.json
```

The directory name must match the `name` field in frontmatter exactly.

### SKILL.md Format

```markdown
---
name: skill-name
description: What this skill does and when to use it. Be specific — this text
  is the only thing the agent reads before deciding whether to invoke this skill.
---

# Skill Name

## Setup

What needs to happen once before the skill can be used.

## Usage

Concrete commands and workflows. Use {baseDir} to reference the skill directory.

## When to Use

Be explicit — help the agent understand the boundaries of when this skill applies
vs. when a different approach is better.
```

### The Frontmatter Fields

**`name`** (required): Lowercase letters, numbers, and hyphens only. Max 64 characters. Must match the parent directory name exactly.

**`description`** (required): This is the most important field. It determines when the agent loads and uses this skill. Max 1024 characters.

**`compatibility`** (optional): Environment requirements — OS, required binaries, network access. Useful for skills that only work on specific platforms.

**`allowed-tools`** (optional, experimental): Pre-approve specific tools so the agent doesn't have to ask permission mid-skill.

**`metadata`** (optional): Arbitrary key-value pairs — author, version, etc.

### Where Skills Are Discovered

For pi specifically, skills are loaded from:
- `~/.pi/agent/skills/` — global, available everywhere
- `.pi/skills/` — project-level, available in that project
- Paths listed in settings

The skills in `agent-skills/` in this repo are the global ones installed at `~/.pi/agent/skills/`.

## Writing a Good Description

The description field is everything. It's the only text the agent reads before deciding whether to invoke a skill. A bad description means the skill never gets used, or gets used at the wrong time.

**The description should answer:**
1. What does this skill do? (capability)
2. When should I use it? (trigger conditions)
3. When should I *not* use it? (boundaries, if non-obvious)

**Good:**
```yaml
description: Web search and content extraction via Brave Search API. Use for
  searching documentation, facts, or any web content. Lightweight, no browser
  required. Prefer this over browser-tools when the page doesn't require
  JavaScript execution.
```

**Bad:**
```yaml
description: Helps with web searches.
```

The difference: the good description includes the capability, the trigger condition, the key constraint (no browser required), and the decision boundary relative to another skill (when to prefer it over browser-tools). The model can route correctly because it has enough information.

**Practical advice:**
- Write the description *after* writing the rest of the skill. You'll know what it actually does.
- Include the specific keywords that would appear in a user's request ("PDF", "screenshot", "deploy").
- If you have two similar skills, the descriptions should explicitly differentiate them.
- Use the "When to Use" section inside the skill body for detail; use the frontmatter description for routing.

## Getting the Most Value

### Treat Scripts as First-Class Citizens

The scripts in a skill are what separate a skill from just a markdown file. Good skills encapsulate the *how* in executable code, so the agent doesn't have to figure out implementation details — it just calls the script.

The brave-search skill is a good example of this done right: `search.js` and `content.js` are complete, robust scripts with proper argument parsing, error handling, and consistent output format. The agent doesn't need to know the Brave API — it just calls the script.

Design your scripts for agent use:
- **Consistent output format.** The agent needs to parse the output. Plain text or structured text beats verbose logging.
- **Meaningful exit codes.** Agents check whether commands succeed.
- **Self-documenting `--help`.** If the agent ever invokes the script with wrong arguments, it should get clear feedback about what's expected.

### Use `{baseDir}` for Portable Paths

The `{baseDir}` placeholder in SKILL.md gets replaced with the absolute path to the skill directory at runtime. Always use it when referencing scripts and assets:

```bash
{baseDir}/scripts/process.sh input.txt
```

Never hardcode paths. Skills need to work wherever they're installed.

### Progressive Disclosure in the Skill Body

The same principle that governs when the full skill loads should govern the skill content itself. Put the most commonly needed information first, less common details later or in linked reference files.

For example:
- **At the top of SKILL.md**: the main usage commands, the common workflow
- **In the middle**: options, flags, variants
- **In `references/`**: API documentation, edge cases, detailed configuration

An agent working through a common task shouldn't have to wade through edge-case documentation first. An agent handling an unusual case should be able to find the detail without it cluttering the main workflow.

### Keep Skills Focused

One skill, one capability. The CUPID principle applies here: does one thing well.

The temptation is to grow a skill into a general-purpose toolkit. Resist it. A focused skill has a clear description (easier routing), a shorter SKILL.md (less context used), and scripts that do one thing reliably.

If a skill is growing too large, split it. The brave-search and browser-tools skills are separate precisely because they solve different problems: one is lightweight and stateless, the other requires a running browser process. Bundling them would make the description vague and the setup instructions confusing.

### The Compound Interest Effect

From NK's AI Cookbook: "if you invest 30 minutes encoding a skill today, it saves you 5 minutes every time it's used — across every team member, every project, forever."

The compounding comes from two directions:

**Across time**: unlike humans, skills don't forget. An agent that drifts back to old behavior in a later session is one of the most common frustrations with LLM workflows. A skill file that loads every time doesn't drift.

**Across agents and machines**: a skill checked into a shared repository means the capability is available identically on your laptop, a VPS, a colleague's machine, and to any agent tool that supports the standard (Claude Code, pi, Cursor, VS Code, GitHub Copilot — all support the Agent Skills standard).

This is the real value of the Agent Skills standard over ad-hoc CLAUDE.md instructions: portability. The skill you write today works wherever the standard is supported.

### Iterate from Real Usage

The best way to improve a skill is to use it and watch where it fails or requires extra prompting. The failure modes are usually:
- **Wrong routing**: the agent uses the skill when it shouldn't, or doesn't use it when it should → fix the description
- **Missing context**: the agent has to ask clarifying questions mid-skill → add the answers to the skill body
- **Script brittleness**: the scripts fail on edge cases → improve the scripts
- **Scope creep**: the skill tries to do too much → split it

Yegge describes this as the "Desire Paths" approach: watch what the agent tries to do, then make the skill match what the agent naturally attempts. The skill becomes agent-friendly through iteration, not upfront design.

## A Worked Example: The Brave Search Skill

The brave-search skill in this repo is a good reference for what a well-designed skill looks like.

**What makes it work well:**

1. **Clear description with routing boundary**: *"Lightweight, no browser required"* tells the agent when to prefer this over browser-tools. The routing decision is embedded in the description.

2. **Self-contained setup**: the Setup section tells you exactly what to do once — create an account, get a key, run `npm install`. No tribal knowledge required.

3. **Concrete, copy-pasteable commands**: the Usage section shows actual commands with actual flags. No ambiguity about syntax.

4. **Output format documentation**: the agent knows what to expect back from the scripts, which means it can parse and use the results reliably.

5. **Explicit "When to Use" section**: tells the agent the right situations, which reinforces correct routing.

**What could be added over time:**
- Error handling guidance (what to do if the API key is missing or rate-limited)
- Examples of queries that work well vs. queries that don't
- Notes on combining search results with content extraction for specific use cases

These would go in a `references/` directory rather than the main SKILL.md, keeping the primary flow clean.
