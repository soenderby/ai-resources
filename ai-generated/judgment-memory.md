# Judgment Memory

A synthesis of what this repo has collected about durable captured judgment for AI coding agents — what it is, how it's produced, where it lives, what breaks, and why it matters.

---

## What this document is

This is a concept synthesis, not a survey. It draws on 18 accepted sources from the [judgment memory research](./judgment-memory-research/) to define a concept that recurs across the collection but didn't have a name: **judgment memory** — durable captured judgment that changes how future agents plan, implement, review, or decide.

The term is this repo's working label. Nobody else uses it yet. That could mean it names something real that hasn't been named, or that existing terms are sufficient and this one is unnecessary. This document makes the case for the former, while being honest about where the evidence is thin.

---

## The concept

An AI coding agent starts every session with a blank context window. Without intervention, it has no knowledge of your project's conventions, your past mistakes, your architectural decisions, or your team's preferences. It generates plausible code that may violate every principle your team cares about.

The obvious fix is to write instructions: put your conventions in a file, load them at session start. This is what CLAUDE.md, AGENTS.md, and .cursorrules do. But not all instructions are equal. "Use 2-space indentation" is a configuration setting. "Use composition over inheritance because our domain model evolves faster than our type hierarchy" is captured judgment — it encodes not just what to do but *why*, drawn from experience that took effort to acquire.

**Judgment memory** is the subset of persistent agent context that encodes rationale, heuristics, preferences, lessons, and failure-mode knowledge rather than mere facts or configuration. It is what makes the difference between an agent that follows rules and an agent that has inherited taste.

### What judgment memory is not

- **Task memory** — what work exists and its state (Beads, issue trackers). This is coordination infrastructure, not judgment.
- **General context** — project structure, build commands, file layout. Factual, not judgment.
- **Chat history** — what happened in a previous session. Ephemeral, not curated.
- **The model's training** — what the model already knows. Not project-specific, not updatable by you.

The test: if you removed this piece of context, would the agent make a decision that a knowledgeable team member would object to? If yes, it's judgment memory. If no, it's just context.

## How it's produced

The research found four distinct production mechanisms, each with different effort-quality tradeoffs.

### 1. Manually authored by humans

The most common mechanism. A developer writes CLAUDE.md, AGENTS.md, ADRs, specification documents, or review checklists. The judgment comes from the human's experience and is expressed in natural language.

**Strengths:** Highest quality. Captures rationale, taste, and domain knowledge that no automated system can extract. The developer decides what matters.

**Weaknesses:** Highest effort. Requires the developer to notice and articulate implicit knowledge — something most people are bad at. Maintenance is manual. The [Agent READMEs paper](../resources/articles.md#agent-readmes-an-empirical-study-of-context-files-for-agentic-coding) found these files grow through incremental additions but rarely shrink, suggesting pruning is neglected.

**Artifacts:** CLAUDE.md, AGENTS.md, .cursorrules, ADRs, the [Codified Context](../resources/projects.md#codified-context-infrastructure) three-tier system, the [Compound Engineering Plugin](../resources/projects.md#compound-engineering-plugin)'s `docs/plans/` and `docs/solutions/`.

### 2. Automatically captured by agents

The agent itself notices patterns, corrections, and preferences during work and records them for future sessions.

**Strengths:** Low effort. Captures corrections that the developer made but might not think to write down. The capture happens in the flow of work.

**Weaknesses:** Quality depends on the agent's judgment about what's worth remembering. No evidence yet on whether auto-captured judgment is as useful as manually authored judgment. Accumulation without pruning is a known problem.

**Artifacts:** Claude Code's auto memory (MEMORY.md), the [Compound Engineering Plugin](../resources/projects.md#compound-engineering-plugin)'s /ce:compound step.

### 3. Automatically detected from codebase

Tools analyze code and git history to infer conventions, patterns, and trends — without anyone writing rules.

**Strengths:** Detects conventions that the team follows but hasn't articulated. Tracks whether patterns are rising or declining. Can detect "battle scars" — code that looks wrong but exists for a reason.

**Weaknesses:** Can only capture what's visible in code, not the rationale behind it. May detect patterns that are coincidental rather than intentional. Convention detection is pattern-matching, not understanding.

**Artifacts:** [Codebase Context MCP](../resources/projects.md#codebase-context-mcp) (convention detection, memory decay, trend tracking).

### 4. Mined from human practice at scale

Judgment is extracted from large-scale human activity (code reviews, pull requests) across many projects, then abstracted into reusable patterns.

**Strengths:** Broad — captures patterns from hundreds or thousands of projects. Distills the collective judgment of experienced engineers.

**Weaknesses:** Patterns from one ecosystem may not fit a specific project. Abstraction strips away the project-specific context that makes judgment most valuable. May embed biases from the source projects.

**Artifacts:** [Awesome Reviewers](../resources/projects.md#awesome-reviewers) (470+ review prompts mined from 1,000+ open source projects).

## Where it lives

### The loading-strategy spectrum

The [Codified Context paper](../resources/articles.md#codified-context-infrastructure-for-ai-agents-in-a-complex-codebase) introduced the clearest architectural model: a **hot/warm/cold** spectrum based on how and when judgment is loaded into the agent's context window.

**Hot memory** — always loaded at session start. The constitution, the CLAUDE.md, the top-level AGENTS.md. This is where your most universal judgment lives: project-wide conventions, architectural principles, workflow instructions. The constraint is space: Claude Code recommends staying under 200 lines per file, and longer files reduce adherence. Budget your hot memory carefully.

**Warm memory** — loaded per task. Specialized agent specs invoked through trigger tables, path-scoped rules in `.claude/rules/`, skills loaded on demand. This is where domain-specific judgment lives: "when working on the save system, here's what you need to know about the synchronization failures we've already solved."

**Cold memory** — retrieved on demand. Detailed specifications, past solution writeups, architectural rationale documents. Available through MCP, grep, or explicit tool calls. This is the long-term knowledge base — too large to fit in context, but available when the agent knows to look.

### The storage formats

| Format | Example | Judgment density | Maintenance model |
|---|---|---|---|
| Monolithic instructions file | CLAUDE.md, AGENTS.md | Variable — often low | Incremental addition, rare pruning |
| Modular scoped rules | .claude/rules/*.md | High when well-scoped | Per-file review |
| Architecture Decision Records | ADRs via Archgate | High — decision + rationale | Supersede, don't delete |
| Specification documents | Codified Context knowledge base | Variable | Session-linked updates + drift detection |
| Review prompts | Awesome Reviewers | Medium — patterns without project context | Community contribution |
| Auto-captured notes | MEMORY.md | Variable — agent decides | Accumulation with size cap |

## What breaks

Six confirmed failure modes, with evidence.

### 1. Specification staleness

The most dangerous failure mode. When code changes but the spec doesn't, agents generate code based on outdated information. The Codified Context paper: "On at least two occasions, outdated context documents caused agents to generate code that conflicted with recent refactors." The problem is compounded because **agents trust documentation absolutely** — they have no mechanism for suspecting that a spec might be wrong.

**Mitigations found:** Context drift detection (Codified Context), memory decay (Codebase Context MCP), biweekly review cycles.

### 2. Context bloat

Claude Code documentation: files over 200 lines show reduced adherence. The Agent READMEs paper found files grow through incremental additions but rarely shrink. The predictable result: context files become cluttered with a mix of stable conventions, ephemeral learnings, and contradictory instructions, until the agent can't prioritize.

**Mitigations found:** Modular rules files, the hot/warm/cold loading strategy, the 200-line guidance.

### 3. Brevity bias

The [ACE paper](https://arxiv.org/abs/2510.04618) identified this: when contexts are iteratively optimized by automated systems, they tend to collapse toward short, generic prompts. The specific judgment — the failure-mode knowledge, the domain-specific rationale — gets stripped out because it's harder to measure its contribution than it is to measure prompt length. This is the opposite of bloat, and equally harmful: an over-pruned context loses the judgment that made it valuable.

**Mitigations found:** None well-documented. This is an open problem.

### 4. Contradictory instructions

Claude Code documentation: "If two rules contradict each other, Claude may pick one arbitrarily." When judgment memory accumulates across files (CLAUDE.md, rules/, auto memory, nested AGENTS.md), contradictions are inevitable and currently undetected.

**Mitigations found:** None systematic. Some teams designate a single source of truth per domain.

### 5. Convention without rationale

Observed across sources: rules like "use 2-space indentation" without "because X" are configuration, not judgment. They're easy to generate and easy to follow, but they don't help the agent make better decisions in novel situations. The Agent READMEs paper found that instructions are heavily skewed toward functional operations, with non-functional requirements (security, performance, maintainability) almost absent. The *hard* judgment — the kind that prevents architectural drift — is the kind least often written down.

### 6. Judgment that prevents learning

Not confirmed with evidence, but identified as a risk: judgment memory that encodes one team's taste could prevent agents from discovering better approaches. A rule like "always use Redux for state management" captures a preference, but if followed rigidly, it prevents the agent from exploring alternatives that might be better for the current problem. This is the difference between judgment as guidance and judgment as dogma.

## The maintenance gap

Maintenance is the weakest area in the research. Everyone acknowledges that stale judgment is dangerous. Nobody has a complete answer for how to keep it fresh.

**What exists:**
- Manual review cycles (Codified Context: biweekly, ~30-45 min)
- Context drift detection (Codified Context: script flags when code changes without spec updates)
- Memory decay (Codebase Context MCP: stale entries get flagged)
- Convention trend tracking (Codebase Context MCP: rising/declining adoption percentages)
- CI enforcement (Archgate: ADR rules checked at merge time)
- Size caps (Claude Code: 200-line guideline for CLAUDE.md)

**What's missing:**
- Systematic pruning — no source described a principled method for deciding what to remove from judgment memory
- Contradiction detection — no tool currently flags when two pieces of judgment memory conflict
- Quality measurement — no method for evaluating whether judgment memory is actually improving agent output
- Decay curves — no evidence on how quickly different kinds of judgment go stale

The Codified Context paper reports the most honest maintenance data: 1–2 hours per week for a single-developer project. It's not clear how this scales to teams, or whether the cost is linear or superlinear.

## Why this concept matters for this repo

This repo tracks three themes that judgment memory sits at the intersection of:

1. **Memory for agents** — judgment memory is the highest-value kind of agent memory. Task memory (Beads) tells agents what to work on. Judgment memory tells agents how to work well.

2. **Context engineering** — judgment memory is the content that context engineering manages. Good context engineering determines when and how captured judgment gets loaded. But the context engineering literature mostly discusses *mechanisms* (compaction, progressive disclosure, sub-agents), not *what to put in the context*. Judgment memory addresses the content problem.

3. **Software quality with agents** — this is where judgment memory has the most leverage. The [software quality synthesis](./software-quality-with-agents.md) identifies the core problem: agents make code production cheap, but quality still requires human judgment. Judgment memory is the attempt to make that judgment persist across sessions instead of requiring the human to re-apply it every time.

The [Compound Engineering Plugin](../resources/projects.md#compound-engineering-plugin) — the seed source for this research — embodies this intersection. Its /ce:compound step explicitly captures judgment after each work cycle so that "each unit of engineering work should make subsequent units easier — not harder." That compounding effect is the promise of judgment memory, if the maintenance problem can be solved.

## Open questions

1. **Is the term worth keeping?** No existing term covers the concept precisely. "Codified context" is too broad. "Design rationale" is too narrow. "Tribal knowledge" names the pre-capture state. But "judgment memory" is currently used only in this repo. If the concept proves useful for retrieval, curation, and synthesis, the term earns its place. If not, the underlying observations still stand.

2. **What does good maintenance look like?** The biggest gap. Nobody has published a systematic approach to pruning, conflict detection, or quality measurement for agent context files. The Codified Context paper's biweekly review is the closest, and it's essentially manual.

3. **Does judgment memory work at team scale?** Almost all evidence is from single developers or small teams. The Codified Context paper explicitly flags this: "effectiveness in team settings has not been evaluated." When five developers each have different judgment about the same domain, how does that get reconciled?

4. **What's the right ratio of judgment to configuration?** The Agent READMEs paper shows that context files are dominated by configuration. Is that because configuration is what agents need most, or because judgment is harder to articulate?

5. **Can judgment transfer across projects?** The Awesome Reviewers project extracts patterns across 1,000+ projects. But judgment is often context-dependent — a pattern that's good advice for Next.js may be harmful for a real-time system. The conditions under which judgment transfers safely are not well understood.
