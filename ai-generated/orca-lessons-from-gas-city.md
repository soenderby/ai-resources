# Design Guidance: Infrastructure Patterns for Orca's Go Rewrite

> Generated by Claude Sonnet 4 on 2026-04-07. Reviewed and revised 2026-04-07.
>
> Design guidance for the Orca Go rewrite. Identifies concrete infrastructure patterns — drawn from analysis of Gas City (an orchestration-builder SDK by the Gas Town team) — that could benefit Orca. Each suggestion is filtered for compatibility with Orca's philosophy (thin shell, agent autonomy, single-developer scale) and includes explicit tradeoffs.
>
> This document is written for the Orca project. It does not assume familiarity with Gas City or Gas Town. Where a concept originates from that system, it is explained in Orca's terms.

---

## Orca terminology

These terms are used throughout the document. Readers working in the orca codebase will know them; this section exists so the document is self-contained.

- **Agent loop**: The per-agent run cycle. In the current bash implementation (`agent-loop.sh`): create a branch, render the prompt, execute the agent CLI (e.g., `claude`), parse the summary, append metrics. One iteration = one run.
- **Planner** (`plan.sh`): The component that reads the queue, assigns issues to agent slots based on labels and constraints (`px:exclusive`, `ck:*`, `meta:tracker`), and produces a deterministic assignment plan before agents start.
- **`metrics.jsonl`**: Append-only structured log. Each line is a JSON object recording one completed run — includes agent name, issue ID, start/end timestamps, duration, outcome (success/failure), and branch name.
- **Drain mode**: Orca's existing mechanism for stopping agents gracefully. When drain is active, agents finish their current run but don't claim new work. Currently triggered explicitly; the proposal in this document (#4) makes it the default on `stop`.
- **`ORCA_HOME`**: Environment variable pointing to the directory containing orca's scripts. Enables cross-project operation — orca's scripts live in one location and operate on any target repo.
- **Watch**: A separate Go binary in the orca ecosystem ([github.com/soenderby/watch](https://github.com/soenderby/watch)) that provides real-time agent monitoring via CLI and TUI. It reads tmux sessions, agent identities, and orca artifacts to build an agent-centric view across projects.
- **Resume mode / fresh mode**: How session continuity is handled. In **resume mode**, the LLM provider's session is preserved across interactions — the agent retains conversation history (e.g., `claude --resume <session-key>`). In **fresh mode**, each run starts a new provider session with no memory of previous runs. Currently, all orca workers use fresh mode. The operator's interactive session implicitly uses resume mode (the human's conversation with the agent persists).
- **tq**: A planned replacement for Beads (`br`) as orca's task queue — a minimal, file-based, zero-dependency task tracker designed as both a CLI tool and an embeddable Go library ([github.com/soenderby/task-queue](https://github.com/soenderby/task-queue)). Currently in design phase.

## Assumptions

This document assumes:

1. **Orca will continue to use tmux as its session runtime.** Sections #3 and #8 are tmux-specific. If the Go rewrite introduces an alternative session backend, those sections need revisiting.

2. **The Go rewrite preserves orca's current architecture** — agent loop, git worktrees for isolation, structured artifacts, file-based locking — and ports it to Go rather than redesigning from scratch.

3. **Scale stays at 1-4 agents for the near term.** The "what to skip" decisions use this as justification. If the Go rewrite enables higher scale, revisit those decisions — particularly agent-to-agent messaging, metadata-based routing, and multi-step workflow templates.

4. **tq will replace Beads as the task queue.** Section #2 is partly motivated by this transition. The abstraction it proposes is good practice regardless, but the urgency depends on tq happening.

5. **Workers don't need conversation continuity by default.** Workers use fresh mode (no memory across runs). This is the right default for independent issue execution, but some workflows — particularly retrying a previously failed task — might benefit from the agent knowing what was tried before. If this becomes a pattern, per-issue resume mode should be reconsidered.

## The process model question

Before any specific proposal, the Go rewrite must decide: **what is orca's main process model?**

The current bash implementation is fire-and-forget: `orca.sh start` launches tmux sessions and exits. The sessions run independently. `orca.sh status` and `orca.sh stop` are separate invocations that inspect or kill sessions. Orca itself is not a running process — it's a CLI tool that manipulates tmux state.

Several proposals in this document require a different model:

| Proposal | Requires |
|---|---|
| Drain timeout (#4) | Orca must stay running during shutdown to wait for agents to finish |
| Idle timeout / crash-loop detection (#5) | Something must check agent health periodically while agents are running |
| Periodic automation (#6) | A long-running process or separate daemon that executes maintenance tasks on a schedule |

Three options:

**A. Stay fire-and-forget.** Orca remains a stateless CLI. Health checks and periodic tasks are implemented as a separate binary (`orca-daemon` or `orca-cron`) that runs independently. Drain timeout is implemented by making `orca stop` block until agents exit (the CLI stays alive for the duration of the drain).

**B. Become a daemon.** `orca start` launches agents *and* stays running as a supervisor. It owns the health patrol, periodic tasks, and graceful shutdown. `orca stop` signals the daemon to drain and exit.

**C. Hybrid.** `orca start` remains fire-and-forget for agent launching. A separate `orca watch` (not to be confused with the Watch tool) or `orca daemon` command starts the supervisor if the operator wants health checks and periodic tasks. The operator can run with or without the daemon.

The document does not prescribe a choice. But proposals #4, #5, and #6 are only feasible under models B or C. Proposals #1, #2, #3, #7, and #8 work under any model.

## 1. Make the operator agent explicit; distinguish two agent types

### The problem

Orca's actual operating model has two agent types, but only one is acknowledged:

**Batch workers** are launched by `orca.sh start`. They use fresh mode (no memory across runs), claim issues from the queue, execute work, and produce structured artifacts. Orca manages their lifecycle.

**The operator agent** is an interactive session where the human talks to an agent that drives orca commands — "start two workers on the auth module," "check the queue and reprioritize." It uses resume mode (conversation continuity). Orca does not manage it, name it, or know it exists.

This means Watch can't distinguish the operator session from any other interactive session. There's no standard prompt that teaches the agent about orca commands. The relationship between the operator agent and batch workers is implicit knowledge in the human's head.

### What Orca should do

**Formalize the operator agent:**

- **Identity**: Register it in the agent identity registry (`agents.json`) so Watch can track it. Give it a name (e.g., `operator`, or project-specific like `orca-operator`).
- **Prompt**: Create an `OPERATOR_PROMPT.md` that includes:
  - Knowledge of all orca commands and their purpose
  - How to read `metrics.jsonl` and session summaries
  - How to interpret worker status and intervene
  - The queue state query (`tq ready` or equivalent)
  - When to stop workers, adjust priorities, or escalate
- **Lifecycle**: The operator agent starts first and stops last. Workers depend on it conceptually (even if not enforced). The operator agent is the "home session" — the one the human returns to.

This requires no code changes to Orca. It's a configuration and documentation change that names the architecture already in use.

**Distinguish two agent types in configuration:**

```go
type AgentConfig struct {
    Name     string
    Type     string // "operator" or "worker"
    // ...
}
```

The **operator** type: persistent session (resume mode), not managed by the batch scheduler, receives status summaries, registered in Watch with a distinct role.

The **worker** type: fresh session per run, managed by the batch scheduler (start N, drain on stop), subject to health checks, produces structured artifacts.

This is a naming and configuration distinction, not a deep architectural one. It makes the implicit two-type pattern explicit and gives each type appropriate defaults.

### Tradeoffs

| Gain | Cost |
|---|---|
| Watch can identify and track the operator session | One more file to maintain (`OPERATOR_PROMPT.md`) |
| New users understand the intended workflow immediately | Risk of over-formalizing a flexible pattern — the operator might want to drive orca from different sessions in different contexts, or not use an agent at all (just CLI commands from a regular shell) |
| The operator prompt can evolve as a reusable artifact | If the prompt is too prescriptive, it constrains how the human uses the agent |
| Naming the pattern enables designing for it (status summaries, check-in workflows) | Creates an expectation that the operator agent *should* exist, which adds a setup step for casual use |
| Each type gets appropriate defaults (resume vs fresh, managed vs unmanaged) | Adds a type system where there was none. Every future feature must consider "does this apply to operators, workers, or both?" |
| Watch can display operator and worker sessions differently | If someone uses orca without an operator agent (just `orca.sh start` and `orca.sh status` from a shell), the type system is unnecessary overhead |

The main risk is premature formalization. If the operator agent is just "whatever interactive session I happen to be in," formalizing it adds ceremony without value. The test: does the operator prompt meaningfully improve the agent's ability to drive orca? If yes, formalize. If you'd skip reading it yourself, it's not worth maintaining.

### Design questions

- Should the operator agent be launchable via `orca.sh`? Or remain manually started by the human? The latter is simpler and matches the current model. The former means orca needs to manage interactive sessions — a different kind of work than batch execution.
- Should orca produce a status summary optimized for the operator agent's context window (not just `orca.sh status` for humans)?
- Should the operator prompt include a "check-in" workflow — periodic review of worker progress, queue state, and any escalations?

## 2. Configurable work query

### The problem

Orca hardcodes `br ready --json` in agent prompts, the planner, and queue-read scripts. The transition to tq requires changing this in multiple places. Different projects using cross-project operation (`ORCA_HOME`) may want different queue semantics.

### What Orca should do

Define the work query as a single configurable string, resolved at one point:

- **Config level**: In the project config or an environment variable (`ORCA_WORK_QUERY`).
- **Prompt injection**: The operator prompt and agent prompt reference the configured query, not a hardcoded command.
- **Planner integration**: The planner uses the configured query.

This makes the tq transition a one-line change. It also enables project-specific customization for cross-project operation.

### Concrete interface

```go
// In orca config or environment
type QueueConfig struct {
    ReadyCommand  string // default: "tq ready --json"
    ClaimCommand  string // default: "tq claim {id} --actor {agent}"
    CloseCommand  string // default: "tq close {id} --reason {reason}"
}
```

**Scope note**: These three commands are the operations the orca harness calls directly (the agent loop and planner). Agents may also call other queue commands during their work — `tq create` (discovered work), `tq comment` (progress notes), `tq dep add` (dependencies). Those are agent-initiated and belong in the agent prompt, not in the harness config. The harness only needs to know how to find, claim, and close work.

The Go rewrite should define this interface even if the initial implementation just shells out. This is the boundary where tq-the-library can later replace tq-the-CLI without changing anything above it.

### Tradeoffs

| Gain | Cost |
|---|---|
| tq transition becomes a one-line config change | Indirection — debugging queue issues now requires checking what command is configured, not just reading the code |
| Projects can customize queue behavior without forking orca | More configuration surface to document and test |
| Clean boundary for later tq library embedding | If there's only ever one queue implementation, the abstraction is unnecessary ceremony |

This is low-risk. The abstraction is small, the payoff during the tq migration is concrete, and the worst case (only one implementation ever) costs one extra config field.

## 3. Per-project tmux socket isolation

### The problem

Orca uses the global tmux server. Sessions share namespace with the operator's regular tmux sessions. Two orca invocations on different projects can collide. Watch has to filter through all tmux sessions to find orca-related ones.

### What Orca should do

Use `tmux -L orca-<project>` for all tmux operations. tmux's `-L` flag creates a separate server instance with its own session namespace. Each project gets full isolation.

Benefits:
- Session names can be simple (no project prefix needed for uniqueness)
- Watch scanning becomes: list sessions on the `orca-<project>` socket — all are relevant
- Multiple orca invocations on different projects are fully isolated
- The operator's personal tmux sessions are unaffected

The socket name should be configurable (for the case where the operator wants to share a socket across projects). Default to `orca-<project>`.

### Tradeoffs

| Gain | Cost |
|---|---|
| Complete session isolation between projects and personal use | Can't see all orca sessions with a plain `tmux ls` — need `tmux -L orca-<project> ls` |
| Watch scanning is simpler and faster | The operator agent (if it lives in a personal tmux session) is on a different server than the workers — jumping between them requires specifying the socket |
| No naming collisions | One more thing to remember when debugging: which socket am I on? |
| Enables true multi-project parallel operation | If the operator wants a unified view across projects, they need Watch (or must know all socket names) |

The "can't see everything with plain `tmux ls`" is the real cost. It pushes the operator toward Watch (which knows all sockets) and away from raw tmux commands. This is probably fine if Watch is working well, but it's a real friction increase for ad-hoc debugging.

## 4. Drain timeout on stop

*Requires process model B or C (see "The process model question"), or a blocking `stop` implementation under model A.*

### The problem

`orca.sh stop` kills agents immediately. An agent mid-merge, mid-test-run, or mid-commit produces a messy state — partially merged branches, uncommitted changes, broken worktrees.

### What Orca should do

Make drain-then-kill the default stop behavior:

1. `orca stop` sets the drain flag (a file, e.g., `$PROJECT_ROOT/.orca-drain`) that the agent loop checks between runs
2. Workers finish their current run but don't claim new work
3. Wait up to `ORCA_DRAIN_TIMEOUT` (default 5m) for all workers to exit
4. Force-kill any remaining sessions
5. `orca stop --force` skips the drain and kills immediately (current behavior)

Under process model A, `orca stop` itself blocks while waiting for sessions to drain. Under models B/C, the daemon handles the wait.

Orca already has a drain mode concept (agents check a drain flag between runs). The change is making `stop` use it by default, with a timeout and force-kill fallback.

### Tradeoffs

| Gain | Cost |
|---|---|
| No more killed-mid-merge states | `stop` is no longer instant — operator must wait up to 5 minutes (or use `--force`) |
| Cleaner worktree state after shutdown | A stalled agent blocks shutdown until the timeout expires |
| Agents can write final summaries and close issues properly | More complex stop logic — need to track which agents have exited and which are still draining |
| Reduces manual cleanup after restarts | The drain flag file is a new coordination mechanism that must be checked reliably |

The 5-minute wait on `stop` is the main UX cost. When you want to stop and go to lunch, waiting for a stalled agent is frustrating. The `--force` escape hatch is essential. Consider also: if drain timeout expires, log *which* agent was still running and *what* it was doing — that information is valuable for understanding what went wrong.

## 5. Idle timeout and crash-loop detection

*Requires a periodic trigger mechanism — either the periodic task runner (#6) or an external cron job.*

### The problem

If an agent stalls (infinite loop, stuck on a broken merge, burning tokens without progress), the operator has to notice manually. If an agent keeps crashing and restarting on the same broken task, it burns tokens and compute in a loop until someone intervenes.

### What Orca should do

Implement two lightweight health checks:

**Idle detection**: `metrics.jsonl` has timestamps and durations for every completed run. A periodic check (every 60s) compares the current run's elapsed time against a threshold (configurable, default: hard cap like 30m). When triggered: kill the session, mark the issue as failed (or return it to the queue), log the event.

**Crash-loop detection**: Track consecutive failures per agent in a simple counter file. If an agent fails N times (default 3) within M minutes (default 30), stop restarting it and flag it for operator attention. The counter resets on a successful run.

Both checks depend on a periodic trigger. See #6 (periodic automation) for the mechanism, or implement them as a standalone cron-style check.

### Tradeoffs

| Gain | Cost |
|---|---|
| Can run overnight without babysitting — stalled agents get killed automatically | False positives: a legitimately long-running task gets killed at the timeout. Some tasks (large refactors, complex merges) genuinely take longer than average. |
| Crash loops don't burn unlimited tokens | Quarantined agents stop doing useful work. If the failure is transient (network blip, API rate limit), the agent would have recovered on the next try but is now stuck. |
| Health state is observable (Watch can display it) | More state to manage — counter files, quarantine flags, threshold configuration |
| Issues that cause crashes get surfaced instead of silently retried | The operator must actively un-quarantine agents, adding a manual step to recovery |

The false positive risk on idle timeout is real. A hard 30-minute cap will kill agents doing legitimate long work. Possible mitigations: per-issue timeout override (a label like `long-running`), or basing the threshold on the specific issue's estimated complexity rather than a global cap. But all mitigations add complexity. Start with a generous hard cap and tighten based on observed run durations.

## 6. Periodic automation

*Requires process model B or C (see "The process model question"), or a separate binary.*

### The problem

Orca has several maintenance tasks that should run periodically but are currently manual or missing:
- Branch cleanup (`gc-run-branches --apply`)
- Stale worktree detection
- Queue health checks (orphan `in_progress` items with no active session)
- Idle timeout and crash-loop detection (see #5)
- Metrics aggregation for the operator agent

### What Orca should do

Implement a simple periodic task runner. Each task is a named shell command that runs at a configurable interval:

```go
type PeriodicTask struct {
    Name     string
    Command  string        // shell command to execute
    Interval time.Duration // how often to run
    Enabled  bool
}
```

If orca adopts process model B (daemon), the task runner is a goroutine in the daemon process. Under model A or C, it's a separate binary (`orca-cron` or `orca-daemon`) that runs independently of the CLI.

Tasks are defined in config. Output goes to a log file. Failures are counted (for crash-loop-style detection on the tasks themselves).

### Built-in tasks

These should ship as defaults:

| Task | Interval | Purpose |
|---|---|---|
| `health-check` | 60s | Idle timeout + crash-loop detection (#5) |
| `queue-audit` | 5m | Orphan `in_progress` items, stale claims |
| `branch-cleanup` | 1h | Prune merged run branches |
| `metrics-summary` | 10m | Write a compact status summary for the operator agent |

### Tradeoffs

| Gain | Cost |
|---|---|
| Maintenance tasks run reliably without operator intervention | Introduces a long-running process to an ecosystem that currently has none (aside from tmux sessions). This is a meaningful complexity increase. |
| Health checks and cleanup are formalized instead of ad-hoc | Task failures need their own error handling — a broken cleanup script shouldn't crash the runner |
| New maintenance tasks are just config entries, not code changes | More configuration surface. Each task has interval, enabled, and command — that's N × 3 settings to manage. |
| The operator agent can rely on up-to-date status summaries | Periodic tasks running during agent work could cause interference — e.g., branch cleanup while an agent is mid-merge. Need to consider locking. |

### Design principle

Keep the task runner minimal. If a task needs agent judgment, it should create an issue in the queue and let an agent handle it — not become a complex automated workflow. The periodic runner handles infrastructure; agents handle intelligence.

## 7. Prompt templates with variables

### The problem

Orca has a single static `AGENT_PROMPT.md` (or `ORCA_PROMPT.md` per project). Agent-specific details (name, project, worktree path, queue command) are either hardcoded or injected through environment variables that the agent has to know to read.

### What Orca should do

Use Go's `text/template` for prompt rendering. Define a standard set of variables:

```go
type PromptContext struct {
    Agent       string // agent name
    Project     string // project name
    WorkDir     string // worktree path
    WorkQuery   string // configured work query command (see #2)
    ProjectRoot string // main repo root
    OrcaHome    string // orca installation path
    RunID       string // current run identifier (for workers)
}
```

One prompt template serves all agents and all projects. Per-project customization via `ORCA_PROMPT.md` overrides (checked first, falls back to default template).

Example:
```
You are {{.Agent}} working on {{.Project}}.
Your worktree is at {{.WorkDir}}.

To find available work: {{.WorkQuery}}
```

For the operator agent, the template can include dynamic context:
```
Active workers: {{.WorkerCount}}
Last metrics summary: {{.MetricsSummaryPath}}
```

### Tradeoffs

| Gain | Cost |
|---|---|
| One template serves all agents and projects | Templates are harder to read than plain markdown — `{{.WorkQuery}}` is less obvious than `tq ready --json` |
| Agent-specific details are injected correctly without the agent needing to know about environment variables | Template rendering failures (missing variable, bad syntax) produce confusing errors at a critical moment (agent startup) |
| Per-project overrides work naturally (provide your own template, use the same variables) | The variable set becomes a compatibility surface — adding or removing a variable affects all templates |
| The operator prompt can include live state | Dynamic context in prompts creates a coupling between the prompt and orca internals (what if the metrics summary format changes?) |

The readability cost is real. A static `AGENT_PROMPT.md` is immediately human-readable. A template with `{{.WorkDir}}` requires knowing what gets substituted. Mitigation: an `orca prompt --render` command that shows the fully rendered prompt for a given agent and project. This is useful for debugging and for the operator to verify what agents actually see.

## 8. Pre-start hook separation

### The problem

Orca's `agent-loop.sh` combines environment setup, worktree preparation, branch creation, prompt rendering, agent execution, and result parsing in one script. Changing one concern risks breaking another.

### What Orca should do

In the Go rewrite, define clean lifecycle phases:

```go
type AgentLifecycle struct {
    PreStart []string // shell commands before session creation
    OnDeath  func()   // unexpected session death handler
}
```

**PreStart** is where worktree setup, branch creation, and environment validation happen. It runs before any tmux session exists. If it fails, the agent doesn't start. Project-specific setup (worktree layout, branch naming) lives in PreStart hooks, not in the core agent loop.

**OnDeath** fires when a session dies unexpectedly. This is where crash-loop counting (#5) and issue re-queuing happen.

Start with these two only. A third phase — **SessionSetup** (tmux configuration after session creation: theming, keybindings, status bars) — can be added later if the need arises from real usage. Don't build it speculatively.

### Tradeoffs

| Gain | Cost |
|---|---|
| Project-specific setup lives in hooks, not in the core agent loop | More concepts to understand — "why didn't my agent start?" now requires checking which lifecycle phase failed |
| The OnDeath hook is a clean place for crash recovery logic | Shell command hooks are hard to test and debug — errors in hook scripts are a common source of mysterious failures |
| Project-specific customization doesn't require forking orca | More configuration surface. Each agent potentially has a list of shell commands to maintain. |

The main risk is over-engineering. If most projects never customize hooks, the separation adds complexity without value. The two proposed hooks (PreStart and OnDeath) have clear, distinct responsibilities and known use cases. Resist adding more until real usage demands them.

## What to skip

These patterns from larger orchestration systems don't fit Orca's philosophy or scale. Each includes a condition for when to revisit.

| Pattern | Description | Why skip | Revisit when |
|---|---|---|---|
| **Continuous reconciliation loop** | A long-running controller that continuously compares desired state to actual state and corrects drift. | Orca's explicit start/stop is simpler. Periodic tasks (#6) cover maintenance needs without a full control loop. | Desired-state drift becomes a real problem (agents and config getting out of sync routinely). |
| **Configuration layering** | Reusable configuration bundles composed, overridden, and merged into a final config. | Orca has one operating model. One config file per project is enough. | Orca supports multiple distinct orchestration patterns that share common infrastructure. |
| **Multi-step workflow templates** | Declarative definitions of multi-step agent workflows with checkpoint recovery and step dependencies. | At 1-4 agents, the operator agent handles sequencing in conversation. | Agent count regularly exceeds what the operator can sequence manually. |
| **Agent-to-agent messaging** | A structured message system where agents send messages to other agents. | At current scale, the operator agent is the communication hub. | Agents need to coordinate with each other without operator mediation. |
| **Automated iterative refinement** | Bounded loops where the system retries and refines agent output until a quality threshold is met. | Orca's agents complete or escalate (by leaving a summary). No automated retry loops needed. | A class of tasks exists where automated retry with feedback consistently outperforms single-shot execution. |
| **Multi-provider abstraction** | An abstraction layer that supports running agents on different platforms (tmux, Kubernetes, subprocess) via configuration. | Premature. Start with tmux. Keep the runtime interface clean enough that providers *could* be added later. | Need to run agents on a different platform (cloud, CI, containers). |
| **Metadata-based work routing** | Work items routed to specific agents based on metadata labels and routing rules. | Work routing is simple: agent claims from queue. | Different agent specializations exist (one for coding, one for testing, one for docs) that need different work streams. |

## Implementation sequence

### Phase 0: Now (no code changes needed)

1. **Operator agent formalization** (#1, documentation part) — identity registration in `agents.json`, dedicated `OPERATOR_PROMPT.md`, Watch integration. This is configuration and documentation, not code.

### Phase 1: Foundation (during the Go rewrite)

These should be built into the Go architecture from the start, because retrofitting them later is expensive:

2. **Configurable work query** (#2) — define the `QueueConfig` interface even if initial implementation shells out to `tq`
3. **Per-project tmux socket** (#3) — one flag on all tmux calls
4. **Lifecycle hooks** (#8) — clean separation of PreStart and OnDeath
5. **Prompt templates** (#7) — use Go `text/template` from day one
6. **Two agent types** (#1, code part) — operator and worker as config-level concepts with distinct defaults

### Phase 2: Graceful shutdown (after core rewrite works)

7. **Drain timeout on stop** (#4) — drain flag file, configurable timeout, force-kill fallback. Requires the process model question to be settled.

### Phase 3: Robustness (after enough operational experience to know what actually breaks)

These address failure modes. Don't build them until you've seen the failures:

8. **Periodic task runner** (#6) — this must come before #9 and #10, since they depend on it
9. **Idle timeout** (#5) — flag runs exceeding threshold, kill and re-queue
10. **Crash-loop detection** (#5) — quarantine agents that fail repeatedly

## Open questions

1. **What is the process model?** (See "The process model question" above.) This is the most consequential architectural decision and affects the feasibility of #4, #5, and #6.

2. **How should the operator agent receive status?** Options: read a status file (pull), receive a tmux message when something important happens (push), or include status in the prompt template at render time (ambient). The ecosystem principle is "no tool pushes information at the operator." But a notification when a worker quarantines might be worth the interruption. Start with pull (status file from periodic task #6) and add push only if the operator consistently misses important events.

3. **Where does the operator prompt live?** Options: in the orca repo (ships with orca), in the target project (project-specific), or generated from config (dynamic). Recommendation: a default ships with orca, projects can override with `OPERATOR_PROMPT.md` in the project root.

4. **How much of this should be in orca vs. Watch?** The operator agent's identity registration, status display, and session management overlap with Watch's concerns. Proposed boundary: orca owns execution and artifacts, Watch owns observation and navigation, the operator agent uses both. Avoid duplicating functionality.
