# Lessons from Gas City for Orca's Go Rewrite

> Generated by Claude Sonnet 4 on 2026-04-07.
>
> Design guidance document for the Orca project. Captures concrete concepts, techniques, and functionality from Gas City that could benefit Orca — filtered for what fits Orca's philosophy (thin shell, agent autonomy, single-developer scale) and what doesn't.
>
> Sources: [Gas City repo](https://github.com/gastownhall/gascity) (README, config reference, "Coming from Gas Town" migration guide, examples), [Orca repo](https://github.com/soenderby/orca) (README, ecosystem.md, docs/), and the [existing comparison](orca-gastown-gascity-comparison.md).

---

## Context

Orca is a batch execution engine for autonomous coding agents. It manages transport — tmux sessions, git worktrees, locking, task queue coordination, structured artifact capture — so agents can focus on doing work. It is being rewritten from bash to Go.

Gas City is an orchestration-builder SDK extracted from Gas Town. It provides runtime providers, declarative configuration, health patrol, work routing, orders, formulas, and a controller/supervisor loop.

Orca and Gas City solve overlapping problems at different scales with different philosophies. Orca's design principle — "when the harness reaches a runtime decision point, the answer is to give the decision to an agent, not to write a conditional" — means many Gas City features are out of scope. But Gas City has solved specific infrastructure problems that Orca will encounter as it matures.

This document identifies what to adopt, what to adapt, and what to skip.

---

## 1. Make the operator agent explicit

### The problem

Orca's actual operating model has two agent types: batch workers (launched by `orca.sh start`) and an implicit operator agent (an interactive session where the human talks to an agent that drives orca commands). The operator agent has no name, no dedicated prompt, no registered identity, and no defined lifecycle in the system.

### What Gas City does

The Mayor is a configured agent like any other — with a prompt template, a persistent session (`wake_mode = "resume"`), a specific `work_query`, and a declared role. Nothing about it is hardcoded in the SDK; it's pack configuration.

### What Orca should do

Formalize the operator agent as a first-class concept:

- **Identity**: Register it in the agent identity registry (`agents.json`) so Watch can track it. Give it a name (e.g., `operator`, or project-specific like `orca-operator`).
- **Prompt**: Create an `OPERATOR_PROMPT.md` that includes:
  - Knowledge of all orca commands and their purpose
  - How to read `metrics.jsonl` and session summaries
  - How to interpret worker status and intervene
  - The queue state query (`tq ready` or equivalent)
  - When to stop workers, adjust priorities, or escalate
- **Lifecycle**: The operator agent starts first and stops last. Workers depend on it conceptually (even if not enforced). The operator agent is the "home session" — the one the human returns to.
- **Session type**: Persistent / resume mode. The human wants conversation continuity across interactions. Workers use fresh sessions per run.

This requires no code changes to Orca. It's a configuration and documentation change that names the architecture already in use.

### Design questions

- Should the operator agent be launchable via `orca.sh`? Or remain manually started by the human?
- Should orca produce a status summary that's optimized for the operator agent's context window (not just `orca.sh status` for humans)?
- Should the operator prompt include a "check-in" workflow — periodic review of worker progress, queue state, and any escalations?

## 2. Configurable work query

### The problem

Orca hardcodes `br ready --json` in agent prompts, the planner, and queue-read scripts. The transition to tq requires changing this in multiple places.

### What Gas City does

`work_query` and `sling_query` are per-agent configurable shell commands:

```toml
work_query = "bd ready --unassigned --json --limit=1"
sling_query = "bd update {} --set-metadata gc.routed_to=myproject/worker"
```

### What Orca should do

Define the work query as a single configurable string, resolved at one point:

- **Config level**: In the project config or an environment variable (`ORCA_WORK_QUERY`).
- **Prompt injection**: The operator prompt and agent prompt reference the configured query, not a hardcoded command.
- **Planner integration**: `plan.sh` (or its Go equivalent) uses the configured query.

This makes the tq transition a one-line change. It also enables project-specific customization for cross-project operation (`ORCA_HOME`).

### Concrete interface

```go
// In orca config or environment
type QueueConfig struct {
    ReadyCommand  string // default: "tq ready --json"
    ClaimCommand  string // default: "tq claim {id} --actor {agent}"
    CloseCommand  string // default: "tq close {id} --reason {reason}"
}
```

The Go rewrite should define this interface even if the initial implementation just shells out. This is the boundary where tq-the-library can later replace tq-the-CLI.

## 3. Per-project tmux socket isolation

### The problem

Orca uses the global tmux server. Sessions share namespace with the operator's regular tmux sessions. Two orca invocations on different projects can collide. Watch has to filter through all tmux sessions to find orca-related ones.

### What Gas City does

`tmux -L <socket>` gives each city its own tmux server. Default: the city name.

### What Orca should do

Use `tmux -L orca-<project>` for all tmux operations. This is a small change (one flag on every `tmux` invocation) with concrete benefits:

- Session names can be simple (no project prefix needed for uniqueness)
- Watch scanning becomes: list sessions on the `orca-<project>` socket, all of them are relevant
- Multiple orca invocations on different projects are fully isolated
- The operator's personal tmux sessions are unaffected

### Implementation note

The socket name should be configurable (for the case where the operator wants to share a socket across projects). Default to `orca-<project>`.

## 4. Drain timeout on stop

### The problem

`orca.sh stop` kills agents immediately. An agent mid-merge, mid-test-run, or mid-commit produces a messy state — partially merged branches, uncommitted changes, broken worktrees.

### What Gas City does

Two-phase shutdown:
1. Signal agents to stop claiming new work (drain)
2. Wait `drain_timeout` (default 5m per agent) for current work to complete
3. If still running after timeout, force-kill

### What Orca should do

Make drain-then-kill the default stop behavior:

1. `orca.sh stop` sets a drain flag (or signals workers via a file/lock)
2. Workers finish their current run but don't claim new work
3. Wait up to `ORCA_DRAIN_TIMEOUT` (default 5m) for all workers to exit
4. Force-kill any remaining sessions

Orca already has a drain mode concept. The change is making `stop` use it by default, with a `--force` flag for immediate kill.

### Implementation detail

The drain signal needs to be visible to agents. Options:
- A file (`$ORCA_HOME/.drain`) that the agent loop checks between runs
- A tmux environment variable set on the session
- Sending a specific message to the agent

The file approach is simplest and matches Orca's file-based philosophy.

## 5. Idle timeout and crash-loop detection

### The problem

If an agent stalls (infinite loop, stuck on a broken merge, burning tokens without progress), the operator has to notice manually. If an agent keeps crashing and restarting on the same broken task, it burns resources in a loop.

### What Gas City does

Two mechanisms in the controller's health patrol:
- `idle_timeout`: kill and restart sessions that haven't produced output in N time
- `max_restarts` within `restart_window`: if an agent crashes N times in M minutes, quarantine it

### What Orca should do

Implement both as lightweight checks in the Go rewrite:

**Idle detection**: `metrics.jsonl` already has timestamps and durations for every run. A periodic check (every 60s) that compares the current run's elapsed time against a threshold (configurable, default 2× the median run duration, or a hard cap like 30m) catches most stalls. When triggered: kill the session, mark the issue as failed, log the event.

**Crash-loop detection**: Track restarts per agent in a simple counter file. If an agent fails N times (default 3) within M minutes (default 30), stop restarting it and flag it for operator attention. The operator agent's check-in workflow (see #1) should surface quarantined agents.

### When to run these checks

These checks need a periodic trigger. See #6 (periodic automation).

## 6. Periodic automation (exec orders)

### The problem

Orca has several maintenance tasks that should run periodically but are currently manual or missing:
- Branch cleanup (`gc-run-branches --apply`)
- Stale worktree detection
- Queue health checks (orphan `in_progress` items with no active session)
- Idle timeout and crash-loop detection (see #5)
- Metrics aggregation for the operator agent

### What Gas City does

"Exec orders" — TOML-defined periodic shell commands that run on a trigger (interval, event, or condition gate) without creating an agent session:

```toml
type = "exec"
command = "orca.sh gc-run-branches --apply"
gate = "interval"
interval = "1h"
```

### What Orca should do

Implement a simple periodic task runner in the Go rewrite. Not the full Gas City orders system (trigger types, event gates, condition gates), just:

```go
type PeriodicTask struct {
    Name     string
    Command  string        // shell command
    Interval time.Duration
    Enabled  bool
}
```

A goroutine runs each enabled task at its interval. Tasks are defined in config. Output goes to a log file. Failures are counted (for crash-loop-style detection on the tasks themselves).

### Built-in tasks

These should ship as defaults:

| Task | Interval | Purpose |
|---|---|---|
| `health-check` | 60s | Idle timeout + crash-loop detection |
| `queue-audit` | 5m | Orphan `in_progress` items, stale claims |
| `branch-cleanup` | 1h | Prune merged run branches |
| `metrics-summary` | 10m | Write a compact status summary for the operator agent |

### Design principle

Keep the task runner minimal. If a task needs agent judgment, it should create an issue in the queue and let an agent handle it — not become a complex automated workflow.

## 7. Prompt templates with variables

### The problem

Orca has a single static `AGENT_PROMPT.md` (or `ORCA_PROMPT.md` per project). Agent-specific details (name, project, worktree path, queue command) are either hardcoded or injected through environment variables.

### What Gas City does

Go templates with variables:

```
You are {{.Agent}} working in {{.Rig}} at {{.WorkDir}}.
Check for work: {{.WorkQuery}}
```

### What Orca should do

Use Go's `text/template` for prompt rendering. Define a standard set of variables:

```go
type PromptContext struct {
    Agent       string // agent name
    Project     string // project/rig name
    WorkDir     string // worktree path
    WorkQuery   string // configured work query command
    ProjectRoot string // main repo root
    OrcaHome    string // orca installation path
    RunID       string // current run identifier (for workers)
}
```

One prompt template serves all agents and all projects. Per-project customization via `ORCA_PROMPT.md` overrides (checked first, falls back to default template).

### Benefit for the operator agent

The operator prompt template can include dynamic context:

```
Current queue state: {{.QueueSummary}}
Active workers: {{.WorkerCount}}
Last metrics summary: {{.MetricsSummaryPath}}
```

This turns the operator prompt from static documentation into a live briefing.

## 8. Pre-start hook separation

### The problem

Orca's `agent-loop.sh` combines environment setup, worktree preparation, branch creation, prompt rendering, agent execution, and result parsing in one script. Changing one concern risks breaking another.

### What Gas City does

Three lifecycle hooks:
- `pre_start`: before session creation (worktree setup, env preparation)
- `session_setup`: after session creation (tmux config, keybindings)
- `session_live`: safe to re-apply without restart (theming, status bars)

### What Orca should do

In the Go rewrite, define clean lifecycle phases:

```go
type AgentLifecycle struct {
    PreStart     []string // shell commands before session creation
    SessionSetup []string // shell commands after session creation
    OnRun        func()   // the agent loop itself
    OnDrain      func()   // graceful shutdown signal
    OnDeath      func()   // unexpected session death
}
```

`PreStart` is where worktree setup, branch creation, and environment validation happen. `SessionSetup` is where tmux configuration and prompt injection happen. `OnDeath` is where cleanup and re-queue logic live.

The separation means:
- Project-specific setup (worktree layout, branch naming) lives in `PreStart` hooks, not in the core agent loop
- Tmux configuration changes don't require agent restarts
- The `OnDeath` hook is where crash-loop counting and issue re-queuing happen

## 9. Two session types as a first-class concept

### The problem

Orca treats all agents as batch workers. But the actual workflow has two distinct types with different lifecycle needs.

### What Gas City does

Distinguishes:
- Persistent agents (crew): named, `wake_mode = "resume"`, interactive, long-lived
- Pool agents (polecats): ephemeral, `wake_mode = "fresh"`, batch, claim-and-execute

### What Orca should do

Define two agent types in the configuration:

```go
type AgentConfig struct {
    Name     string
    Type     string // "operator" or "worker"
    WakeMode string // "resume" (operator) or "fresh" (worker)
    // ...
}
```

The operator type:
- Starts first, stops last
- Resume mode (conversation continuity)
- Not managed by the batch scheduler
- Receives status summaries from periodic tasks
- Registered in Watch with a distinct role

The worker type:
- Fresh session per run
- Managed by the batch scheduler (start N, drain on stop)
- Subject to idle timeout and crash-loop detection
- Produces structured artifacts

This is a naming and configuration distinction, not a deep architectural one. But it makes the implicit two-agent-type pattern explicit and gives each type appropriate defaults.

## What to skip

These Gas City features don't fit Orca's philosophy or scale:

| Feature | Why skip |
|---|---|
| **Controller reconciliation loop** | Orca's explicit start/stop is simpler. A continuously running controller is overhead for a single-developer tool. Periodic tasks (#6) cover the maintenance needs. |
| **Packs and pack composition** | Orca has one operating model. Configuration layering is platform complexity. One config file per project is enough. |
| **Formulas and molecules** | Multi-step workflow templates are valuable at scale. At 1-4 agents, the operator agent handles sequencing in conversation. |
| **Mail system** | Agent-to-agent messaging matters at 20+ agents. At Orca's scale, the operator agent is the communication hub. |
| **Convergence loops** | Bounded iterative refinement is sophisticated. Orca's agents complete or escalate. No automated retry loops needed. |
| **Provider abstraction** | Orca's Go rewrite should keep the runtime interface clean enough that providers *could* be added later, but implementing multiple providers now is premature. Start with tmux. |
| **Kubernetes provider** | If Orca needs cloud scaling, it's outgrown its scope and should evaluate Gas City properly. |
| **Sling/routing system** | Work routing in Orca is simple: agent claims from queue. No metadata-based routing needed at current scale. |

## Implementation sequence

Aligned with the Go rewrite phases:

### Phase 1: Foundation (during rewrite)

These should be built into the Go architecture from the start:

1. **Configurable work query** — define the interface even if initial implementation shells out to `tq`
2. **Per-project tmux socket** — one flag on all tmux calls
3. **Lifecycle hooks** — clean separation of pre_start / session_setup / on_death
4. **Two agent types** — operator and worker as config-level concepts
5. **Prompt templates** — use Go `text/template` from day one

### Phase 2: Operator experience (after core rewrite)

These improve the operating model:

6. **Operator agent formalization** — identity registration, dedicated prompt, Watch integration
7. **Drain timeout on stop** — drain flag file, configurable timeout, force-kill fallback
8. **Periodic task runner** — goroutine-based, config-defined, with built-in health check and queue audit tasks

### Phase 3: Robustness (after operational experience)

These address failure modes observed in real use:

9. **Idle timeout** — flag runs exceeding threshold, kill and re-queue
10. **Crash-loop detection** — quarantine agents that fail repeatedly
11. **Metrics summary** — periodic compact status for the operator agent's context

## Open questions

1. **Should the operator agent be launched by orca?** Or should orca remain a tool the operator agent calls? The latter is simpler and matches the current model. The former gives orca more control over the operator agent's lifecycle.

2. **How should the operator agent receive status?** Options: read a status file (pull), receive a nudge when something important happens (push), or include status in the prompt template (ambient). Gas City's principle is "no tool pushes information at the operator." But a nudge when a worker quarantines might be worth the interruption.

3. **Should periodic tasks be a library or a separate binary?** A goroutine in the orca process is simplest. But a separate `orca-cron` binary would be more composable and testable. The Go rewrite's architecture should make either possible.

4. **Where does the operator prompt live?** Options: in the orca repo (ships with orca), in the target project (project-specific), or generated from config (dynamic). Probably: a default ships with orca, projects can override with `OPERATOR_PROMPT.md`.

5. **How much of this should be in orca vs. Watch?** The operator agent's identity registration, status display, and session management overlap with Watch's concerns. The boundary should be: orca owns execution and artifacts, Watch owns observation and navigation, the operator agent uses both.
