# Design Guidance: Infrastructure Patterns for Orca's Go Rewrite

> Generated by Claude Sonnet 4 on 2026-04-07.
>
> Design guidance for the Orca Go rewrite. Identifies concrete infrastructure patterns — drawn from analysis of Gas City (an orchestration-builder SDK by the Gas Town team) and other agent orchestration systems — that could benefit Orca. Each suggestion is filtered for compatibility with Orca's philosophy (thin shell, agent autonomy, single-developer scale) and includes explicit tradeoffs.
>
> This document is written for the Orca project. It does not assume familiarity with Gas City or Gas Town. Where a concept originates from those systems, it is explained in Orca's terms.

---

## Context

Orca is a batch execution engine for autonomous coding agents. It manages transport — tmux sessions, git worktrees, locking, task queue coordination, structured artifact capture — so agents can focus on doing work. It is being rewritten from bash to Go.

This document identifies infrastructure patterns that Orca will likely need as it matures. Each pattern is described as a standalone concept with a concrete proposal, implementation notes, and an honest assessment of what it costs.

---

## 1. Make the operator agent explicit

### The problem

Orca's actual operating model has two agent types: **batch workers** (launched by `orca.sh start`) and an **implicit operator agent** (an interactive session where the human talks to an agent that drives orca commands). The operator agent — the session where you say "start two workers on the auth module" or "check the queue and reprioritize" — has no name, no dedicated prompt, no registered identity, and no defined lifecycle in the system.

This means Watch can't distinguish the operator session from any other interactive session. It means there's no standard prompt that teaches the agent about orca commands. It means the relationship between the operator agent and the batch workers is implicit knowledge in the human's head.

### What Orca should do

Formalize the operator agent as a first-class concept:

- **Identity**: Register it in the agent identity registry (`agents.json`) so Watch can track it. Give it a name (e.g., `operator`, or project-specific like `orca-operator`).
- **Prompt**: Create an `OPERATOR_PROMPT.md` that includes:
  - Knowledge of all orca commands and their purpose
  - How to read `metrics.jsonl` and session summaries
  - How to interpret worker status and intervene
  - The queue state query (`tq ready` or equivalent)
  - When to stop workers, adjust priorities, or escalate
- **Lifecycle**: The operator agent starts first and stops last. Workers depend on it conceptually (even if not enforced). The operator agent is the "home session" — the one the human returns to.
- **Session type**: The operator agent uses persistent, resume-mode sessions (the human wants conversation continuity). Workers use fresh sessions per run. This distinction currently exists in practice but isn't named anywhere.

This requires no code changes to Orca. It's a configuration and documentation change that names the architecture already in use.

### Tradeoffs

| Gain | Cost |
|---|---|
| Watch can identify and track the operator session | One more file to maintain (`OPERATOR_PROMPT.md`) |
| New users understand the intended workflow immediately | Risk of over-formalizing a flexible pattern — the operator might want to drive orca from different sessions in different contexts |
| The operator prompt can evolve as a reusable artifact | If the prompt is too prescriptive, it constrains how the human uses the agent |
| Naming the pattern enables designing for it (status summaries, check-in workflows) | Creates an expectation that the operator agent *should* exist, which adds a setup step for casual use |

The main risk is premature formalization. If the operator agent is just "whatever interactive session I happen to be in," formalizing it adds ceremony without value. The test: does the operator prompt meaningfully improve the agent's ability to drive orca? If yes, formalize. If you'd skip reading it yourself, it's not worth maintaining.

### Design questions

- Should the operator agent be launchable via `orca.sh`? Or remain manually started by the human?
- Should orca produce a status summary optimized for the operator agent's context window (not just `orca.sh status` for humans)?
- Should the operator prompt include a "check-in" workflow — periodic review of worker progress, queue state, and any escalations?

## 2. Configurable work query

### The problem

Orca hardcodes `br ready --json` in agent prompts, the planner, and queue-read scripts. The transition to tq requires changing this in multiple places. Different projects using cross-project operation (`ORCA_HOME`) may want different queue semantics.

### What Orca should do

Define the work query as a single configurable string, resolved at one point:

- **Config level**: In the project config or an environment variable (`ORCA_WORK_QUERY`).
- **Prompt injection**: The operator prompt and agent prompt reference the configured query, not a hardcoded command.
- **Planner integration**: `plan.sh` (or its Go equivalent) uses the configured query.

This makes the tq transition a one-line change. It also enables project-specific customization for cross-project operation.

### Concrete interface

```go
// In orca config or environment
type QueueConfig struct {
    ReadyCommand  string // default: "tq ready --json"
    ClaimCommand  string // default: "tq claim {id} --actor {agent}"
    CloseCommand  string // default: "tq close {id} --reason {reason}"
}
```

The Go rewrite should define this interface even if the initial implementation just shells out. This is the boundary where tq-the-library can later replace tq-the-CLI without changing anything above it.

### Tradeoffs

| Gain | Cost |
|---|---|
| tq transition becomes a one-line config change | Indirection — debugging queue issues now requires checking what command is configured, not just reading the code |
| Projects can customize queue behavior without forking orca | More configuration surface to document and test |
| Clean boundary for later tq library embedding | If there's only ever one queue implementation, the abstraction is unnecessary ceremony |

This is low-risk. The abstraction is small, the payoff during the tq migration is concrete, and the worst case (only one implementation ever) costs one extra config field.

## 3. Per-project tmux socket isolation

### The problem

Orca uses the global tmux server. Sessions share namespace with the operator's regular tmux sessions. Two orca invocations on different projects can collide. Watch has to filter through all tmux sessions to find orca-related ones.

### What Orca should do

Use `tmux -L orca-<project>` for all tmux operations. tmux's `-L` flag creates a separate server instance with its own session namespace. Each project gets full isolation.

Benefits:
- Session names can be simple (no project prefix needed for uniqueness)
- Watch scanning becomes: list sessions on the `orca-<project>` socket — all are relevant
- Multiple orca invocations on different projects are fully isolated
- The operator's personal tmux sessions are unaffected

The socket name should be configurable (for the case where the operator wants to share a socket across projects). Default to `orca-<project>`.

### Tradeoffs

| Gain | Cost |
|---|---|
| Complete session isolation between projects and personal use | Can't see all orca sessions with a plain `tmux ls` — need `tmux -L orca-<project> ls` |
| Watch scanning is simpler and faster | The operator agent (if it lives in a personal tmux session) is on a different server than the workers — jumping between them requires specifying the socket |
| No naming collisions | One more thing to remember when debugging: which socket am I on? |
| Enables true multi-project parallel operation | If the operator wants a unified view across projects, they need Watch (or must know all socket names) |

The "can't see everything with plain `tmux ls`" is the real cost. It pushes the operator toward Watch (which knows all sockets) and away from raw tmux commands. This is probably fine if Watch is working well, but it's a real friction increase for ad-hoc debugging.

## 4. Drain timeout on stop

### The problem

`orca.sh stop` kills agents immediately. An agent mid-merge, mid-test-run, or mid-commit produces a messy state — partially merged branches, uncommitted changes, broken worktrees.

### What Orca should do

Make drain-then-kill the default stop behavior:

1. `orca.sh stop` sets a drain flag (a file, e.g., `$ORCA_HOME/.drain`) that the agent loop checks between runs
2. Workers finish their current run but don't claim new work
3. Wait up to `ORCA_DRAIN_TIMEOUT` (default 5m) for all workers to exit
4. Force-kill any remaining sessions
5. `orca.sh stop --force` skips the drain and kills immediately (current behavior)

Orca already has a drain mode concept. The change is making `stop` use it by default.

### Tradeoffs

| Gain | Cost |
|---|---|
| No more killed-mid-merge states | `stop` is no longer instant — operator must wait up to 5 minutes (or use `--force`) |
| Cleaner worktree state after shutdown | A stalled agent blocks shutdown until the timeout expires |
| Agents can write final summaries and close issues properly | More complex stop logic — need to track which agents have exited and which are still draining |
| Reduces manual cleanup after restarts | The drain flag file is a new coordination mechanism that must be checked reliably |

The 5-minute wait on `stop` is the main UX cost. When you want to stop and go to lunch, waiting for a stalled agent is frustrating. The `--force` escape hatch is essential. Consider also: if drain timeout expires, log *which* agent was still running and *what* it was doing — that information is valuable for understanding what went wrong.

## 5. Idle timeout and crash-loop detection

### The problem

If an agent stalls (infinite loop, stuck on a broken merge, burning tokens without progress), the operator has to notice manually. If an agent keeps crashing and restarting on the same broken task, it burns tokens and compute in a loop until someone intervenes.

### What Orca should do

Implement two lightweight health checks:

**Idle detection**: `metrics.jsonl` already has timestamps and durations for every run. A periodic check (every 60s) compares the current run's elapsed time against a threshold (configurable, default: hard cap like 30m, or 2× the median run duration). When triggered: kill the session, mark the issue as failed (or return it to the queue), log the event.

**Crash-loop detection**: Track consecutive failures per agent in a simple counter file. If an agent fails N times (default 3) within M minutes (default 30), stop restarting it and flag it for operator attention. The counter resets on a successful run.

### Tradeoffs

| Gain | Cost |
|---|---|
| Can run overnight without babysitting — stalled agents get killed automatically | False positives: a legitimately long-running task gets killed at the timeout. Some tasks (large refactors, complex merges) genuinely take longer than average. |
| Crash loops don't burn unlimited tokens | Quarantined agents stop doing useful work. If the failure is transient (network blip, API rate limit), the agent would have recovered on the next try but is now stuck. |
| Health state is observable (Watch can display it) | More state to manage — counter files, quarantine flags, threshold configuration |
| Issues that cause crashes get surfaced instead of silently retried | The operator must actively un-quarantine agents, adding a manual step to recovery |

The false positive risk on idle timeout is real. A hard 30-minute cap will kill agents doing legitimate long work. Possible mitigations: per-issue timeout override (a label like `long-running`), or basing the threshold on the specific issue's estimated complexity rather than a global cap. But all mitigations add complexity. Start with a generous hard cap and tighten based on observed run durations.

## 6. Periodic automation

### The problem

Orca has several maintenance tasks that should run periodically but are currently manual or missing:
- Branch cleanup (`gc-run-branches --apply`)
- Stale worktree detection
- Queue health checks (orphan `in_progress` items with no active session)
- Idle timeout and crash-loop detection (see #5)
- Metrics aggregation for the operator agent

### What Orca should do

Implement a simple periodic task runner in the Go rewrite. Each task is a named shell command that runs at a configurable interval:

```go
type PeriodicTask struct {
    Name     string
    Command  string        // shell command to execute
    Interval time.Duration // how often to run
    Enabled  bool
}
```

A goroutine runs each enabled task at its interval. Tasks are defined in config. Output goes to a log file. Failures are counted (for crash-loop-style detection on the tasks themselves).

### Built-in tasks

These should ship as defaults:

| Task | Interval | Purpose |
|---|---|---|
| `health-check` | 60s | Idle timeout + crash-loop detection |
| `queue-audit` | 5m | Orphan `in_progress` items, stale claims |
| `branch-cleanup` | 1h | Prune merged run branches |
| `metrics-summary` | 10m | Write a compact status summary for the operator agent |

### Tradeoffs

| Gain | Cost |
|---|---|
| Maintenance tasks run reliably without operator intervention | Orca gains a long-running process (the task runner goroutine). Currently orca is invoked, does work, exits. A periodic runner changes the execution model. |
| Health checks and cleanup are formalized instead of ad-hoc | Task failures need their own error handling — a broken cleanup script shouldn't crash orca |
| New maintenance tasks are just config entries, not code changes | More configuration surface. Each task has interval, enabled, and command — that's N × 3 settings to manage. |
| The operator agent can rely on up-to-date status summaries | Periodic tasks running during agent work could cause interference — e.g., branch cleanup while an agent is mid-merge. Need to consider locking. |

**Architecture question**: Should the periodic runner be a goroutine in the orca process, or a separate binary (`orca-cron`)? A goroutine is simpler but couples the task runner to orca's lifecycle (if orca stops, tasks stop). A separate binary is more composable but adds deployment complexity. The Go rewrite's architecture should make either possible — define the task runner as a package, wire it into the main binary, and extraction is a future option if needed.

### Design principle

Keep the task runner minimal. If a task needs agent judgment, it should create an issue in the queue and let an agent handle it — not become a complex automated workflow. The periodic runner handles infrastructure; agents handle intelligence.

## 7. Prompt templates with variables

### The problem

Orca has a single static `AGENT_PROMPT.md` (or `ORCA_PROMPT.md` per project). Agent-specific details (name, project, worktree path, queue command) are either hardcoded or injected through environment variables that the agent has to know to read.

### What Orca should do

Use Go's `text/template` for prompt rendering. Define a standard set of variables:

```go
type PromptContext struct {
    Agent       string // agent name
    Project     string // project name
    WorkDir     string // worktree path
    WorkQuery   string // configured work query command (see #2)
    ProjectRoot string // main repo root
    OrcaHome    string // orca installation path
    RunID       string // current run identifier (for workers)
}
```

One prompt template serves all agents and all projects. Per-project customization via `ORCA_PROMPT.md` overrides (checked first, falls back to default template).

Example:
```
You are {{.Agent}} working on {{.Project}}.
Your worktree is at {{.WorkDir}}.

To find available work: {{.WorkQuery}}
```

For the operator agent, the template can include dynamic context:
```
Active workers: {{.WorkerCount}}
Last metrics summary: {{.MetricsSummaryPath}}
```

### Tradeoffs

| Gain | Cost |
|---|---|
| One template serves all agents and projects | Templates are harder to read than plain markdown — `{{.WorkQuery}}` is less obvious than `tq ready --json` |
| Agent-specific details are injected correctly without the agent needing to know about environment variables | Template rendering failures (missing variable, bad syntax) produce confusing errors at a critical moment (agent startup) |
| Per-project overrides work naturally (provide your own template, use the same variables) | The variable set becomes a compatibility surface — adding or removing a variable affects all templates |
| The operator prompt can include live state | Dynamic context in prompts creates a coupling between the prompt and orca internals (what if the metrics summary format changes?) |

The readability cost is real. A static `AGENT_PROMPT.md` is immediately human-readable. A template with `{{.WorkDir}}` requires knowing what gets substituted. Mitigation: `orca.sh prompt --render` command that shows the fully rendered prompt for a given agent and project. This is useful for debugging and for the operator to verify what agents are actually seeing.

## 8. Pre-start hook separation

### The problem

Orca's `agent-loop.sh` combines environment setup, worktree preparation, branch creation, prompt rendering, agent execution, and result parsing in one script. Changing one concern risks breaking another.

### What Orca should do

In the Go rewrite, define clean lifecycle phases:

```go
type AgentLifecycle struct {
    PreStart     []string // shell commands before session creation
    SessionSetup []string // shell commands after session creation
    OnRun        func()   // the agent loop body
    OnDrain      func()   // graceful shutdown signal (see #4)
    OnDeath      func()   // unexpected session death
}
```

**PreStart** is where worktree setup, branch creation, and environment validation happen. It runs before any tmux session exists. If it fails, the agent doesn't start.

**SessionSetup** runs after the tmux session is created but before the agent begins work. This is where tmux configuration (theming, keybindings, status bars) happens. Changing tmux appearance shouldn't require restarting the agent.

**OnDeath** fires when a session dies unexpectedly. This is where crash-loop counting (see #5) and issue re-queuing happen.

### Tradeoffs

| Gain | Cost |
|---|---|
| Project-specific setup (worktree layout, branch naming) lives in hooks, not in the core agent loop | More concepts to understand — "why didn't my agent start?" now requires checking which lifecycle phase failed |
| Tmux configuration changes don't require agent restarts | Hook execution order and error handling become a design surface — what happens if PreStart succeeds but SessionSetup fails? |
| The OnDeath hook is a clean place for crash recovery logic | Shell command hooks are hard to test and debug — errors in hook scripts are a common source of mysterious failures |
| Project-specific customization doesn't require forking orca | More configuration surface. Each agent potentially has three lists of shell commands to maintain. |

The main risk is over-engineering. If most projects never customize hooks, the separation adds complexity without value. Start with PreStart and OnDeath only (the two with clear, distinct responsibilities). Add SessionSetup later if the need arises from real usage.

## 9. Two session types as a first-class concept

### The problem

Orca treats all agents as batch workers. But the actual workflow has two distinct types with different lifecycle needs: the operator agent (persistent, interactive, conversation continuity) and batch workers (ephemeral, fresh session per run, claim-and-execute).

### What Orca should do

Define two agent types in the configuration:

```go
type AgentConfig struct {
    Name     string
    Type     string // "operator" or "worker"
    // ...
}
```

The **operator** type:
- Persistent session with conversation continuity (the agent resumes where it left off)
- Starts first, stops last
- Not managed by the batch scheduler
- Receives status summaries from periodic tasks
- Registered in Watch with a distinct role

The **worker** type:
- Fresh session per run (no memory from previous runs)
- Managed by the batch scheduler (start N, drain on stop)
- Subject to idle timeout and crash-loop detection
- Produces structured artifacts (summary.json, run.log)

### Tradeoffs

| Gain | Cost |
|---|---|
| The implicit two-type pattern gets a name, making it designable and documentable | Adds a type system where there was none. Every future feature must consider "does this apply to operators, workers, or both?" |
| Each type gets appropriate defaults (resume vs fresh, managed vs unmanaged) | The operator type may not need to be in orca's config at all — it's an external session that happens to call orca commands. Formalizing it couples orca to a specific operating style. |
| Watch can display operator and worker sessions differently | If someone uses orca without an operator agent (just `orca.sh start` and `orca.sh status` from a regular shell), the type system is unnecessary overhead |
| Enables type-specific features: health checks for workers only, status summaries for operator only | Two types today, but will there be three tomorrow? Type systems grow. |

This is closely tied to #1 (operator agent). If the operator agent is formalized, having two types is natural. If the operator agent remains implicit ("whatever session I'm in"), this distinction adds complexity for a pattern that not everyone uses. Implement #1 first; if it proves valuable, #9 follows naturally.

## What to skip

These patterns from larger orchestration systems don't fit Orca's philosophy or scale:

| Pattern | Description | Why skip for Orca |
|---|---|---|
| **Continuous reconciliation loop** | A long-running controller that continuously compares desired state to actual state and corrects drift. | Orca's explicit start/stop is simpler. A continuously running controller is overhead for a single-developer tool. Periodic tasks (#6) cover the maintenance needs without a full control loop. |
| **Configuration layering / pack composition** | A system where reusable configuration bundles (packs) are composed, overridden, and merged into a final config. | Orca has one operating model. Configuration layering is platform complexity. One config file per project is enough. |
| **Multi-step workflow templates** | Declarative definitions of multi-step agent workflows with checkpoint recovery and step dependencies. | Valuable at scale. At 1-4 agents, the operator agent handles sequencing in conversation. |
| **Agent-to-agent messaging** | A mail system where agents send structured messages to other agents. | Matters at 20+ agents. At Orca's scale, the operator agent is the communication hub. |
| **Automated iterative refinement** | Bounded loops where the system retries and refines agent output until a quality threshold is met. | Orca's agents complete or escalate (by leaving a summary). No automated retry loops needed. |
| **Multi-provider abstraction** | An abstraction layer that supports running agents on different platforms (tmux, Kubernetes, subprocess) via configuration. | Orca's Go rewrite should keep the runtime interface clean enough that providers *could* be added later, but implementing multiple providers now is premature. Start with tmux. |
| **Metadata-based work routing** | A system where work items are routed to specific agents based on metadata labels and routing rules. | Work routing in Orca is simple: agent claims from queue. No metadata-based routing needed at current scale. |

## Implementation sequence

Aligned with the Go rewrite phases:

### Phase 1: Foundation (during rewrite)

These should be built into the Go architecture from the start, because retrofitting them later is expensive:

1. **Configurable work query** (#2) — define the interface even if initial implementation shells out to `tq`
2. **Per-project tmux socket** (#3) — one flag on all tmux calls
3. **Lifecycle hooks** (#8) — clean separation of pre_start and on_death at minimum
4. **Prompt templates** (#7) — use Go `text/template` from day one

### Phase 2: Operator experience (after core rewrite works)

These improve the operating model but aren't load-bearing:

5. **Operator agent formalization** (#1) — identity registration, dedicated prompt, Watch integration
6. **Two session types** (#9) — operator and worker as config-level concepts
7. **Drain timeout on stop** (#4) — drain flag file, configurable timeout, force-kill fallback

### Phase 3: Robustness (after enough operational experience to know what actually breaks)

These address failure modes. Don't build them until you've seen the failures:

8. **Periodic task runner** (#6) — goroutine-based, config-defined
9. **Idle timeout** (#5) — flag runs exceeding threshold, kill and re-queue
10. **Crash-loop detection** (#5) — quarantine agents that fail repeatedly

## Open questions

1. **Should the operator agent be launched by orca?** Or should orca remain a tool the operator agent calls? The latter is simpler and matches the current model. The former gives orca more control over the operator agent's lifecycle, but it means orca needs to manage interactive sessions — a different kind of work than batch execution.

2. **How should the operator agent receive status?** Options: read a status file (pull), receive a tmux message when something important happens (push), or include status in the prompt template at render time (ambient). The ecosystem principle is "no tool pushes information at the operator." But a notification when a worker quarantines might be worth the interruption. Start with pull (status file from periodic task #6) and add push only if the operator consistently misses important events.

3. **Should periodic tasks be in-process or separate?** A goroutine in the orca process is simplest. But a separate `orca-cron` binary would be more composable and testable, and wouldn't stop running when orca stops. The Go rewrite's architecture should make either possible — define the task runner as a package, wire it in, and extraction is a future option.

4. **Where does the operator prompt live?** Options: in the orca repo (ships with orca), in the target project (project-specific), or generated from config (dynamic). Recommendation: a default ships with orca, projects can override with `OPERATOR_PROMPT.md` in the project root.

5. **How much of this should be in orca vs. Watch?** The operator agent's identity registration, status display, and session management overlap with Watch's concerns. Proposed boundary: orca owns execution and artifacts, Watch owns observation and navigation, the operator agent uses both. Avoid duplicating functionality.
