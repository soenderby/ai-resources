The goal is not to canonize one implementation. It is to capture the recurring structure that shows up across the strongest examples: **one bounded task per loop, explicit state on disk, deterministic verification, and manual review points by default**.

---

## When this pattern fits

Use a Ralph-style loop when:
- the work can be broken into small, bounded increments
- you want a simple control plane rather than multi-agent orchestration
- you need progress to survive between agent invocations
- the main bottleneck is reliable execution, not broad parallel exploration

It is a worse fit when:
- the task is fundamentally breadth-first research or search
- many subproblems can be explored independently in parallel
- the hard part is cross-task coordination rather than single-threaded progress
- the verification story is weak or non-deterministic

---

## Design principles

1. **One task per loop**  
   Do not ask the agent to clear the whole backlog in one go.

2. **Persistent state lives in files**  
   Plans, specs, logs, and progress survive outside the chat session.

3. **Verification is explicit**  
   Each loop should have a concrete way to check whether it improved the system.

4. **Human pause is the default**  
   Continuous looping is possible, but quality-first use keeps a review point between iterations.

5. **Prompts are operational artifacts**  
   Expect to tune them as you learn failure modes.

Notes:

- `fix_plan.md` is the current prioritized task ledger for the loop.
- `specs/` holds product or behavior requirements.
- `.ralph/logs/` gives you an auditable trail of iterations.

## `scripts/ralph-loop.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

MODE="${1:-build}"          # build | plan
MAX_ITERS="${MAX_ITERS:-0}" # 0 = infinite
AUTO="${AUTO:-0}"           # 0 = manual pause, 1 = continuous

mkdir -p .ralph/logs
ITER=0

while :; do
  ITER=$((ITER + 1))
  TS="$(date +%Y%m%d-%H%M%S)"
  LOG=".ralph/logs/${TS}-${MODE}-iter${ITER}.log"

  echo "=== ITERATION $ITER ($MODE) ==="

  PROMPT_FILE="prompts/${MODE}.md"
  if [ ! -f "$PROMPT_FILE" ]; then
    echo "Missing $PROMPT_FILE"
    exit 1
  fi

  scripts/run-agent.sh "$PROMPT_FILE" | tee "$LOG"

  echo "--- verify ---" | tee -a "$LOG"
  if scripts/verify.sh | tee -a "$LOG"; then
    echo "Verification passed." | tee -a "$LOG"
  else
    echo "Verification failed. Check logs and tune prompts/specs." | tee -a "$LOG"
  fi

  if [ "$MAX_ITERS" -gt 0 ] && [ "$ITER" -ge "$MAX_ITERS" ]; then
    echo "Reached MAX_ITERS=$MAX_ITERS"
    break
  fi

  # Quality-first default: pause for review between iterations.
  if [ "$AUTO" -eq 0 ]; then
    read -r -p "Press Enter for next loop (Ctrl+C to stop)... " _
  fi
done
```

This intentionally keeps the control plane thin:
- pick mode
- run one prompt
- verify
- log
- optionally pause

If you keep adding supervisor behavior into this shell script, you may be drifting away from the reason to use Ralph in the first place.

## `prompts/build.md`

```markdown
Study @specs/*, @AGENTS.md, and @fix_plan.md first.

Your task:
1) Choose exactly ONE highest-priority incomplete item from @fix_plan.md.
2) Before coding, search the codebase thoroughly. Do not assume missing functionality.
3) Implement the item fully. Avoid placeholders and fake completions.
4) Run the most relevant targeted tests for the changed unit.
5) If tests fail, fix the issue or document the blocker in @fix_plan.md.
6) Update @fix_plan.md status for the item you worked on.
7) If you discover a stable reusable build/test command, add it briefly to @AGENTS.md.

Rules:
- One item per loop.
- Prefer small, reviewable diffs.
- Keep changes aligned with specs.
- If uncertain, note the uncertainty in @fix_plan.md and proceed conservatively.
- Do not silently rewrite priorities.
```

## `prompts/plan.md`

```markdown
Planning mode only. Do not do major implementation.

1) Study @specs/*, @AGENTS.md, and the current code.
2) Audit @fix_plan.md for:
   - duplicates
   - stale items
   - wrong priority
   - missing critical work
3) Rewrite @fix_plan.md into a prioritized, actionable checklist.
4) For each item include:
   - a short task statement
   - definition of done
   - expected verification method
5) Keep the list compact and high-signal.
```

## `fix_plan.md` template

```markdown
# Fix Plan

## P0
- [ ] Parser handles X edge case
  - Done when: tests A and B pass
  - Verify: `npm test -- parser`

## P1
- [ ] Refactor module Y for spec compliance
  - Done when: behavior matches `specs/product-spec.md` section 3
  - Verify: unit tests + lint
```

The plan file should be:
- small enough to scan quickly
- concrete enough to execute without reinterpretation every loop
- easy to revise when reality changes

## Run

```bash
chmod +x scripts/*.sh

# Planning pass (1–2 loops)
MAX_ITERS=2 ./scripts/ralph-loop.sh plan

# Build pass (manual review between iterations)
./scripts/ralph-loop.sh build
```

If you later switch to fully automatic looping, add stronger guardrails first: explicit completion conditions, failure thresholds, rate-limit handling, and a clean stop condition. The repo examples suggest that these are where the simple loop starts to harden into a production system.
