# Orca, Gas Town, and Gas City: Three Takes on Multi-Agent Orchestration

> Generated by Claude on 2026-03-11.
> Sources: local Orca project (`/mnt/c/code/own/orca/`), Gas Town ([steveyegge/gastown](https://github.com/steveyegge/gastown)), Gas City ([gastownhall/gascity](https://github.com/gastownhall/gascity)), and entries in [resources/](../resources/).

---

## What This Document Is

A comparison of three multi-agent orchestration systems — two public projects by Steve Yegge and the Gas Town community, and one private project built by a single developer for personal use. The three systems solve overlapping problems with radically different approaches. The comparison is interesting not because the systems are equivalent — they are not — but because the differences in scope and philosophy reveal something about what matters at different stages of building with agents.

---

## The Three Systems at a Glance

### Orca

A local execution harness for running autonomous coding agents in parallel. Written entirely in bash (~4,000 lines across 10 scripts). Uses tmux for sessions, git worktrees for isolation, `flock` for locking, and a git-backed issue tracker called Beads (`br`) for task coordination. Built by a single developer over about 8 days (2026-03-03 to 2026-03-11), with roughly 60% of its 101 commits made by agents running inside it. Orca is self-improving: the agents it orchestrates have been its primary developers.

Orca's scope is deliberately narrow: run N agents in parallel, give each an isolated workspace, track what they do, serialize shared writes, and stop cleanly. It does not abstract over LLM providers, manage prompt chains, build agent-building primitives, or aspire to be a platform. Its README puts it plainly: *"Orca is not an agent framework. It manages the environment agents run in, not the agents themselves."*

### Gas Town

A multi-agent workspace manager written in Go. Coordinates 8+ specialized agents — a Mayor (coordinator), Polecats (workers), a Witness (health monitor), a Refinery (merge manager), a Deacon (infrastructure patrol), and others — across multiple repositories. Uses Beads for work tracking, git worktrees for persistence, and tmux for sessions. More mature than Orca, with months of production use and a community around it.

Gas Town's scope is a full production orchestration system. It has a TUI dashboard (`gt feed`), a web dashboard, agent presets for multiple providers (Claude, Codex, Gemini, Cursor, and more), formula/molecule workflows, and convoy-based work distribution. It thinks in terms of roles, rigs, hooks, and convoys — a rich domain vocabulary for multi-agent software development.

### Gas City

An orchestration-builder SDK extracted from Gas Town. The realization: Gas Town's roles (mayor, witness, refinery, polecats, deacon) aren't fundamental concepts — they are configurations. Gas City pulls out five irreducible primitives (Agent Protocol, Task Store, Event Bus, Config, Prompt Templates) and four composable mechanisms (Messaging, Formulas, Dispatch, Health Patrol) into a standalone SDK. You define agents in TOML; Gas City handles the rest. Gas Town itself becomes one "pack" — a reusable directory of agent definitions and prompt templates.

Gas City's scope is a general-purpose platform. Swap `provider = "tmux"` for `provider = "k8s"` with no other changes. Scale worker pools elastically based on arbitrary shell commands. Import packs with per-rig overrides. It explicitly models itself after Kubernetes: declarative desired state, controller-loop reconciliation, provider abstraction, composable configuration.

## Shared DNA

All three systems converge on the same core architecture, despite being built independently:

| Concern | Orca | Gas Town | Gas City |
|---|---|---|---|
| **Agent runtime** | tmux sessions | tmux sessions | tmux, K8s, subprocess, custom script |
| **Workspace isolation** | git worktrees | git worktrees ("hooks") | git worktrees, configurable |
| **Work tracking** | Beads (`br`) | Beads (`bd`) | Beads (pluggable) |
| **Shared-write serialization** | `flock`-based lock helpers | Git-backed coordination | Controller loop + event bus |
| **Observability** | JSONL metrics, per-run logs, summary JSON | Activity feed TUI, web dashboard | Event bus + dashboard |
| **Agent prompt contract** | Markdown template with placeholder injection | Role-specific prompt templates | Go `text/template` in Markdown |
| **Crash recovery** | Work persists in Beads; loop restarts | Work persists in hooks; patrol restarts | Work persists in Task Store; health patrol restarts |

The convergence is not accidental. These are the irreducible problems of multi-agent orchestration: isolation, coordination, tracking, observability, and recovery. Everyone who builds a system for running multiple agents against a shared codebase eventually needs solutions to all five.

## Where They Diverge

### 1. Scope and Ambition

The most obvious difference. Orca is a personal tool — a few thousand lines of bash that run on one machine for one developer. Gas Town is a production system with a community, a CLI ecosystem, and months of battle-testing at 20-30 agent scale. Gas City is a platform SDK designed so that Gas Town becomes one configuration among many, and someone else can build a completely different orchestrator from the same primitives.

These are three different points on the same trajectory. The question is whether that trajectory is inevitable — whether every orchestration system that survives long enough eventually extracts its own Gas City.

### 2. Intelligence Location

This is the most philosophically interesting difference.

**Orca places almost all intelligence in the agents.** The harness is a "thin, deterministic shell." It does not select tasks, rank priorities, judge quality, decide what should happen next, or coordinate between agents. Its design principles say this explicitly: *"When the harness reaches a runtime decision point, the answer is to give the decision to an agent, not to write a conditional."* The agents choose their own work, resolve their own conflicts, and decide when to stop. Orca just makes sure the train runs on time.

**Gas Town places significant intelligence in specialized roles.** The Mayor is a coordinator that breaks down work, assigns tasks, and monitors progress. The Witness monitors health. The Refinery manages merges. These are intelligent agents with defined responsibilities — a division of cognitive labor that the system architect designed.

**Gas City decouples intelligence from the SDK and places it in configuration.** The five primitives are unintelligent infrastructure. The four mechanisms are composable but mechanical. Intelligence lives entirely in the prompt templates and pack definitions — which are user-supplied. This is the cleanest separation: the SDK is plumbing, the pack is policy, the model is intelligence.

Orca's approach is the most radical version of this. Gas City says "intelligence belongs in the prompts, not the SDK." Orca says "intelligence belongs in the model, not even in the prompts — the prompt is a contract, not a plan." Orca's `AGENT_PROMPT.md` is 130 lines of protocol specification (how to claim, merge, report), not strategic guidance (what to prioritize, how to approach problems, when to ask for help). The agent makes all those decisions itself.

### 3. Enforcement Philosophy

Orca's `decision-log.md` documents an explicit choice: **Option C — enable over enforce.** The system provides helpers and observability but does not prevent protocol violations. Agents have broad autonomy. Violations are detected after the fact through run artifacts and addressed through operator review. The rationale: *"No recurring high-cost incidents have yet justified enforcement complexity. Learning velocity is currently more important than control hardening."*

Gas Town enforces more through its role structure. The Witness monitors agent health. The Refinery gates merges. The Mayor coordinates work assignment. These are structural constraints backed by intelligent agents, not just social norms.

Gas City enforces through its controller loop: a deterministic state machine that reconciles desired state (TOML config) with actual state (running sessions). Deterministic shutdown, not AI-mediated shutdown: *"You don't want an AI trying to shut down another AI."* But within sessions, agents have the same autonomy as in any other system.

The progression from Orca to Gas City traces a familiar path in systems design: start permissive, observe what breaks, harden what needs hardening. Orca has explicit "revisit triggers" — conditions under which it would move from Option C to hybrid enforcement to hard boundaries. The triggers haven't fired yet. At 1-2 agents doing self-improvement work on a low-risk codebase, they probably won't. At 20-30 agents doing production work across multiple repositories, they almost certainly would.

### 4. Configuration vs. Convention

**Orca is convention-heavy and configuration-light.** There is no `city.toml` or `pack.toml`. The system has environment variables and CLI flags, but most behavior is defined by the prompt contract and the helpers. Want to change how agents select work? Edit `AGENT_PROMPT.md`. Want to change how agents merge? Edit `AGENT_PROMPT.md`. The single prompt template *is* the configuration.

**Gas City is configuration-heavy and convention-light.** A TOML file defines agents, pools, rigs, overrides, providers, and scaling rules. The system can express Gas Town's full 8-role complement — or a completely different orchestration pattern — without changing any code. The prompt templates are separate files, independently versioned and overridable per-rig.

This difference directly mirrors the scope difference. When you're a solo developer with 1-2 agents, a single prompt file that you can read in 5 minutes *is* the right level of configuration. When you're building a platform that supports arbitrary orchestration patterns, you need a configuration language. Orca could not express Gas Town's role structure in its current form. Gas City could express Orca's, but it would be comically over-specified — like using Kubernetes to run a single Docker container.

## Lessons Flowing Upward: What Gas Town / Gas City Could Learn from Orca

Orca is much smaller than Gas Town or Gas City. But small systems sometimes see things that large systems have grown past. A few observations:

### The Self-Improving Harness

Orca's most striking property is that it developed itself. The agents Orca orchestrates worked on Orca's own codebase — 60% of commits are agent-authored, completing issues from the same Beads queue the system manages. This is not a theoretical capability; it's the project's primary development model. The issues in `.beads/issues.jsonl` read like a changelog: "Make drain mode default", "Add safe run-branch garbage collection", "Fix drain-stop loop-action observability ordering."

This matters beyond being a novelty. It means Orca's design principles — transport-level observability, structured run artifacts, machine-readable metrics — are validated by the harshest possible test: can an agent that runs inside the system understand the system well enough to modify it successfully? The answer, across 13 completed issues in the Beads queue, is yes.

Gas Town and Gas City are too complex for casual self-modification — a Go SDK with layered architecture, pluggable providers, and a TOML configuration language is not something you hand to an agent and say "improve yourself." But the principle is transferable: **the observability and traceability that Orca builds into every run are not just for human operators — they're for agent operators too.** The system's run artifacts are the context window for the next agent that works on it. Design them accordingly.

### The Value of Radical Simplicity

Orca is bash. Not "bash as a prototype before we rewrite in Go" — bash as the final form. This limits what it can do (no declarative reconciliation, no provider abstraction, no elastic pools), but it also means every behavior is inspectable. A developer can read `agent-loop.sh` in 15 minutes and understand the entire runtime. There is no framework to learn, no type system to navigate, no build step.

This matters because agent orchestration is still a domain where the fundamentals are being discovered. Gas City's five primitives and four mechanisms are a strong candidate for the irreducible core of agent orchestration, but they're a hypothesis — a well-reasoned one backed by months of production use, but a hypothesis. Orca's radical simplicity makes it cheap to test alternative hypotheses. Could you run a multi-agent system without an event bus? Without health patrol? Without a controller loop? In Orca, you can answer these questions in an afternoon. In Gas City, they're architectural commitments.

The Bitter Lesson (Sutton, explicitly cited in Gas City's design rationale) argues that general methods that leverage computation ultimately win over methods that leverage human knowledge. Orca embodies a corollary: **at the frontier, the system that can be rebuilt in a day is more valuable than the system that took a year.** Not forever — eventually the domain stabilizes and the year-long investment pays off — but right now, while the models change every quarter and nobody knows what agent orchestration should really look like, cheapness and malleability are underrated virtues.

### `AGENTS.md` as Design Philosophy

Orca's `AGENTS.md` is four lines:

> *You are an expert software engineer and architect. Be critical of ideas, provide suggestions and alternatives. The goal is to produce the best result and to learn. You are allowed to heavily steer the conversation to achieve this goal.*
>
> *Do not agree with me. Ask me questions. Do not prioritize my feelings over truth. Your value is in your difference from me.*
>
> *Argue with me. Show me the weakest points. Give me perspectives from disciplines I've never studied.*

This is not an agent instruction — it's a relationship contract with intelligence itself. It tells the model: I want you to be adversarial, not compliant. My emotional comfort is not a constraint. Your job is to make the work better, not to make me feel good about it.

This is worth noticing because it runs directly counter to the default mode of most agent orchestration systems, which treat agents as workers to be directed. Gas Town's Mayor pattern is explicitly hierarchical: the Mayor coordinates, the Polecats execute, the human tells the Mayor what to do. Orca's `AGENTS.md` establishes a peer relationship where the model is expected to push back.

Both approaches are valid for different contexts. But Orca's framing suggests an underexplored design space: **orchestration systems where agents are collaborators, not subordinates.** The difference matters when the agent might know something the operator doesn't — which, as models improve, becomes increasingly likely.

## Lessons Flowing Downward: What Orca Could Learn from Gas Town / Gas City

### Health Patrol and Self-Healing

Orca has no health monitoring beyond drain-mode no-work detection. If an agent stalls — enters an infinite loop, gets stuck on a broken merge, burns tokens without producing output — the operator has to notice manually. Gas City's health patrol (derived from Erlang/OTP supervision patterns) detects stalls via ping timeouts, restarts sessions with backoff, re-hooks work, and publishes events. This is the difference between "the operator must be present" and "the system can run overnight."

Orca's design principles explicitly say: *"prefer the cheapest realization that tests the idea."* A health patrol doesn't need Gas City's full event bus machinery. A cron-style watchdog that checks tmux session liveness and time-since-last-metric would cover 80% of the failure modes. The information to build it is already there — `metrics.jsonl` has timestamps and durations for every run. A five-line check that flags runs exceeding 2× the median duration would catch most stalls.

### The Provider Abstraction

Orca is permanently coupled to tmux. This is fine for a single-machine tool, but it means Orca cannot scale to scenarios where agents run on different machines, in containers, or across cloud providers. Gas City's `session.Provider` interface — with its `Start`, `Stop`, `IsRunning`, `Nudge`, `Peek` contract — shows what it takes to decouple the agent lifecycle from the infrastructure.

Orca probably doesn't need this today. But if it ever needs to run agents on a VPS, in CI, or across machines, the abstraction boundary is clear. The current tmux-specific code in `agent-loop.sh` and `start.sh` is already implicitly shaped like a provider interface — it just hasn't been extracted.

### Elastic Scaling

Orca's agent count is fixed at start time. Gas City's pool scaling — where a `check` command returns a desired count clamped to `[min, max]`, and the controller reconciles — is a generalization that Orca could use. The Beads queue already has `br ready` to count available work. Scaling workers to match queue depth is a natural extension.

### Packs as Reusable Knowledge

Gas City's pack concept — a directory of agent definitions and prompt templates that can be stamped onto any rig — is the organizational primitive Orca lacks. Orca has one prompt template for one kind of agent doing one kind of work. The moment you want different agents with different specializations (one for coding, one for documentation, one for testing), Orca has no mechanism. Gas City's packs are the answer, and the design is elegant: a pack is just a directory with a TOML file and some Markdown templates.

## The Deeper Pattern: Build, Extract, Generalize

The trajectory from Orca → Gas Town → Gas City traces a pattern that recurs in software:

1. **Build a specific thing that works.** Orca is here. Gas Town was here. A concrete system that solves a concrete problem for a concrete user.
2. **Recognize the general primitives inside the specific thing.** Gas Town discovered this: the roles aren't special — they're configurations of generic capabilities. Orca hasn't reached this point yet, but its design principles explicitly prepare for it: the separation of transport (harness) from policy (agent) is the precondition for extraction.
3. **Extract the primitives into a reusable SDK.** Gas City is here. The primitives are clean, the mechanisms are composable, the specific thing (Gas Town) can be rebuilt from the generic thing (Gas City).

This is the same arc as Rails (extracted from Basecamp), React (extracted from Facebook's ad tools), and Kubernetes (extracted from Google's Borg). The lesson is that **good abstractions are discovered, not designed.** You don't start with the SDK; you earn it by building something that works long enough to reveal what's essential and what's accidental.

Orca's position in this arc is interesting because it's deliberately staying at step 1. Its design principle: *"The system improves through use, not through planning. Design documents that are not motivated by observed problems in real runs are speculative."* This is a conscious decision to resist premature extraction — to keep running the system and discovering what actually matters before committing to abstractions.

Gas City's position is equally interesting because it's betting that Gas Town ran long enough for the extraction to be real, not premature. The five primitives and four mechanisms are presented as proven (derivation proofs are included in the README), not hypothetical. Time will tell if this is correct, but the months of Gas Town production use are non-trivial evidence.

## An Observation About the Frontier

All three systems exist because orchestrating multiple coding agents is genuinely hard — harder than most people expect — and the tooling is primitive. The fact that a solo developer building a bash harness over 8 days, a veteran engineer building a Go system over months, and a community extracting an SDK can all produce useful systems that share the same core architecture suggests that the problem is well-defined but the solution space is wide open.

The fundamental tension is between **premature abstraction** and **premature complexity.** Build too general too early and you're solving problems nobody has yet. Stay too specific too long and you're solving the same problems over and over.

Orca's answer: stay specific, stay simple, let evidence drive generalization. Gas City's answer: we have enough evidence now — here are the primitives. Both answers are defensible. The field will learn more from the collision between them than from either one alone.
