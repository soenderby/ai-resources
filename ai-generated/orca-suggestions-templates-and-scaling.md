# Orca: Two Concrete Suggestions

> Generated by Claude (claude-sonnet-4-20250514) on 2026-03-11.
> Context: produced after a [comparison analysis](orca-gastown-gascity-comparison.md) of Orca, Gas Town, and Gas City. These suggestions are informed by patterns observed in the larger systems but designed to stay within Orca's design philosophy — small, evidence-driven, no premature abstraction.

## 1. Work Approach Templates

### The Problem

`AGENT_PROMPT.md` is 130 lines of coordination protocol — how to claim, merge, and report. It says nothing about *how to approach work*. Every agent gets identical behavioral framing regardless of whether the issue is a race condition fix, a documentation update, or a test coverage improvement. The agent infers the right approach entirely from the issue description.

This works — 13/13 completed issues demonstrate it. But it means the agent is doing two things simultaneously: figuring out what the work is, and figuring out how to work. Explicit approach guidance could reduce variance on specific task types and improve quality where the issue description doesn't fully specify "how" (most of them).

### Why This Matters Beyond the Obvious

This is also a cheap experiment that tests whether Orca's single-template architecture is a deliberate simplification or an accidental constraint. If approach-guided agents produce noticeably different outcomes (fewer follow-ups, smaller diffs, better test coverage), that's a signal about where to invest. If they don't, that's equally valuable — it means the model's own judgment is sufficient and adding complexity here is waste.

### Suggested Implementation

**Add a single placeholder to the existing template.** No second prompt file, no per-agent routing, no configuration language.

1. **Add `__WORK_APPROACH__` placeholder to `AGENT_PROMPT.md`.** Place it after the "Execution Workflow" section. When no approach file is provided, the placeholder resolves to empty — current behavior is preserved exactly.

2. **Add env var `ORCA_WORK_APPROACH_FILE`.** Points to a markdown snippet file. Default: empty (no approach guidance).

3. **Substitute in `write_prompt_file()`.** In `agent-loop.sh`, the function already does placeholder substitution for 14 placeholders. Add one more:

   ```bash
   local approach_text=""
   if [[ -n "${ORCA_WORK_APPROACH_FILE:-}" && -f "${ORCA_WORK_APPROACH_FILE}" ]]; then
     approach_text="$(cat "${ORCA_WORK_APPROACH_FILE}")"
   fi
   prompt_text="${prompt_text//__WORK_APPROACH__/${approach_text}}"
   ```

4. **Pass through `start.sh`.** Add `ORCA_WORK_APPROACH_FILE` to the env vars injected into each tmux session (same pattern as `ORCA_PRIMARY_REPO` and friends).

5. **Create two starter approach files.** These are examples, not prescriptions — the operator edits them based on what they learn:

   **`approaches/code.md`**:
   ```markdown
   ## Work Approach: Code Changes

   - Run existing tests before changing anything. Know what passes now.
   - Keep the diff minimal. If you see an improvement opportunity outside the issue scope, create a follow-up instead of fixing it.
   - If you're uncertain about a design choice, document the alternatives you considered in a `br comments add` note rather than guessing silently.
   - After implementation, verify your change didn't break unrelated tests.
   ```

   **`approaches/docs.md`**:
   ```markdown
   ## Work Approach: Documentation

   - Read all files referenced by the issue before writing anything.
   - Check that your changes are consistent with README.md, OPERATOR_GUIDE.md, and docs/design.md.
   - Do not modify code files. If you discover a code issue, create a follow-up.
   - Ensure examples in documentation actually match current CLI behavior and script interfaces.
   ```

### Scope of Change

- `agent-loop.sh`: ~5 lines (read file, substitute placeholder)
- `start.sh`: ~3 lines (env var passthrough)
- `AGENT_PROMPT.md`: 1 line (placeholder insertion)
- `approaches/`: 2 new short markdown files
- `README.md`: mention the new knob in the Runtime Knobs section

### How to Evaluate

Compare runs with and without approach guidance on similar issue types. The data is already in `metrics.jsonl`:
- **Duration and token usage** — does guidance make agents faster or slower?
- **Result distribution** — fewer `failed` or `blocked` outcomes?
- **Discovery count** — does the code approach produce fewer follow-ups (tighter scope)?
- **Qualitative** — read the `summary.md` and diffs. Is the work different?

Run at least 5-10 issues with each configuration before drawing conclusions. A single good or bad run proves nothing.

### What This Is Not

- Not per-agent template assignment. All agents in a batch share the same approach. The operator chooses.
- Not automatic task-type routing. The harness doesn't classify issues. Consistent with design principle 4: *"Do not encode runtime reasoning in the harness."*
- Not a pack system. There's no TOML, no directory convention, no composition model. It's a text file injected into a placeholder.

## 2. Queue-Aware Start

### The Problem

Agent count is fixed at start time. If you run `orca start 3 --continuous` with 1 ready issue, two agents launch, discover there's nothing to do, burn tokens syncing and scanning the queue, and drain. Looking at current metrics: 8 of 24 runs (33%) were `no_work`. Each `no_work` run still costs — the two recent failed runs burned ~10,000 tokens each to accomplish nothing.

The operator has to predict the right agent count before starting, but the queue is dynamic. This prediction is always somewhat wrong.

### Suggested Implementation: Two Stages

**Stage 1: Start-time capping (do this first).** Before the agent-launch loop in `start.sh`, check queue depth and don't launch more agents than there are ready issues.

Add after `validate_all_worktrees_before_launch` and before the launch loop:

```bash
# Cap launch count to queue depth unless overridden
if [[ "${ORCA_FORCE_COUNT:-0}" != "1" ]]; then
  queue_depth="$(br ready --json 2>/dev/null | jq 'length' 2>/dev/null || echo "")"
  if [[ -n "${queue_depth}" && "${queue_depth}" =~ ^[0-9]+$ ]]; then
    running_count=0
    for i in $(seq 1 "${COUNT}"); do
      if tmux has-session -t "${SESSION_PREFIX}-${i}" 2>/dev/null; then
        running_count=$((running_count + 1))
      fi
    done
    available_work=$((queue_depth - running_count))
    if (( available_work < 0 )); then
      available_work=0
    fi
    needed=$((running_count + available_work))
    if (( needed < COUNT )); then
      echo "[start] capping to ${needed} agents (${queue_depth} ready issues, ${running_count} already running)"
      COUNT="${needed}"
    fi
  fi
fi
```

The `ORCA_FORCE_COUNT=1` escape hatch lets the operator override when they know work is coming (e.g., about to bulk-add issues).

**Why stage 1 is enough for now:** drain mode already handles the stop side. When agents run out of work, they stop. The waste is on the *start* side — launching agents that immediately discover there's nothing to do. Capping at launch time eliminates that.

**Stage 2: Runtime scaler (do this only if stage 1 isn't enough).** A separate script — not inside the harness — that can be called to add agents when the queue grows during operation.

New command: `orca scale --max N`

Behavior:
1. Count ready issues (`br ready --json | jq length`)
2. Count running agent sessions (`tmux ls | grep -c "^${SESSION_PREFIX}-"`)
3. If ready issues exceed running agents and running < max, call `start.sh` with the higher count (idempotent — `start.sh` already skips running sessions)
4. If ready issues are at or below running agents, do nothing (drain mode handles the down side)
5. One-shot execution, not a daemon

```bash
#!/usr/bin/env bash
set -euo pipefail

MAX="${1:?Usage: scale.sh <max-agents>}"
SESSION_PREFIX="${SESSION_PREFIX:-orca-agent}"
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

queue_depth="$(br ready --json 2>/dev/null | jq 'length' 2>/dev/null || echo "0")"
running="$(tmux ls 2>/dev/null | grep -c "^${SESSION_PREFIX}-" || echo "0")"

desired=$((queue_depth < MAX ? queue_depth : MAX))

if (( desired > running )); then
  echo "[scale] scaling from ${running} to ${desired} agents (${queue_depth} ready issues, max ${MAX})"
  ORCA_FORCE_COUNT=1 "${SCRIPT_DIR}/start.sh" "${desired}" --continuous
else
  echo "[scale] no scaling needed (${running} running, ${queue_depth} ready, max ${MAX})"
fi
```

For continuous scaling, the operator wraps it: `watch -n 60 ./orca.sh scale 3`. This keeps the scaler stateless and the harness ignorant of scaling concerns. Consistent with design principle 1: the harness handles transport, not decisions.

### Scope of Change

**Stage 1 only:**
- `start.sh`: ~15 lines (queue depth check, cap logic, force override)
- `README.md`: document `ORCA_FORCE_COUNT` knob and capping behavior

**Stage 2 (if needed later):**
- New `scale.sh`: ~20 lines
- `orca.sh`: add `scale` subcommand dispatch
- `README.md`: document `orca scale` command

### How to Evaluate

After implementing stage 1, check:
- **`no_work` rate** — should drop significantly. Compare the ratio of `no_work` to total runs before and after.
- **Tokens wasted** — sum tokens on `no_work` runs before and after.
- **Operator friction** — does the capping ever prevent you from launching agents you actually wanted? If so, how often do you reach for `ORCA_FORCE_COUNT`?

If you find yourself repeatedly wishing agents would start mid-session when new work arrives, that's the signal to build stage 2.

### What This Is Not

- Not a controller loop. There is no reconciliation daemon, no desired-state file, no continuous convergence. Orca stays a batch launcher.
- Not elastic pool scaling. Gas City's `check` command model with `[min, max]` clamping is more general but also more complex. This is the 20% of that idea that covers 80% of the waste.
- Not a down-scaler. Drain mode already stops agents when work runs out. The problem is on the start side, not the stop side.

---

## Implementation Order

1. **Stage 1 of queue-aware start.** Smallest change, most immediate payoff, zero risk. Can be evaluated from the next session's `metrics.jsonl`.
2. **Work approach templates.** Slightly more involved but still small. Needs a few sessions to evaluate properly.
3. **Stage 2 scaler.** Only if stage 1 proves insufficient — meaning you observe the queue growing during sessions and wish agents would start automatically.

All three suggestions are independent. They can be implemented in any order or in isolation. None of them changes Orca's architecture, adds a dependency, or introduces a new concept. They sharpen existing behavior.
