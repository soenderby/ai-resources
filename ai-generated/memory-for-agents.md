# Memory for Agents

## What This Document Is

A scoped synthesis of what this repository currently suggests about **agent memory**, with an emphasis on practical systems rather than a comprehensive literature review.

This is **not** a definitive survey of the full field of long-term memory in AI agents. The collection is not broad enough for that yet. It *is* broad enough for something useful: a grounded account of the memory patterns that recur across the sources already in this repo, where the practical evidence is strongest, where the uncertainties are still large, and what experiments seem worth trying.

The center of gravity here is:

- coding-agent systems
- local-first / file-based memory
- task and coordination memory
- the relationship between memory and context engineering

## Executive Summary

The current collection supports five main claims.

1. **Memory is a core component of agent systems.** [Lilian Weng’s *LLM Powered Autonomous Agents*](../resources/articles.md#llm-powered-autonomous-agents) still provides the clearest architectural framing: memory sits alongside planning, tool use, and action. Across the practical sources, the same pattern appears: once tasks become longer, more cumulative, or more parallel, pure in-context reasoning stops being enough.

2. **The strongest practical evidence in this repo is for external task memory.** The most convincing memory systems here are not rich autobiographical architectures but durable work records: [Beads](../resources/projects.md#beads), [Gastown](../resources/projects.md#gastown), [Gas City](../resources/projects.md#gas-city), and [BeadHub](../resources/articles.md#beadhub-coordination-for-ai-programming-teams). They all converge on the same need: tasks, dependencies, discovered work, and resumption have to survive across sessions.

3. **“Agent memory” is doing at least three different jobs.** The sources repeatedly point to three distinct things that get called memory: **task memory**, **context memory**, and **knowledge memory**. Keeping those separate makes the topic much easier to reason about.

4. **Local-first, inspectable memory looks more promising than opaque platform memory.** A strong thread in the collection is that useful memory should often live in files, repositories, or local data stores that humans and tools can inspect. The practical advantage is not ideology but auditability, portability, and composability.

5. **The evidence is much weaker for richer long-term memory claims than for simpler persistence patterns.** The collection contains intriguing ideas about more semantic or human-like memory, but the practical evidence here is still much stronger for explicit external memory than for sophisticated internal cognitive analogues.

In short: **this repo strongly supports memory as durable external structure, and only weakly supports memory as a rich internal cognitive system.**

## Three Kinds of Agent Memory

## 1. Task memory

This is the clearest and strongest form in the repo.

Task memory answers questions like:

- What work exists?
- What is ready now?
- What depends on what?
- What has already been tried?
- What new work was discovered?
- What can another agent or a future session resume?

This is the logic behind [Introducing Beads: A coding agent memory system](../resources/articles.md#introducing-beads-a-coding-agent-memory-system). Yegge’s central claim is that markdown plans are the wrong abstraction for long-horizon agent work; what agents need is **external memory** in the form of a structured, queryable issue graph. Memory here is not a transcript or a vector store. It is a live map of work.

The same idea appears operationally in:

- [Beads](../resources/projects.md#beads)
- [Gastown](../resources/projects.md#gastown)
- [Gas City](../resources/projects.md#gas-city)
- [BeadHub](../resources/articles.md#beadhub-coordination-for-ai-programming-teams)
- [Don’t Send an Agent to do Code’s Job](../resources/articles.md#dont-send-an-agent-to-do-codes-job)

This is the most mature memory pattern in the collection because the object being stored is clear, the operational value is obvious, and multiple projects converge on roughly the same solution.

## 2. Context memory

This is adjacent to, but different from, task memory.

Context memory is about what gets loaded into the model’s active working context right now: constraints, prior decisions, relevant source material, plans, and tools. The strongest framing in the repo comes from:

- [NK’s AI Cookbook](../resources/articles.md#nks-ai-cookbook)
- [Don’t Build Multi-Agents](../resources/articles.md#dont-build-multi-agents)
- [2025: The Year in LLMs](../resources/articles.md#2025-the-year-in-llms)
- [Agentic Engineering Patterns](../resources/articles.md#agentic-engineering-patterns)

Simon Willison’s **context rot** framing is especially useful here: larger context windows do not remove memory problems if the context becomes bloated, stale, or poorly selected. The practical lesson in the current collection is that long context helps, but does not eliminate the need for external memory, retrieval, compression, or careful selection.

## 3. Knowledge memory

This is the least mature but arguably most interesting form.

Knowledge memory is not just task state. It is a structured body of information that can be navigated and reused later: notes, entities, concepts, summaries, links, and accumulated background knowledge.

The clearest example in the repo is Harper Reed’s [*My now immaculate knowledge graph of life*](../resources/articles.md#my-now-immaculate-knowledge-graph-of-life). Reed’s workflow is not primarily about coding agents; it is about turning messy conversational exhaust into local markdown, extracted entities, and linked structure that grows more useful over time. In repo terms, this is less like a task board and more like a developing semantic memory.

The repo’s own evolving knowledge-garden design is an adjacent case: if this repository becomes a useful local corpus of concepts, syntheses, and cross-links, it could itself become a memory substrate for future agent work.

## What Seems Strongest, and What Is Still Uncertain

## Strongest practical claims

### External task memory for coding agents
This is the most robust claim in the repo.

The evidence is strongest because:

- multiple sources converge on it
- the problem is concrete
- the object being stored is well-defined
- the operational value is obvious
- the surrounding systems are already doing real work

If someone asked, “What kind of agent memory should I take seriously today?”, this collection would most confidently answer: **persistent external task memory**.

### Local, inspectable, queryable memory substrates
This is also relatively strong. The repo repeatedly favors systems where memory is visible, travels with the work, and can be manipulated by normal tools. That matters because memory is unusually easy to make opaque. Local-first memory does not solve everything, but it makes memory much easier to debug, trust, and compose into workflows.

### A split between memory for coordination and memory for understanding
The task-memory systems in the repo are mostly about coordination:

- who should work on what
- what is blocked
- what is ready
- what needs resumption

The knowledge-memory examples are more about understanding:

- what ideas recur
- what entities connect
- what themes emerge over time
- what should be retrievable as background knowledge

Both are memory, but they serve different functions. That distinction already seems useful.

## Main uncertainties

### We do not yet know which kinds of memory produce the biggest gains
The collection strongly suggests that memory matters, but it is much weaker on **comparative evidence**. It does not yet tell us how the practical returns compare between:

- a task ledger like Beads
- retrieval over session history
- linked-note knowledge stores
- large-context replay
- compressed summaries
- hybrid systems combining several of the above

The evidence here is architectural and anecdotal, not benchmarked.

### We do not yet know how much long context reduces the need for external memory
The collection pushes against the simplistic story that long context solves memory. But it is still missing stronger evidence on the tradeoff boundary:

- When is a larger context window enough?
- When does retrieval plus compression beat replay?
- When does external memory improve quality, and when does it mainly add complexity?

### We do not yet have good evaluation methods for memory quality
This may be the biggest gap. It is easy to say a memory system exists; much harder to say whether it is good.

A useful memory system should probably be judged on things like:

- retrieval usefulness
- staleness resistance
- ability to survive interruptions
- support for handoff between agents or sessions
- reduction in redundant work
- reduction in context-window bloat
- human interpretability

But the current collection contains little explicit evaluation work along those lines.

### We do not yet understand the failure modes well enough
The repo contains some adjacent warnings — context rot, supervision fatigue, prompt injection, drift — but still limited explicit discussion of memory-specific failure modes such as:

- memory pollution
- stale or misleading summaries
- retrieval of irrelevant but plausible context
- over-accumulation of low-value traces
- fragmentation across too many memory stores
- inability to distinguish durable facts from temporary state

### The evidence is weakest where the ideas are most exciting
The most interesting possibilities in the repo are also the least proven:

- long-lived proactive agents with decaying memory
- personal or organizational knowledge graphs as agent memory
- the repo itself as a memory substrate for agents
- richer semantic memory emerging from accumulated documents

These point beyond chat-session assistance, but in the current collection they are still closer to promising patterns than settled practice.

## Practical Experiments Worth Trying

If the goal is not just to discuss memory but to develop useful tools and processes, several experiments seem well matched to this repository.

## 1. Concept-index-as-memory

**Question:** Can a small concept layer act as useful external memory for future synthesis and curation work?

**Setup:** Use [`ai-generated/concept-index.md`](./concept-index.md) during the next 5–10 resource additions and the next 2 synthesis tasks. Note whether it helped with retrieval, framing, or cross-linking, and where the concepts felt too broad, too overlapping, or too thin.

**Success looks like:** easier conceptual placement of new resources, faster synthesis outlining, and fewer repeated rediscoveries.

## 2. Task-memory-for-research

**Question:** Does a Beads-like task ledger improve long-horizon research work, not just coding work?

**Setup:** Treat a research topic as a set of structured work items rather than a linear note. Track source-finding, reading, synthesis, open questions, and discovered subproblems as resumable tasks. Compare this against a normal markdown planning process.

**Success looks like:** easier resumption after interruption, less forgotten follow-up work, and better separation between “known”, “uncertain”, and “needs reading”.

## 3. Memory distillation

**Question:** What is the smallest durable summary artifact that remains useful across sessions?

**Setup:** For a few topics, create three artifacts after a session: a raw notes dump, a short structured summary, and a concept-oriented note with links and open questions. Later, resume work using only one of the three and compare.

**Success looks like:** one format reliably supports faster and better restart while requiring less context replay.

## 4. Retrieval vs replay

**Question:** When is retrieval better than simply dumping prior material back into context?

**Setup:** For one continuing topic, run two workflows:
- **Replay** — feed the model prior notes wholesale
- **Retrieval** — provide a short summary plus only the most relevant linked entries

Compare output quality, time to useful answer, and amount of human correction needed.

**Success looks like:** retrieval wins on clarity, precision, or effort.

## 5. Memory hygiene

**Question:** What lightweight practices prevent memory stores from turning into junk drawers?

**Setup:** Try a simple hygiene policy for the concept index or future memory files:
- every entry must have a definition
- every entry must link to at least 3 sources
- every entry must list at least one adjacent concept or boundary
- prune anything not used in 30 days

**Success looks like:** the memory layer stays navigable, trustworthy, and resistant to low-value accumulation.

## 6. Repo-as-memory-substrate

**Question:** Can this repository itself become a usable long-term memory substrate for agent-assisted research?

**Setup:** For one topic — for example agent memory — work only from:
- `resources/`
- `librarian-notes.md`
- `ai-generated/concept-index.md`
- one synthesis document

Then ask whether that local corpus is enough to support a second round of better synthesis without going back to the web first.

**Success looks like:** the repo begins to compound understanding, not just store sources.

## Working Conclusion

Based on the current collection, the most defensible claim is this:

**The practical state of agent memory today looks stronger as external infrastructure than as internal cognition.**

The repo contains real evidence that agents benefit from:

- persistent task memory
- local, inspectable memory substrates
- searchable prior work
- linked knowledge artifacts that outlive a session

It contains much weaker evidence that richer, more human-like long-term memory architectures are already well-understood or broadly proven in practice.

That is a useful clarification. The most promising path may not be to imitate human memory first. It may be to build explicit, durable, inspectable memory layers that let agents and humans accumulate useful context over time.
