# What Should Be Retrieved vs Represented vs Maintained in an Agent-Memory Layer for This Repo?

## What This Document Is

A follow-up note for the adjacent-fields experiment. It uses the current concept index plus the recent syntheses on agent memory and adjacent fields to answer a practical design question:

> If this repository becomes a more explicit memory layer for humans and agents, what should be **retrieved**, what should be **represented**, and what should be **maintained**?

This is deliberately repo-specific. The goal is not a general theory of agent memory, but a useful design cut for `ai-resources`.

---

## Short Answer

### Retrieve
Retrieve the **smallest relevant subset** of prior knowledge needed for the task at hand.

In this repo, that usually means:
- specific entries in `resources/articles.md`, `resources/people.md`, or `resources/projects.md`
- relevant sections from synthesis documents in `ai-generated/`
- relevant concept entries from `ai-generated/concept-index.md`
- relevant curatorial notes from `librarian-notes.md`

### Represent
Represent the **stable structure** that helps retrieval and synthesis work better over time.

In this repo, that most plausibly means:
- canonical resource entries
- concept entries
- synthesis documents
- explicit links between concepts, sources, and open questions
- lightweight task/research ledgers for bounded investigations

### Maintain
Maintain the **derived layers** aggressively enough that they remain trustworthy.

In this repo, that most importantly means:
- concept-index hygiene
- synthesis freshness / datedness
- pruning weak or unused generated structures
- keeping canonical annotations and cross-links coherent
- making explicit what is exploratory versus stable

That is the basic split:

- **retrieve selectively**
- **represent minimally but explicitly**
- **maintain continuously where structure accumulates**

## Retrieval: What Should Be Retrieved

The adjacent-fields work suggests that retrieval should be treated as a first-class problem. So the key question is not “what information exists?” but “what information should be surfaced for this task?”

## 1. Retrieve answers, not files

The default retrieval unit should usually not be a whole markdown file. It should be the smallest useful unit, such as:

- a single resource entry
- a section of a synthesis
- one concept entry
- one dated note in `librarian-notes.md`

Why:
- reduces context bloat
- makes relevance easier to judge
- fits the context-engineering lesson that overstuffed context rots

## 2. Retrieve by question type

Different tasks want different retrieval targets.

### If the task is source discovery
Retrieve:
- `resources/articles.md`
- `resources/people.md`
- `resources/projects.md`
- matching concept entries if they exist

### If the task is synthesis
Retrieve:
- relevant concept entries
- the strongest source entries for those concepts
- one or two prior synthesis documents
- related gaps or connections from `librarian-notes.md`

### If the task is repo design / process improvement
Retrieve:
- `ai-generated/knowledge-gardening-for-ai-resources.md`
- concept entries related to memory, local-first knowledge, and agentic knowledge gardening
- experiment notes and results

### If the task is maintenance
Retrieve:
- generated artifacts likely to be stale or overlapping
- related concept entries
- any existing notes on gaps, rejected structures, or curatorial decisions

## 3. Retrieve adjacent concepts deliberately

One of the clearest results of this follow-up pass is that adjacent concepts matter. A query about “agent memory” should often also retrieve:

- context engineering
- agentic knowledge gardening
- local-first knowledge
- anti-hype curation

The current concept index already helped with this. It did not answer the whole question directly, but it surfaced the right neighboring frames quickly.

## 4. Do not retrieve raw history by default

Things that should usually **not** be first-choice retrieval targets:
- full prior conversations
- entire long documents when one section will do
- low-signal generated notes
- stale experiments without clear current relevance

The repo should prefer curated, compact, and interpretable artifacts over raw replay.

## Representation: What Should Be Represented Explicitly

Representation should stay lightweight. The adjacent-fields synthesis strongly argues against over-structuring.

## 1. Represent stable conceptual handles

The concept index is justified when it gives the repo durable handles for recurring ideas.

Good candidates for explicit representation are things like:
- memory for agents
- context engineering
- multi-agent orchestration
- agentic knowledge gardening
- local-first knowledge
- supervision fatigue
- anti-hype curation

These are useful because they connect otherwise separate source entries and make retrieval more intelligent.

## 2. Represent relationships that improve future retrieval

Not every relationship deserves explicit encoding. The ones that do are the ones that repeatedly help answer questions.

Examples:
- concept ↔ source
- concept ↔ adjacent concept
- person ↔ article/project
- synthesis ↔ core source cluster
- concept ↔ open question

This is enough structure to make the repo easier to navigate without turning it into a heavyweight knowledge graph.

## 3. Represent research work when the task is long-horizon

The experiment workspace is already evidence for this.

For bounded investigations, it is worth explicitly representing:
- tasks
- candidate vs accepted sources
- extracted notes
- cross-cutting distinctions
- results / lessons

This is not canonical knowledge. It is working memory for research. But it is clearly useful when the work spans multiple sessions.

## 4. Represent uncertainty explicitly when it matters

One thing the recent syntheses did well is distinguish:
- what seems strongest
- what is uncertain
- where evidence is weak

That should become a representation habit. A memory layer that stores only conclusions and not uncertainty becomes brittle and misleading.

## 5. Do not represent everything

Things that probably should **not** be represented explicitly unless a real use case appears:
- a full ontology of the field
- dense formal schemas for every source type
- graph structure for its own sake
- every weak relationship between entries
- raw session traces as first-class durable knowledge

The correct default here is still minimal commitment.

## Maintenance: What Needs Ongoing Care

Maintenance is where this repo is most likely to win or lose as a knowledge garden.

## 1. The concept index needs active hygiene

The concept index is the most obvious maintenance hotspot.

It should probably be kept healthy by rules like:
- every concept needs a clear definition
- every concept links to multiple sources
- every concept has at least one adjacent concept or boundary
- concepts that never help retrieval or synthesis should be pruned or merged

This matters because concept layers decay quietly if left alone.

## 2. Syntheses need visible datedness

Synthesis documents are snapshots of understanding, not timeless truth.

They should therefore be maintained by:
- dating them clearly
- revisiting them when the source base materially changes
- marking where confidence is weak
- avoiding silent drift into canonical truth

## 3. Experiment artifacts need explicit status

Experiment outputs are useful, but they should not blur into the stable structure of the repo.

Each experiment artifact should be easy to understand as one of:
- active working material
- useful reusable pattern
- historical note
- candidate for pruning

## 4. Canonical files need coherence more than churn

`resources/` and `librarian-notes.md` should still be maintained differently from generated layers.

The priority there is:
- quality of annotation
- coherence of cross-links
- preservation of taste and standards
- avoiding duplication

This layer should be curated carefully, but not subjected to the same kind of churn as experiments and syntheses.

## 5. Ownership matters even in a personal repo

The KM lesson transfers directly: if nobody owns upkeep, decay is the default.

In this repo that means at minimum:
- generated layers should not be allowed to accumulate indefinitely without review
- weak structures should be removed, not just ignored
- new layers should only be added if there is a plausible maintenance path

## The Most Useful Design Rule

The strongest practical rule that emerges from this follow-up is:

> Retrieve selectively, represent only what repeatedly pays for itself, and maintain the derived layers more aggressively than the canonical ones.

That is probably the right operating principle for this repo.

It implies:
- retrieval should get smarter before representation gets heavier
- representation should follow repeated use, not precede it
- maintenance should focus on the layers most likely to decay into low-signal clutter

## Working Conclusion

For this repo, the right split currently looks like this:

### Retrieve
- the smallest relevant curated or synthesized unit
- preferably by concept and section, not by whole-file replay

### Represent
- stable conceptual handles, durable links, bounded research structures, and explicit uncertainty
- but only where that structure repeatedly improves retrieval or synthesis

### Maintain
- concept layers, syntheses, and experiment-derived structures actively
- canonical resource files carefully and coherently
- prune weak or stale generated structures instead of letting them accumulate

That is a much more useful framing than asking, in the abstract, whether the repo should “have memory.”
