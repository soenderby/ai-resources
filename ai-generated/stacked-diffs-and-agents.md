# Stacked Diffs and AI Agents: A Hypothesis

**Status:** Early-stage synthesis. Connects an established software engineering practice to the agent era based on limited evidence. The strongest AI-specific source (Graphite) has a commercial interest in the conclusion. Treat as a hypothesis worth watching, not an established finding.

## What stacked diffs are

Stacked diffs (also called stacked PRs or stacked changes) are a code review workflow where the unit of review is a **single commit** rather than a feature branch. You break work into small, dependent changes that each build, pass tests, and have a clear description. Each change is reviewed independently. You keep working on the next change without waiting for review of the previous one.

The concept originates from Meta's Phabricator and Google's Critique. It is well-established in those engineering cultures and has been formalized into tools like Graphite, Sapling, and Gerrit. Git itself added `--update-refs` to better support the workflow. Gabbard's 2018 essay remains the clearest argument for why stacked diffs produce higher-quality commit history and faster throughput than traditional PRs.

## The problem stacked diffs might solve for agents

AI agents produce code much faster than humans can review it. This creates an asymmetry: generation is cheap, review is expensive. The traditional PR workflow — one branch per feature, reviewed as a single diff — was already strained when humans wrote all the code. With agents producing thousands of lines in minutes, it strains further. Graphite cites studies showing only 24% of large PRs (>1000 lines) receive any review comments. Cursor's CEO, announcing the Graphite acquisition: "reviewing code looks the same as it did three years ago. It's becoming a larger portion of people's time as the time to write code shrinks."

This connects to several documented problems in the collection: [supervision fatigue](../resources/articles.md#the-human-in-the-loop-is-tired) (the cognitive load of reviewing high-volume output), [comprehension debt](../resources/articles.md#comprehension-debt) (the gap between code that exists and code anyone understands), and [work intensification](../resources/articles.md#ai-doesnt-reduce-workit-intensifies-it) (taking on more because AI makes it feel possible).

Stacked diffs are one possible structural answer: make the output reviewable by making each piece small and focused.

## Hypothesized benefits for agent workflows

The following are plausible inferences from combining "stacking helps humans" with "agents produce a lot of code." They are not backed by rigorous evidence.

**Reviewability.** An agent asked to build a feature might produce 2,000 lines touching 30 files. As a single PR, nobody will review that carefully. As a stack — schema → API → service logic → UI → tests — each piece is comprehensible and can receive genuine review.

**Earlier course correction.** In a monolithic PR, you discover the agent made a bad architectural decision only after the whole feature is done. In a stack, foundational decisions are visible first (they're the bottom of the stack), and you can redirect before the agent builds on a flawed base.

**Smaller blast radius.** If one commit in a stack is broken, only that commit and its dependents need fixing. In a monolithic PR, bad decisions are tangled with everything else. Stacking gives you `git bisect`-friendly history by default.

**Parallel agent work.** When multiple agents work in parallel, they produce changes that must merge. Small, atomic diffs conflict less and resolve more easily than large feature branches. Appleton noted this in her Gas Town critique: "This fits how agents naturally work. They're already producing tiny, focused changes rather than sprawling multi-day branches."

**Async human-agent collaboration.** The fundamental stacking benefit — you don't wait for review to start the next piece — maps to agent loops. The agent submits diff 1 and immediately starts diff 2. The human reviews asynchronously. Neither blocks the other.

**Forced planning.** Graphite found that naively telling Claude to "make small PRs" produced messy boundaries — tests before dependencies, refactors split across PRs where none could pass CI. The fix was forcing the agent to plan the stack first. Stacking imposes the same decomposition discipline on agents that it imposes on humans. This is the most interesting claim and rests on a single observation.

**Comprehension debt mitigation.** Each diff has a narrow thesis, a clear description, and a reviewer who actually read it. The commit history becomes meaningful documentation rather than a sequence of opaque agent-session blobs.

## Where it probably doesn't help

These are the author's inferences, not sourced claims.

- **Hallucinations within a single commit.** A small, focused diff can still be wrong.
- **System-level coherence.** Each diff might be fine individually while the whole doesn't cohere. This is the argument from [Don't Build Multi-Agents](../resources/articles.md#dont-build-multi-agents): actions carry implicit decisions that aren't visible to other agents (or reviewers looking at isolated diffs).
- **Bad plans, neatly executed.** If the decomposition itself is wrong, a well-stacked execution of a bad plan is still a bad outcome.
- **Domains without clean decomposition.** CSS aesthetics, visual design, UX feel — work that doesn't separate into atomic units.

## What would strengthen or weaken this hypothesis

**Would strengthen:** Controlled studies comparing agent output quality with and without stacking. Team-level reports on review thoroughness for stacked agent PRs vs. monolithic ones. Evidence that stacking reduces the rate of agent-introduced regressions.

**Would weaken:** Evidence that the overhead of planning stacks (and teaching agents to plan them) exceeds the review benefit. Evidence that reviewers rubber-stamp stacked diffs just as readily as monolithic ones. Evidence that agents produce worse code when forced to decompose (e.g., losing coherence across stack boundaries).

As of March 2026, neither body of evidence exists in any substantial form.
