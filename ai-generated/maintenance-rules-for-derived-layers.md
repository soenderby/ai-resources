# Maintenance Rules for Derived Layers

## What This Document Is

A practical maintenance protocol for the repo’s **derived layers**:

- `ai-generated/concept-index.md`
- synthesis documents in `ai-generated/`
- experiment workspaces in `ai-generated/experiments/`

The goal is simple: keep derived structure useful without letting it silently decay into clutter.

This is deliberately lighter than a governance framework. It is a set of operational rules for a small, local-first knowledge garden.

---

## Core Principle

Derived artifacts should be easier to create than canonical ones, but they should also be easier to prune.

`resources/` and `librarian-notes.md` are the durable curated layer.
`ai-generated/` is the exploratory and synthetic layer.

That means:
- generated files may be provisional
- generated files should be dated
- generated files should be revisited or removed when they stop being useful
- no generated structure gets to become permanent by accident

---

## Artifact Classes

Treat derived artifacts as one of four classes.

## 1. Active working artifacts
Examples:
- experiment notes
- task ledgers
- intermediate synthesis drafts

Rule:
- optimize for usefulness, not polish
- okay to be incomplete
- should clearly live inside `ai-generated/experiments/` or another obviously working area

## 2. Stable syntheses
Examples:
- `memory-for-agents.md`
- `adjacent-fields-for-agent-memory.md`
- `knowledge-gardening-for-ai-resources.md`

Rule:
- should be readable and reasonably self-contained
- should state scope clearly
- should be treated as dated snapshots, not timeless truth

## 3. Navigation artifacts
Examples:
- `concept-index.md`
- reading paths
- future retrieval indexes

Rule:
- judged primarily by whether they improve retrieval and synthesis
- should remain compact and high-signal
- should be pruned aggressively if they become noisy

## 4. Historical artifacts
Examples:
- completed experiments no longer in active use
- syntheses superseded by better versions but still worth keeping as historical records

Rule:
- keep only if they preserve a useful design decision, learning path, or field snapshot
- otherwise archive or remove

## Rules for `concept-index.md`

The concept index is the highest-maintenance derived artifact and should be treated accordingly.

## Keep a concept only if it pays for itself
A concept should remain in the index only if it does at least one of these:
- improves retrieval
- improves synthesis framing
- connects otherwise separate parts of the repo
- supports recurring questions

If it does none of these, merge it, rewrite it, or remove it.

## Minimum quality bar for each concept
Each concept should have:
- a clear definition
- a short statement of why it matters
- multiple connected sources
- adjacent concepts or boundaries
- useful questions

Thin concepts should not accumulate indefinitely.

## Overlap is acceptable; confusion is not
Concepts may overlap, but each should have a clear center of gravity.
If two concepts repeatedly answer the same question, merge or sharpen them.

## Review trigger
Review concept entries when:
- a new synthesis introduces a recurring distinction
- an experiment shows a concept is doing useful work
- a concept is repeatedly skipped during retrieval
- a concept becomes obviously stale or redundant

## Rules for synthesis documents

## Every synthesis is a dated claim set
Synthesis documents should be treated as:
- scoped
- time-bound
- confidence-limited

They should not silently become canonical truth.

## Keep synthesis close to sources
A synthesis should make it easy to answer:
- what is this based on?
- what is uncertain?
- what is missing?

If a synthesis drifts too far from its source base, revise it or mark it as stale.

## Revise only when the source base changes materially
Do not churn syntheses constantly.
Revise when:
- several new high-signal sources change the picture
- a key distinction was missing and is now available
- the old synthesis has become misleading

## Supersession rule
If a new synthesis replaces an old one, either:
- keep both because they capture different scopes, or
- explicitly treat the old one as historical

Avoid silent duplication.

## Rules for experiment workspaces

## Experiments must have a stop condition
Every experiment should define:
- what counts as enough sources
- what outputs are expected
- when to stop gathering and start synthesizing

This prevents endless collection.

## Experiments need resumability
At minimum, an experiment workspace should make it easy to recover:
- current task state
- accepted sources
- extracted concepts
- produced artifacts
- next action

## Experiments should record what did not transfer
This is now a standing rule.
Elegant but non-transferable ideas should be captured explicitly so they do not leak into later syntheses as unearned assumptions.

## Archive aggressively
When an experiment is done, keep:
- sources
- notes
- results
- any durable outputs

But avoid turning every working file into a permanent first-class document if it no longer helps.

## Review Triggers

Derived layers should be reviewed when one of these happens:

- a new concept appears repeatedly across sources
- a retrieval helper starts surfacing low-value results
- multiple syntheses begin overlapping heavily
- an experiment produces a new stable distinction
- a generated file has not been useful for a substantial period
- the repo starts feeling harder to navigate instead of easier

The last one matters most: if the structure increases cognitive load, maintenance is overdue.

## A Lightweight Maintenance Loop

A practical loop for this repo could be:

1. **Add or revise** a derived artifact because it solves a real need
2. **Use it** in actual retrieval or synthesis work
3. **Notice** whether it helps, confuses, or gets ignored
4. **Prune, merge, or sharpen** based on use
5. **Record the lesson** in `librarian-notes.md` if it is likely to matter again

This keeps maintenance tied to use rather than to abstract tidiness.

## Default Biases

When unsure, prefer:
- fewer concepts
- smaller syntheses
- clearer boundaries
- local files over hidden systems
- explicit pruning over silent accumulation

The right failure mode for this repo is probably **under-structuring slightly**, then adding only what proves its value.

## Working Conclusion

The maintenance lesson from the recent experiments is straightforward:

**derived structure is useful, but only if it stays easier to trust than to ignore.**

That means:
- concepts must help retrieval
- syntheses must stay scoped and dated
- experiments must stop and summarize
- weak generated artifacts must be pruned rather than left to rot

If this repo becomes a strong knowledge garden, maintenance discipline will be one of the reasons why.
