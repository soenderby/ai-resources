# Selection Rubric

Use this rubric when deciding which adjacent-field sources to accept for the experiment.

The goal is not to collect the most famous sources. It is to collect the most **transferable** and **operationally useful** ones.

## Core rule

A source should be accepted only if it helps answer at least one of these practical questions:

1. How should an agent **retrieve** the right prior information?
2. What form should durable agent knowledge **take**?
3. How should accumulated information be **maintained** so it stays useful?
4. How should we **evaluate** whether a memory-like system is helping?

If a source is interesting but does not help answer one of those, reject it.

---

## Preferred source types

Prefer, in roughly this order:

1. **Primary sources** that explain a concrete method, system, model, or practice
2. **Practitioner/system-writing** that describes what was built, how it works, and what tradeoffs matter
3. **Clear foundational overviews** that give transferable concepts without drowning in abstraction
4. **Secondary summaries** only when they are unusually clarifying and more useful than the primary source for this experiment

## Desired source mix per field

For each field (IR, KR, KM), prefer:

- at least **one practical/system-oriented source**
- at most **one broad overview**
- ideally **one source with evaluation, tradeoffs, or failure modes**

Avoid a source set made entirely of overviews or entirely of theory.

---

## Acceptance criteria

A source should usually be accepted only if it offers at least one of the following:

- a **reusable design concept**
- a **concrete retrieval / representation / maintenance pattern**
- a **useful evaluation lens**
- a **failure mode or tradeoff** that could affect agent-memory design
- a concept that could plausibly change this repo's structure, process, or future tools

A strong source often supports more than one of these.

---

## Rejection criteria

Reject sources that are mainly:

- definitions without design consequences
- taxonomies without operational payoff
- generic PKM or productivity advice
- vendor positioning or hype
- too domain-specific to transfer meaningfully
- weaker duplicates of a better source already accepted
- highly abstract theory with no clear path into retrieval, representation, maintenance, or evaluation

Interesting is not enough. Transferable is the standard.

---

## Transferability test

Before accepting a source, try to answer these questions:

1. **What is the core transferable concept?**
2. **Where would it apply?** Retrieval, representation, maintenance, or evaluation?
3. **What would it change in an agent-memory design?**
4. **What does not transfer?**

If these cannot be answered clearly, the source should usually be rejected.

## Signal scoring guide

Use the `Signal` and `Transferability` fields in `sources.md` consistently.

### Signal: 5
- unusually clear, original, and relevant
- likely to shape the synthesis directly
- high information density, low fluff

### Signal: 4
- clearly useful and likely worth keeping
- maybe not foundational, but contributes something distinctive

### Signal: 3
- somewhat useful but not central
- keep only if it fills a genuine gap

### Signal: 2
- mostly redundant, vague, or weakly applicable
- usually reject

### Signal: 1
- poor fit for the experiment
- reject

### Transferability: high
- can clearly inform agent-memory design or repo/tool design

### Transferability: medium
- contains useful ideas, but requires interpretation or adaptation

### Transferability: low
- interesting in its own field, but weakly applicable here

Low-transferability sources should usually be rejected unless they are uniquely clarifying.

## Output standard

After reading an accepted source, the agent should be able to produce all of the following in `notes.md`:

- key concepts
- operational takeaway
- why it matters here
- what it might change in repo/tool design
- what does not transfer
- evidence strength

If a source does not support this level of output, it probably should not have been accepted.

---

## Quick decision rule

When in doubt, prefer the source that is:

- more concrete
- more operational
- more transferable
- more likely to change a design decision

and reject the one that is merely more elegant, famous, or abstract.
