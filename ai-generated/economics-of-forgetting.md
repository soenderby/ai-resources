# The Economics of Forgetting: When Context Pruning Helps and When It Hurts

Coding agents accumulate context relentlessly. Every file read, every command executed, every failed edit attempt — all of it stays in the conversation history, sent back to the language model on every subsequent request. The intuition that removing redundant content should save money and improve performance is natural. Several projects have built on this intuition. But the economics are not as straightforward as they appear, because modern LLM APIs cache prompt prefixes aggressively, and breaking that cache has a cost that can exceed the savings from pruning.

This essay investigates context pruning for the [Pi coding agent](https://github.com/badlogic/pi-mono): what existing solutions do, how prompt caching interacts with pruning, and — through empirical analysis of 78 real coding sessions — when pruning actually delivers a net benefit and when it makes things worse.

---

## 1. The Anatomy of a Coding Agent's Context

A Pi session consists of a system prompt, tool definitions, and an ever-growing sequence of messages: user prompts, assistant responses (which may contain tool calls), and tool results. Each request to the LLM includes the full conversation history up to the current point.

Where does the token weight actually live? To find out, I analysed 78 Pi sessions totalling 17,747 JSONL entries.

```bash
# Analyse tool result sizes across all sessions
find ~/.pi/agent/sessions/ -name "*.jsonl" | \
  xargs python3 -c "
import json, sys, os
from collections import defaultdict

tool_results = defaultdict(lambda: {'count': 0, 'total_chars': 0, 'errors': 0})

for sf in sys.argv[1:]:
    with open(sf) as f:
        for line in f:
            try:
                entry = json.loads(line)
            except: continue
            if entry.get('type') != 'message': continue
            msg = entry.get('message', {})
            if msg.get('role') == 'toolResult':
                name = msg.get('toolName', 'unknown')
                chars = sum(len(c.get('text','')) for c in msg.get('content',[])
                            if c.get('type') == 'text')
                tool_results[name]['count'] += 1
                tool_results[name]['total_chars'] += chars
                if msg.get('isError'): tool_results[name]['errors'] += 1

for tool, d in sorted(tool_results.items(), key=lambda x: -x[1]['total_chars']):
    avg = d['total_chars'] / d['count'] if d['count'] else 0
    print(f'{tool:>6}: {d[\"count\"]:>5} calls, avg {avg:>6,.0f} chars, '
          f'{d[\"errors\"]:>3} errors, total {d[\"total_chars\"]:>12,} chars')
"
```

Results across 78 sessions:

| Tool | Calls | Avg chars | Errors | Total chars |
|------|------:|----------:|-------:|------------:|
| `read` | 3,179 | 4,800 | 45 | 15,258,666 |
| `bash` | 3,377 | 2,770 | 312 | 9,355,525 |
| `edit` | 2,033 | 83 | 66 | 169,689 |
| `write` | 461 | 86 | 0 | 39,419 |

Tool results account for the overwhelming majority of context. A detailed trace of a single 216-turn session confirms this:

| Turn | Cumulative chars | Tool result % | Cache read % |
|-----:|-----------------:|--------------:|-------------:|
| 5 | 49,560 | 99.3% | 83.4% |
| 40 | 491,158 | 99.9% | 99.6% |
| 100 | 575,193 | 98.2% | 99.9% |
| 200 | 830,819 | 98.8% | 99.9% |

By turn 5, tool results already constitute 99% of context. User messages contributed 0.1% of the session's 857,755 total characters; assistant text contributed 1.4%; tool results contributed **98.5%**.

The implication is clear: any meaningful context reduction must target tool results, specifically `read` and `bash` output. Everything else is noise in the accounting.

## 2. Existing Approaches

### OpenCode Dynamic Context Pruning (DCP)

The most mature implementation is [opencode-dynamic-context-pruning](https://github.com/Opencode-DCP/opencode-dynamic-context-pruning), a plugin for the OpenCode editor. DCP operates through three zero-cost rule-based strategies and one LLM-assisted strategy:

**Deduplication** identifies repeated tool calls — same tool name, same arguments — and keeps only the most recent output. As the project documentation states: "Identifies repeated tool calls (same tool, same arguments) and keeps only the most recent output. Recalculated when the compress tool runs, so prompt cache is only impacted alongside compression."

**Supersede Writes** removes write-tool results when the written file has subsequently been read, on the logic that the read result contains the current file state and makes the write content redundant.

**Purge Errors** strips the input content from errored tool calls after a configurable number of turns (default: 4), preserving the error message itself while discarding the potentially large input that caused it.

**Compress** is an LLM-driven tool that the model can invoke to summarise a selected range of messages, replacing them with a focused summary. Unlike bulk compaction, this is surgical — the model chooses when and what to compress based on task completion.

A critical design detail: the zero-cost strategies are *not recalculated on every request*. They are recalculated only when the compress tool runs. DCP's documentation is explicit about the reason: "Recalculated when the compress tool runs, so prompt cache is only impacted alongside compression." This batching is not incidental; it is a deliberate response to the cache invalidation problem explored below. The project reports approximately 85% cache hit rates with pruning enabled, compared to roughly 90% without.

### pi-context

[pi-context](https://github.com/ttttmr/pi-context) takes a fundamentally different approach. Rather than automatic rule-based pruning, it gives the agent itself three tools modelled on git: `context_tag` (mark milestones), `context_log` (view a token-annotated skeleton of the conversation), and `context_checkout` (navigate to any tag, carrying a summary). The author describes the philosophy in a [blog post](https://blog.xlab.app/p/51d26495/): "Theoretically, an Agent should be able to perceive 'I need to compress' earlier, achieving more semantic/task-level context compression."

This is not pruning but structured compaction that exploits Pi's session tree for lossless time-travel. It is architecturally interesting — and avoids the cache invalidation problem entirely, because a checkout creates a new branch rather than modifying the existing message sequence.

### pi-extension-observational-memory

[pi-extension-observational-memory](https://www.npmjs.com/package/pi-extension-observational-memory) replaces Pi's default compaction with a structured "observational memory" format containing Observations, Open Threads, and Next Action Bias sections, with configurable observer/reflector token thresholds. It is an improved compaction strategy rather than a pruning tool, and composes with Pi's existing session lifecycle without modifying context on a per-request basis.

## 3. How Prompt Caching Works

Both Anthropic and OpenAI implement prompt caching based on **exact prefix matching**. The LLM API processes a request as an ordered sequence — tool definitions, then system prompt, then messages — and checks whether the beginning of this sequence matches a recently cached computation. If it does, the cached portion is reused, saving both compute and money.

### Anthropic

Anthropic's caching is explicit. Developers mark content blocks with `cache_control` directives; Pi's underlying `pi-ai` library [adds these automatically](https://github.com/badlogic/pi-mono/blob/main/packages/ai/dist/providers/anthropic.js) to tool definitions, the system prompt, and the last content block of the last message. Cached content has a 5-minute TTL, refreshed on each use, with an optional 1-hour TTL available via Pi's `PI_CACHE_RETENTION=long` environment variable.

The pricing structure creates an asymmetry that matters enormously for pruning economics:

| | Price (Sonnet 4) | Relative to base |
|---|---:|---:|
| Base input | $3.00/MTok | 1.0× |
| Cache write | $3.75/MTok | 1.25× |
| Cache read | $0.30/MTok | 0.10× |
| Output | $15.00/MTok | — |

Source: [Anthropic prompt caching documentation](https://docs.claude.com/en/docs/build-with-claude/prompt-caching); [Spring AI blog on Anthropic caching](https://spring.io/blog/2025/10/27/spring-ai-anthropic-prompt-caching-blog/).

Cache reads cost one-tenth of base input price. Cache writes cost 25% more than base. The first cache hit on the second request already recoups the write premium, making caching net-positive almost immediately.

### OpenAI

OpenAI's caching is fully automatic — no code changes or markers required. As their [documentation](https://developers.openai.com/api/docs/guides/prompt-caching) states: "Prompt Caching works automatically on all your API requests (no code changes required) and has no additional fees associated with it." There is no write premium; cached tokens are simply discounted. The discount varies by model generation:

| Model | Cache discount |
|-------|------:|
| GPT-4o | 50% |
| GPT-4.1 | 75% |
| GPT-5.x | 90% |
| GPT-5-nano | 90% |

Source: [OpenAI Prompt Caching 201 cookbook](https://developers.openai.com/cookbook/examples/prompt_caching_201).

The TTL is 5-10 minutes for in-memory caching; extended retention (up to 24 hours) is available for GPT-5.x and GPT-4.1 models.

### The Crucial Implication for Pruning

Both systems cache based on exact prefix matching. In a normal coding agent conversation, each turn appends new messages at the end of an otherwise identical sequence. This means the entire conversation history prior to the new message is a cacheable prefix. Any modification to a message in the middle of the sequence invalidates the cache for everything after it.

The [OpenAI caching cookbook](https://developers.openai.com/cookbook/examples/prompt_caching_201) puts it directly: "Carefully consider the impact of caching from context engineering techniques like compaction."

## 4. Empirical Baseline: What Caching Looks Like in Practice

Before asking whether pruning helps, we need to know what the caching situation actually looks like. I extracted usage data from all 7,701 assistant messages across 78 sessions:

```bash
# Extract cache hit rates per provider across all sessions
find ~/.pi/agent/sessions/ -name "*.jsonl" | \
  xargs python3 -c "
import json, sys
from collections import defaultdict

stats = defaultdict(lambda: {
    'msgs': 0, 'input': 0, 'cache_read': 0, 'cache_write': 0,
    'cost_input': 0, 'cost_cr': 0, 'cost_cw': 0, 'cost_out': 0
})

for sf in sys.argv[1:]:
    with open(sf) as f:
        for line in f:
            try: entry = json.loads(line)
            except: continue
            if entry.get('type') != 'message': continue
            msg = entry.get('message', {})
            if msg.get('role') != 'assistant': continue
            u = msg.get('usage', {})
            c = u.get('cost', {})
            p = msg.get('provider', 'unknown')
            s = stats[p]
            s['msgs'] += 1
            s['input'] += u.get('input', 0)
            s['cache_read'] += u.get('cacheRead', 0)
            s['cache_write'] += u.get('cacheWrite', 0)
            s['cost_input'] += c.get('input', 0)
            s['cost_cr'] += c.get('cacheRead', 0)
            s['cost_cw'] += c.get('cacheWrite', 0)
            s['cost_out'] += c.get('output', 0)

for p, s in sorted(stats.items(), key=lambda x: -x[1]['msgs']):
    total = s['input'] + s['cache_read'] + s['cache_write']
    cr_pct = s['cache_read'] / total * 100 if total else 0
    print(f'{p}: {s[\"msgs\"]} msgs, {cr_pct:.1f}% cache reads, '
          f'total cost \${sum([s[\"cost_input\"],s[\"cost_cr\"],s[\"cost_cw\"],s[\"cost_out\"]]):.2f}')
"
```

| Provider | Messages | Cache read % | Cache write % | Uncached % | Total cost |
|----------|-------:|-------------:|-------------:|-----------:|-----------:|
| OpenAI Codex | 7,235 | 93.7% | 0.0% | 6.3% | $246.56 |
| Anthropic | 326 | 89.7% | 10.2% | 0.1% | $19.88 |
| GitHub Copilot | 144 | 89.4% | 10.4% | 0.2% | $0.00 |

The numbers are striking. On OpenAI Codex, 93.7% of all input tokens were cache reads. On Anthropic, 89.7%. The caching system is already working very effectively. The "redundant" tokens that pruning would target are being served at 10% of base price (Anthropic) or 10% of base price (OpenAI GPT-5.x).

For GitHub Copilot, the cost column tells the story: $0.00. It is subscription-billed. Cache hit rates are irrelevant to cost; they matter only for latency.

## 5. How Much Redundancy Actually Exists?

The case for pruning depends on there being meaningful amounts of redundant content. I scanned all 78 sessions for the three types of redundancy that DCP targets.

### Duplicate Tool Calls

A duplicate tool call is one where the same tool was invoked with identical arguments earlier in the same session. Only the most recent result is informative; earlier ones are redundant.

```bash
# Detect duplicate tool calls (same tool + same arguments)
python3 -c "
import json, os, glob, hashlib
from collections import defaultdict

sessions = glob.glob(os.path.expanduser(
    '~/.pi/agent/sessions/**/*.jsonl'), recursive=True)
dup_sizes = []

for sf in sessions:
    entries = []
    with open(sf) as f:
        for line in f:
            try: entries.append(json.loads(line))
            except: continue

    calls = {}; results = {}; seen = {}
    for entry in entries:
        if entry.get('type') != 'message': continue
        msg = entry['message']
        if msg.get('role') == 'assistant':
            for b in msg.get('content', []):
                if b.get('type') == 'toolCall':
                    h = hashlib.md5(
                        f'{b[\"name\"]}:{json.dumps(b[\"arguments\"], sort_keys=True)}'.encode()
                    ).hexdigest()
                    calls[b['id']] = (b['name'], h)
                    seen.setdefault(h, []).append(b['id'])
        elif msg.get('role') == 'toolResult':
            chars = sum(len(c.get('text','')) for c in msg.get('content',[])
                        if c.get('type') == 'text')
            results[msg['toolCallId']] = chars

    for h, cids in seen.items():
        if len(cids) > 1:
            for cid in cids[:-1]:
                dup_sizes.append((calls[cid][0], results.get(cid, 0)))

read_dups = [c for n,c in dup_sizes if n == 'read']
bash_dups = [c for n,c in dup_sizes if n == 'bash']
print(f'read: {len(read_dups)} dups, median {sorted(read_dups)[len(read_dups)//2]:,} chars')
print(f'bash: {len(bash_dups)} dups, median {sorted(bash_dups)[len(bash_dups)//2]:,} chars')
print(f'total: {sum(c for _,c in dup_sizes):,} chars (~{sum(c for _,c in dup_sizes)//4:,} tokens)')
"
```

Results:

| Tool | Duplicate calls | Median size (chars) | Total chars | Est. tokens |
|------|----------------:|--------------------:|------------:|------------:|
| `read` | 350 | 3,237 | 1,761,396 | ~440,000 |
| `bash` | 510 | 442 | 847,186 | ~212,000 |
| `edit` | 3 | — | 472 | ~118 |
| **Total** | **863 / 9,030** (9.6%) | | **2,609,054** | **~652,000** |

The size distribution of duplicate `read` calls is instructive:

Most duplicated reads fall in the 2,000–10,000 character range. The 44 instances above 10,000 characters are the most economically significant targets.

## 6. The Cache Invalidation Tax

Here is the core problem. Prompt caching works on exact prefix matching. If you alter or remove a message at position *N* in the conversation, the cache is invalidated for every message after position *N*. The tokens after the modification must be reprocessed — at cache-write rates for Anthropic (1.25× base), or at full uncached rates for OpenAI.

### Worked Example: Anthropic Sonnet 4

Consider a 100-turn session on Anthropic Sonnet 4. The context has grown to 80,000 tokens. A `read` call at turn 20 duplicates a later `read` at turn 60. The duplicate result is 2,000 tokens.

**Status quo (no pruning):**

Every turn, the LLM receives ~80,000 tokens. Because this is a growing conversation with an unchanged prefix, most of those are cache reads:

```
Cache reads:  78,000 tokens × $0.30/MTok  = $0.0234
Cache write:   2,000 tokens × $3.75/MTok  = $0.0075  (new content at the end)
                                    Total  = $0.0309 per turn
```

The 2,000 duplicate tokens buried in the history cost $0.0006 per turn at cache-read rates ($0.30/MTok). Cheap.

**With pruning at turn 61 (removing the duplicate at turn 20):**

The prefix changes at position ~20,000 tokens. Everything after that is a cache miss:

```
Cache reads:  20,000 tokens × $0.30/MTok  = $0.0060  (prefix before change)
Cache write:  58,000 tokens × $3.75/MTok  = $0.2175  (everything after change)
New content:   2,000 tokens × $3.75/MTok  = $0.0075
                                    Total  = $0.2310 for this turn
```

**The one-time penalty is $0.2310 − $0.0309 = $0.2001.** The per-turn saving is $0.0006. Break-even requires approximately **334 turns** — far more than most sessions last.

### Worked Example: Larger Duplicate

If the duplicate is 10,000 tokens instead of 2,000:

```
Per-turn saving: 10,000 × $0.30/MTok = $0.003
One-time penalty: ~$0.20 (same cache-miss cost for trailing content)
Break-even: ~67 turns
```

Better, but still marginal for a typical session of 50–200 turns.

### Worked Example: OpenAI GPT-5.x

OpenAI has no cache-write premium. The penalty is the loss of the 90% discount — so uncached tokens cost $1.75/MTok versus $0.175/MTok cached (GPT-5.2 rates). The per-turn savings are similar, but the one-time penalty is lower:

```
Penalty: 58,000 tokens × ($1.75 - $0.175)/MTok = $0.0914
Per-turn saving (2k dup): 2,000 × $0.175/MTok  = $0.00035
Break-even: ~261 turns
```

Still unfavourable for small duplicates. OpenAI's lack of a write premium helps, but the loss of the cache discount is still the dominant cost.

### Worked Example: Subscription Billing

On Claude Pro/Max or GitHub Copilot, there is no per-token cost. Cache invalidation has no financial penalty. The only costs are latency (more tokens to prefill on the first miss) and model attention quality. Both improve with pruning.

**Break-even: immediate.**

## 7. The Batching Insight

DCP's most important design decision is not *what* it prunes but *when*. The zero-cost strategies are recalculated only when the compress tool fires, not on every request. The project documentation explains the reasoning: the prompt cache is already being invalidated by the compression event itself, so batching the automatic pruning alongside it incurs no additional cache penalty.

This turns the economics upside down. If compaction is already breaking the cache, then deduplicating a 2,000-token read at the same time costs nothing extra — you are already paying the cache-miss penalty for the compression. The pruned tokens are pure savings from that point forward.

Pi's architecture supports this directly. The `session_before_compact` event fires before compaction and provides the full message list. An extension could perform deduplication, supersede-write removal, and error purging at this point, then let compaction proceed on the already-pruned message set. The result: a smaller post-compaction context that includes fewer redundant tokens, with zero additional cache cost.

## 8. When Pruning Is Net-Positive

The evidence supports a clear decision matrix:

**Always beneficial, regardless of billing model:**

- **Pruning during compaction events.** The cache is already being broken. Deduplication and supersede-write removal at this point are pure savings. The 652,000 tokens of duplicates and 231,000 tokens of superseded writes across these sessions would have been carried forward into the post-compaction context unnecessarily.

- **Context diagnostics.** A command that displays token breakdown, identifies redundancies, and reports cache hit rates has no downside and helps users make informed decisions. OpenCode DCP's `/dcp context` command is a good model.

- **Pruning very large redundancies (>10,000 tokens).** Even on API billing, a 10,000-token dedup has a break-even horizon of ~67 turns on Anthropic — reasonable for sessions that are already long enough to contain such duplicates.

**Beneficial for subscription users; harmful for API users:**

- **Per-turn deduplication of small results.** On subscription billing (Claude Pro/Max, Copilot), this is free money — better attention, lower latency, no cost. On API billing, the cache invalidation tax makes it net-negative for typical session lengths.

- **Per-turn supersede-write removal and error purging.** Same logic. Beneficial only when tokens are free.

**The critical variable is billing model,** and a pruning extension should know which mode it is operating in. Anthropic API billing and Claude Pro/Max subscription billing produce identical cache-hit statistics in the session log — the difference is whether those statistics map to real money.

## 9. What Should Be Built

Based on this analysis, a context pruning extension for Pi should have three layers:

### Layer 1: Diagnostics (Always On)

A `/context` command that analyses the current session and reports:
- Total tokens by category (system, user, assistant, tool results)
- Cache hit rate for the session
- Identified redundancies: duplicate tool calls, superseded writes, stale error inputs
- Estimated tokens recoverable and the cache-invalidation cost of recovering them

This has no runtime cost, does not modify context, and gives users the information to decide whether pruning is worth enabling.

### Layer 2: Compaction-Batched Pruning (Default)

Hook into Pi's `session_before_compact` event. Before compaction summarises the conversation, apply the zero-cost strategies to the message list being summarised:

1. **Deduplication**: Remove earlier tool results when a later call has identical tool name and arguments
2. **Supersede writes**: Remove write-tool arguments when the file was subsequently read
3. **Purge error inputs**: Strip input content from errored tool calls beyond a configurable turn threshold

Since compaction is already breaking the cache, these modifications are free. The compacted summary will be smaller and cleaner as a result.

Implementation uses the `context` event for per-turn work (subscription mode) and `session_before_compact` for batched work (API mode). Pi's `context` event provides a deep copy of messages that can be modified without affecting the session on disk — exactly the right primitive.

### Layer 3: Aggressive Mode (Subscription Only)

For users on subscription billing, enable per-turn pruning via the `context` event handler. Apply deduplication, supersede-write removal, and error purging on every request. Cache invalidation does not matter because tokens are free. The benefits — reduced context rot, improved model attention, lower latency — are unambiguous.

This mode should be toggled explicitly and should log its actions so users can see what was pruned and how many tokens were saved.

## 10. Conclusion

The instinct that removing waste should save money is sound in most engineering domains. But LLM API economics are unusual: the prompt cache means that "waste" in the conversation history is already being served at 10% of its base price. Removing it breaks the cache for everything that follows, and the one-time penalty for reprocessing the trailing context can exceed hundreds of turns of marginal savings.

The right question is not "is this content redundant?" but "is removing it cheaper than keeping it at cache-read prices?" The answer depends on three variables: **billing model** (subscription makes pruning free; API makes it expensive), **redundancy size** (large duplicates amortise faster), and **timing** (batching with compaction events eliminates the cache penalty entirely).

The practical implication is that a well-designed context pruning tool should not prune by default on API billing. It should diagnose, report, and batch — saving its interventions for the moments when the cache is already being broken, and applying aggressive pruning only when the user has confirmed that tokens are free.
