## Open Questions

### 1. Is "judgment memory" the right label?

The research found no existing source that uses "judgment memory" as a term. The closest terms in use are:

- **Codified context** (Vasilopoulos) — broader, includes factual specs
- **Captured experience** (Vasilopoulos, Case Study 2) — narrower, emphasizes learning from failure
- **Tribal knowledge** (Awesome Reviewers/Baz) — the pre-capture state of what judgment memory stores
- **Agent context files** (academic papers) — the container, not the content
- **Auto memory** (Claude Code) — the automatic capture mechanism
- **Evolving playbooks** (ACE paper) — the maintenance model

None of these is quite right for the concept this research is tracking. "Judgment memory" remains useful because:
- It emphasizes what's stored (judgment, not just facts)
- It distinguishes from task memory (what work exists) and general context (what the project looks like)
- It connects to the repo's existing distinction between "memory that stores work state" and "memory that stores judgment"

But the term has risks:
- "Memory" implies retrieval, which is only half the story (the other half is production and maintenance)
- "Judgment" implies human origin, but auto memory and convention detection produce it automatically
- No one else uses this term, which could be a sign it names something real that hasn't been named, or a sign it names something that doesn't need a separate name

**Provisional answer:** Keep "judgment memory" as the repo's working term. It names a real thing that existing terms only partially cover. But be prepared to abandon it if synthesis reveals that the existing terms are sufficient.

### 2. What is the production mechanism taxonomy?

The research found four distinct production mechanisms for judgment memory:

1. **Manually authored by humans** — CLAUDE.md, AGENTS.md, ADRs, solution docs, postmortem writeups
2. **Automatically captured by agents** — Claude Code auto memory, Every /ce:compound
3. **Automatically detected from codebase** — Codebase Context MCP (convention detection from code and git history)
4. **Mined from human practice at scale** — Awesome Reviewers (extracting review patterns from 1,000+ repos)

These four mechanisms have different tradeoffs:
- (1) is highest quality but highest effort
- (2) is lowest effort but quality depends on agent judgment about what's worth remembering
- (3) is automatic but can only capture what's visible in code, not rationale
- (4) is broad but may not fit any specific project

**Open question:** Is there a fifth mechanism — something like "judgment inferred from tests" or "judgment reverse-engineered from code patterns"? Characterization testing (already in the repo via Feathers) may be adjacent.

### 3. What maintenance mechanisms exist?

The research found these maintenance mechanisms:

- **Manual review cycles** — Codified Context paper: biweekly review, ~30-45 min
- **Context drift detection** — Codified Context paper: script that flags when code changes without spec updates
- **Memory decay** — Codebase Context MCP: stale memories decay and get flagged
- **Convention trend tracking** — Codebase Context MCP: rising/declining adoption percentages
- **CI enforcement** — Archgate: ADR rules checked in CI, violations block merges
- **200-line cap** — Claude Code auto memory: implicit pruning by size limit
- **Incremental addition** — Agent READMEs paper: files grow through additions, rarely deletions

**Gap:** No source described a systematic pruning mechanism for judgment memory. The Codified Context paper mentions biweekly review but doesn't describe how deletions are decided. The 200-line cap is a blunt instrument. Memory decay in Codebase Context MCP is the closest to automated pruning but isn't well documented. This is the biggest maintenance gap.

### 4. What are the confirmed failure modes?

The research found these failure modes with evidence:

1. **Specification staleness** — Codified Context paper: agents generated code using deprecated paths when specs weren't updated. "Agents trust documentation absolutely and out-of-date specs cause silent failures."
2. **Context bloat** — Claude Code docs: >200 lines per file reduces adherence. Reddit: mixing stable rules with ephemeral learnings in one file.
3. **Brevity bias** — ACE paper: iterative optimization collapses toward short, generic prompts, stripping out specific judgment.
4. **Contradictory instructions** — Claude Code docs: "if two rules contradict each other, Claude may pick one arbitrarily."
5. **Convention without rationale** — Observed across sources: rules like "use 2-space indentation" without "because X" are configuration, not judgment. They're easier to generate but less valuable.
6. **Over-reliance on auto-captured judgment** — Reddit thread: "Claude can at any given moment adhere to O(200) rules." System prompt consumes ~50, leaving ~150 for user rules. Auto memory competes for this budget.

**Gap:** No source provided evidence of these suspected failure modes:
- Judgment that helps review but harms exploration
- Local taste calcifying into dogma across a community
- Conflicting judgment from different team members in the same project

3. **Non-coding judgment memory** — All sources are about coding agents. Judgment memory for research agents, writing agents, analysis agents, or operational agents is absent. The Google SRE postmortem material is the closest adjacent source.

5. **Cross-project judgment transfer** — The Awesome Reviewers project extracts patterns across projects, but no source addresses how judgment memory from one project can be safely transferred to another. This may be a fundamentally hard problem (judgment is context-dependent).

6. **Conflict resolution** — When two pieces of judgment memory contradict each other, what happens? No source describes a systematic approach to resolving conflicts in agent context files.

## Adjacent Concept Grounding Summary

### Fields searched and what they contributed

1. **ADR / Design Rationale literature** — Contributed the strongest established vocabulary for one subset of judgment memory (decision rationale). ADRs are well-understood, widely practiced, and now being connected to agent workflows via tools like Archgate. The distinction between passive ADRs (documents) and executable ADRs (Archgate) maps onto a useful spectrum from recorded judgment to enforced judgment.

2. **Organizational / Institutional Memory** — Contributed the framing term "institutional memory" and the basic insight that organizations lose knowledge when people leave. However, most organizational memory literature is too high-level and management-oriented to provide concrete mechanisms. The Forbes piece was rejected for this reason. The Google SRE postmortem culture is the exception — it has concrete mechanisms.

3. **Postmortem / Operational Learning** — Contributed the postmortem-to-playbook loop, which is structurally identical to the review-to-prompt loop in coding agent contexts. The SRE tradition provides the strongest evidence that captured operational judgment can be systematically maintained and reused. The Wheel of Misfortune (training via past postmortems) is a direct analog to agents learning from past solutions.

4. **Tacit Knowledge Capture** — Searched but produced no strong primary sources for this research. The academic literature on tacit knowledge is mostly about interpersonal knowledge transfer in organizations, not about encoding judgment into machine-readable form. The concept is real but the existing literature doesn't connect well to agent workflows.

5. **Knowledge Management** — The broader KM field contributes the idea that knowledge decays by default and requires active maintenance. But the field is large, mostly organizational, and the specific mechanisms (decay, pruning, refresh) are better evidenced in the agent-specific sources than in generic KM literature.
