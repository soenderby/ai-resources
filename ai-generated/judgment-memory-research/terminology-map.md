# Terminology Map

Structured map of terms encountered during research. Each term tracked separately until distinctions are clear.

## Agent context file
- **Exact phrase:** agent context files, agent READMEs, agentic coding manifests
- **Sources:** Agent READMEs paper, Codified Context paper, AGENTS.md spec, Claude Code docs
- **Provisional meaning:** A dedicated file (CLAUDE.md, AGENTS.md, .cursorrules, copilot-instructions.md) that gives an AI coding agent project-specific instructions — architecture, conventions, build commands, coding standards, workflows. Loaded at session start as persistent context.
- **Overlap with judgment memory:** The container for judgment memory, but not identical. Agent context files also store factual context (project structure, build commands) that is not judgment.
- **Key distinction:** Agent context files are the *vessel*; judgment memory is a *kind of content* that can live there. An agent context file with only build commands and file structure has no judgment memory.
- **Confidence:** High — well-established term across multiple academic sources.

## Codified context
- **Exact phrase:** codified context infrastructure, documentation as infrastructure
- **Sources:** Codified Context paper (Vasilopoulos)
- **Provisional meaning:** Structured artifacts written explicitly for machine consumption — documents whose primary audience is an AI agent, not a developer. Treated as "load-bearing artifacts that AI agents depend on to produce correct output."
- **Overlap with judgment memory:** High. Codified context includes judgment (conventions, known failure modes, domain expertise) alongside factual information (subsystem specs, API docs). The "captured experience" case study is almost exactly judgment memory.
- **Key distinction:** Broader than judgment memory. Includes pure factual specifications that an agent needs but that don't encode human judgment per se.
- **Confidence:** High — well-defined in one strong source.

## Context engineering
- **Exact phrase:** context engineering
- **Sources:** Anthropic effective context engineering post, Karpathy (credited), Mei et al. survey (referenced)
- **Provisional meaning:** "The set of strategies for curating and maintaining the optimal set of tokens during LLM inference." The successor to prompt engineering, covering the entire context state — system instructions, tools, MCP, external data, message history.
- **Overlap with judgment memory:** Context engineering is the discipline; judgment memory is one kind of content being engineered. Good context engineering determines *when* and *how* captured judgment gets loaded.
- **Key distinction:** Context engineering is a design discipline. Judgment memory is a kind of information. They relate as practice to content.
- **Confidence:** High — becoming the dominant framing term.

## Auto memory / MEMORY.md
- **Exact phrase:** auto memory, MEMORY.md, learnings, corrections and preferences
- **Sources:** Claude Code docs, Claude Code Best Practices, SFEIR Institute
- **Provisional meaning:** Claude Code's automatic mechanism for recording notes across sessions. Claude writes these based on observed corrections and preferences. Stored in `~/.claude/projects/`. Records "recurring patterns, corrected errors, and observed preferences."
- **Overlap with judgment memory:** Strong. Auto memory is the most widely deployed automatic judgment capture mechanism. It records exactly the kind of corrective judgment the research plan describes.
- **Key distinction:** Auto memory is *automatically captured* by the agent, unlike CLAUDE.md which is *manually authored* by the human. This production mechanism difference matters.
- **Confidence:** High — documented first-party feature.

## Hot memory / cold memory
- **Exact phrase:** hot memory, cold memory, constitution
- **Sources:** Codified Context paper
- **Provisional meaning:** A loading-strategy distinction. Hot memory is always loaded (the constitution/CLAUDE.md). Cold memory is retrieved on demand (detailed specs via MCP). Between them sit specialized agents, invoked per task.
- **Overlap with judgment memory:** Judgment can live in both tiers. Conventions and review heuristics belong in hot memory. Detailed subsystem-specific judgment belongs in cold memory.
- **Key distinction:** This is a *loading strategy*, not a *content type*. The same piece of judgment memory could be hot or cold depending on how frequently it's needed.
- **Confidence:** High — clean architectural distinction from one strong source.

## Tribal knowledge
- **Exact phrase:** tribal knowledge, institutional knowledge
- **Sources:** Awesome Reviewers/Tessl blog, Forbes piece (rejected)
- **Provisional meaning:** Hard-earned lessons that rarely get documented but shape how teams write and maintain quality code. In the Awesome Reviewers context: the review patterns and conventions that experienced engineers apply implicitly but have never written down.
- **Overlap with judgment memory:** Very high. Tribal knowledge is the *input* that judgment memory attempts to capture and externalize. The Awesome Reviewers project is explicitly about converting tribal knowledge into machine-readable form.
- **Key distinction:** Tribal knowledge is implicitly held by people. Judgment memory is explicitly captured and stored for reuse. The conversion from one to the other is the interesting mechanism.
- **Confidence:** High — well-established term, clear meaning.

## Design rationale / decision memory
- **Exact phrase:** design rationale, architecture decision records, decision memory
- **Sources:** Nygard (ADRs), Adolfi blog, Archgate, Chris Swan blog, Parnas (implicit in information hiding)
- **Provisional meaning:** The recorded *why* behind architectural and design decisions. ADRs are the canonical modern format: short documents capturing context, decision, and consequences.
- **Overlap with judgment memory:** Design rationale is a *subset* of judgment memory — specifically the rationale behind structural decisions. Judgment memory also includes review heuristics, coding conventions, and solved-problem patterns that are not about a single architectural decision.
- **Key distinction:** Design rationale is narrower and more decision-focused. Judgment memory includes ongoing preferences and patterns, not just point-in-time decisions.
- **Confidence:** High — well-established in software engineering literature.

## Captured experience
- **Exact phrase:** captured experience, codified context as captured experience
- **Sources:** Codified Context paper (Case Study 2)
- **Provisional meaning:** Lessons learned from implementing a feature that are documented so future agents don't re-derive through trial-and-error. In the case study: a UI synchronization specification that prevented the next implementation from repeating a painful debugging cycle.
- **Overlap with judgment memory:** This is the closest single-source synonym. Captured experience is judgment memory at its most concrete: "we learned X the hard way, so now it's written down."
- **Key distinction:** "Captured experience" emphasizes the learning origin. Judgment memory could also include pre-existing conventions that weren't learned through failure.
- **Confidence:** Medium — used in one strong source but not widely adopted as a term.

## Playbook
- **Exact phrase:** playbooks, evolving playbooks
- **Sources:** Devin/Cognition docs, ACE paper (evolving playbooks), Google SRE postmortem culture
- **Provisional meaning:** Reusable templates that encode how tasks should be done, typically created by experienced engineers. In the ACE paper, contexts are treated as "evolving playbooks" refined through a generate-reflect-curate cycle.
- **Overlap with judgment memory:** Playbooks are a storage format for judgment memory — the step-by-step encoding of "how we do X and why." The "evolving" aspect is a maintenance mechanism.
- **Key distinction:** Playbooks emphasize process (how to do X step by step). Judgment memory is broader — it includes heuristics, preferences, and taste that don't reduce to a checklist.
- **Confidence:** Medium — used across multiple contexts but with varying meaning.

---

## Convention / coding standard
- **Exact phrase:** coding conventions, team conventions, coding standards, house style
- **Sources:** Nearly all sources; most prominent in AGENTS.md spec, Claude Code docs, Codebase Context MCP
- **Provisional meaning:** Rules about how code should be written — naming, formatting, patterns, architectural constraints. Can be explicitly authored or (in Codebase Context MCP) automatically detected from code and git history.
- **Overlap with judgment memory:** Conventions are judgment memory when they include rationale. A rule like "use 2-space indentation" is a convention. A rule like "use composition over inheritance because our domain model evolves faster than our type hierarchy" is judgment memory.
- **Key distinction:** Conventions without rationale are closer to configuration than judgment. The presence of *why* is what makes a convention into judgment memory.
- **Confidence:** High — universal term, but the convention/judgment distinction is important.

## Battle scars
- **Exact phrase:** battle scars
- **Sources:** Codebase Context MCP
- **Provisional meaning:** "Decisions, failures, workarounds that look wrong but exist for a reason — the battle scars that aren't in the comments." Recorded once, surfaced automatically so agents don't "clean up" something that was deliberately left that way.
- **Overlap with judgment memory:** Very high. Battle scars are judgment memory in its most experiential form — knowledge that can only come from having been burned.
- **Key distinction:** More specific than judgment memory. Battle scars are specifically about negative experience (things that went wrong). Judgment memory also includes positive judgment (what works well, what to prefer).
- **Confidence:** Medium — vivid metaphor used in one source but captures something real.

---

## Context drift / specification staleness
- **Exact phrase:** context drift, stale specifications, out-of-date specs
- **Sources:** Codified Context paper, Codebase Context MCP (memory decay)
- **Provisional meaning:** When code changes but documentation/specs don't, agents generate code based on outdated information. The Codified Context paper: "On at least two occasions, outdated context documents caused agents to generate code that conflicted with recent refactors."
- **Overlap with judgment memory:** This is a *failure mode* of judgment memory, not a type of it. Stale judgment is worse than no judgment because agents trust it absolutely.
- **Key distinction:** Context drift describes the degradation mechanism. Judgment memory is the thing that degrades.
- **Confidence:** High — observed and documented failure mode.

---

## Brevity bias
- **Exact phrase:** brevity bias
- **Sources:** ACE paper (Zhang et al. 2026), referenced in Codified Context paper
- **Provisional meaning:** Iterative optimization of context tends to collapse toward short, generic prompts. Agents performing with less embedded domain knowledge make more errors, contradicting the optimization pressure.
- **Overlap with judgment memory:** This is a failure mode where automatic refinement strips out the specific judgment that makes memory valuable. Important for any system that auto-updates its own context.
- **Key distinction:** This is a maintenance failure mode, not a term for judgment memory itself.
- **Confidence:** Medium — identified in one academic source but plausible and important.

---

## Rules
- **Exact phrase:** rules, .claude/rules/, .cursorrules
- **Sources:** Claude Code docs, Cursor docs
- **Provisional meaning:** Modular, topic-specific instruction files that can be scoped to specific file paths or directories. Rules are the next level of organization beyond a monolithic CLAUDE.md. Can be shared via version control.
- **Overlap with judgment memory:** Rules are a storage mechanism for judgment memory — a way to scope captured judgment to the contexts where it applies.
- **Key distinction:** Rules emphasize scoping and organization. They're a structural pattern for *where* judgment lives, not a term for the judgment itself.
- **Confidence:** High — documented feature in major tools.

---

## Skill
- **Exact phrase:** skills, agent skills
- **Sources:** Claude Code docs, MIT Missing Semester, AGENTS.md spec
- **Provisional meaning:** One level of indirection above AGENTS.md: a list of capabilities with descriptions that the agent can "open" (load into context) on demand. Avoids context bloat by loading only what's needed.
- **Overlap with judgment memory:** Skills can encode judgment (e.g., a code-review skill with specific heuristics), but the term primarily describes the loading mechanism, not the content.
- **Key distinction:** Skills are an *access pattern* for context. They may or may not contain judgment.
- **Confidence:** High — well-established feature term.
