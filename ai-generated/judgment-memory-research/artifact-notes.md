- **Location:** `./CLAUDE.md` or `./.claude/CLAUDE.md` in any project; also `~/.claude/CLAUDE.md` for user-level
- **Source/repo:** Claude Code (Anthropic), ubiquitous across Claude Code projects

- **What kind of judgment is stored:** Coding standards, architectural decisions, naming conventions, build/test commands, workflow instructions, project-specific constraints. Ranges from pure configuration (build commands) to rich judgment (architectural rationale, review heuristics).
- **How it is updated:** Manually by humans. Some teams have agents suggest updates. Reddit thread notes: "Stable rules go in .claude/rules/ as separate files. CLAUDE.md becomes the scratch surface for session-learned context that you periodically prune."
- **When it is loaded:** Every session start. Entire contents loaded into context window.
- **What behavior it changes:** All agent behavior — code generation, refactoring, testing, review. The more specific and concise the instructions, the more consistently Claude follows them.
- **Evidence of maintenance:** Agent READMEs paper found 67.4% of Claude Code files modified in multiple commits, with median update interval of ~24 hours. Maintenance is through incremental additions.
- **Observed or likely failure modes:** Context bloat (>200 lines reduces adherence), conflicting instructions across nested files, stale conventions, mixing stable rules with ephemeral learnings, overly generic instructions.
- **Classification:** Hybrid — primarily judgment memory when it contains rationale and heuristics; general context when it's purely factual (build commands, file structure).

- **What kind of judgment is stored:**

  - **Tier 2 (19 agents, ~9,300 lines):** Domain-expert personas with embedded project-specific knowledge. Over half of each spec is domain knowledge (failure modes, patterns, formulas) rather than behavioral instructions. Created when debugging a domain consumed extended sessions without resolution.
  - **Tier 3 (34 docs, ~16,250 lines):** Subsystem specifications, protocol docs, pattern guides. Living documents generated and updated by AI at developer direction.
- **How it is updated:** Constitution and agent specs updated by developer direction after code changes. Specs updated in same session as code (~5 min overhead). Biweekly review pass (~30-45 min). Context drift detector flags when source files change without corresponding spec updates.
- **When it is loaded:** Tier 1 always; Tier 2 per task via trigger table; Tier 3 on demand via MCP retrieval.
- **What behavior it changes:** Prevents repeated debugging failures, maintains convention adherence, coordinates across 283 sessions. Case study: save system spec referenced in 74 sessions with zero save-related bugs.
- **Evidence of maintenance:** 1-2 hours/week total. Drift detector partially automates staleness detection. Biweekly review pass.
- **Observed or likely failure modes:** Specification staleness (documented: twice, agents generated code using deprecated paths). "Agents trust documentation absolutely and out-of-date specs cause silent failures." Brevity bias if automated refinement strips domain knowledge.
- **Classification:** The fullest embodiment of judgment memory at scale. All three tiers contain judgment: Tier 1 = conventions and routing judgment; Tier 2 = domain expertise and failure-mode knowledge; Tier 3 = subsystem design rationale and pattern knowledge.
